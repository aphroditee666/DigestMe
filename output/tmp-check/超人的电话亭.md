## 2. 训练审美 | 懂得分辨好和坏的作品集封面

- 日期: 2026-04-30 00:38
- 链接: https://mp.weixin.qq.com/s?__biz=MzI0Njc5NzM5OQ==&mid=2247506509&idx=1&sn=498de1132b97588585bc8631a63225cb

```
原创 酸梅干超人 2026-04-30 00:38 上海 
 最近课程讲到作品集封面的设计，虽然这属于平面设计，是游离在 UI 知识体系之外的技能，但最难的地方从来都不是软件功能或实战技法的运用，而是对平面审美的积累以及对 “好” 与 “坏” 的区分。 
 所以在教学过程中，经常有同学对一些质量极差的案例发出不知道差在哪的疑问。这是最难回答的问题…当一份作品毫无专业性时，想要用专业的角度去解释它是徒劳的，因为哪里都是问题，解析就等于重头讲一遍所有基础知识。 
 这种认知的缺失会直接导致新手不知道如何动手实践，或以一些错误的案例为参考照抄，所以这是必须要解决的问题。所幸的是，提升认知并不需要多努力的训练还是多复杂的理论知识，只要看完下面的分享就能理解。 
 作品集封面的主要问题整理 
 首先我们展示一些作品集封面设计中最常见且必须要避免的问题，帮助大家先做个排雷。 
 问题1：直接套用常见的模板 
 很多人做封面就是直接找网上的模板套进去，或是直接照着一些“优秀案例”直接抄，这种做法会造成两种负面影响。 
 一个是你能找到的模板和主流平台的优秀案例，往往已经“烂大街”，被无数人用过、看过、抄过了。另一个问题就是你自己改完往往还和原作风格不搭，有些优秀案例的精髓完全没有继承，只剩下画虎不成反类犬的尴尬。 
 这种敷衍的态度是招聘者一眼就能感受出来的，或者直接就能看出这是哪套模板或是谁做的案例。 
 问题2：使用“非主流”字体 
 作品集封面通常也有标题，而标题文本自然需要由字体字形呈现。越是初级的设计师越喜欢选择一些非常离经叛道的字体，不仅视觉效果不佳，而且生怕别人看懂你写了什么。 
 除了选择字体外，还很多初级设计师可能为了表示“诚意”会自己独立设计标题的字体，但呈现的结果大致上就是告诉观看者你不懂字体设计…… 
 问题3：视觉装饰过多且冗余 
 在平面设计中是有密集、聒噪、繁复的设计风格，但这类设计风格并不是特别好驾驭，要做到乱中有序需要有足够多的经验和专业能力支撑。同时，这类设计风格和多数UI项目也非常不搭，即使你勉强做出一个及格的案例，也很难确保内页的设计风格是能跟的上的。 
 这种设计方向除了硬给自己增加工作量以为，也在消耗观看者的耐心，没有任何正向收益，只会给作品集增加负面影响。 
 问题4：使用AI生成的图形元素 
 滥用 AI 也是目前做作品集时最大的问题之一，就是缺素材了，就随便用 AI 生成一个，尤其是那些劣质的 3D 形象和场景。这些元素的生成和获取过于简单，既不能提高画面质感，也不能表现出自己对 AI 应用的深度和认识。 
 可以很确定的说，绝大多数招聘者看见这些元素已经不能叫反感，是 —— 厌恶。 
 问题5：使用了 “非本人” 照片 
 这是一个很荒诞的问题，在自己作品集的封面放别人的照片，尤其是放“老外”的照片，等于在一个正式申请场合里提交个人照片时，你当是上传 “QQ头像”。 
 本来属于没必要提的常识，但收到学员作品集里有不少存在这个问题，就必须重点提出来。同时，也不要拿自己的照片用 AI 进行后期和重新 P 图，这种效果也是一眼假，只会让作品集显得更不真实。 
 上面提到的几个作品集常见问题，不是技术性问题，是认知问题。当设计师不知道什么样的设计是“正确”且“有效”时，就会寄希望于做加法来呈现更特殊的视觉效果，但又因为缺少平面经验和审美积累，所以越加越差，越抹越黑。 
 作品集封面的设计参考 
 想要做好作品集封面，光找其它成熟、优秀的案例时比较困难的，因为公开展示的数量少，很多大家都看过。另一个原因，就是很多封面的设计虽然效果不错，又属于技术门槛偏高的类型，所以并不适合作为案例来参考和模仿。 
 所以我们给大家最终的建议，就是先大量积累 —— 简约成熟的平面排版设计案例。简约是指画面内容元素简单留白多，成熟是指一看就能感觉到专业性强不是乱做的。 
 比如下面这样的案例图： 
 平面设计中最初级的要求，就是用尽可能少的元素和视觉效果来实现有效的信息传递，包括情绪和文案内容，所以这个专业也叫视觉传达。 
 而我们去看这些简约案例的目标，就是要从中理解简单的视觉元素、排版可以实现什么样的视觉效果，它们是怎么组合出能打动你、吸引你注意力的效果的，我们做封面和这些平面排版设计就是在做同一件事。 
 观察优秀的平面案例而不是直接去找现成的封面，还能帮助你们先建立正确的 —— 平面视觉价值观，它会让你们迅速理解优秀的设计应该是什么样的，而不应该是什么样的意识。只要花一两天时间，你就会明白解释封面设计优劣和解释 “神仙姐姐” 和 “如花” 的外貌一样是毫无必要的事情…… 
 这种意识也会推动你们在实践过程中往正确的方向靠拢，其中最有意思的一个影响，就是网上那么多优秀案例应用的素材或图片，缺乏 PS 基础是做不出来的，现在有 AI 辅助，设计的可能性就被拉满，你还会想着生成那些破 3D 卡通图？ 
 最后问题回来了，我们上哪去看这些设计案例呢？我主要建议到 Pinterest 或 Behance 里看（都要挂上）。 
 Pinterest 会更高效一点，可以在搜索栏中搜索平面、封面、网页（用英文）等关键字，然后下滑找到合适的案例，打开详情后还能获得一连串同类推荐。 
 而在 Behance 的案例更多样化，主要查看 “图形设计” 或 “广告” 分类下的案例，每天可以在上班或坐在电脑前的十分钟去刷一刷做个拓展（同时提升整个作品集设计的审美）。 
 最好将你们看到觉得优质或者有代表性的反面案例都做个收集，可以帮助我们更好的建立对平面设计和封面设计的认识。
```

## 4. 求职者应如何优化你的作品集？·下篇

- 日期: 2026-04-22 22:34
- 链接: https://mp.weixin.qq.com/s?__biz=MzI0Njc5NzM5OQ==&mid=2247506471&idx=1&sn=949559623a37ba00e53bc13efd549868

```
原创 酸梅干超人 2026-04-22 22:34 上海 
 本周五会有节专门针对 B 端作品集输出的公开课，下面预约就可以： 
 上一篇作品集的分享解释了作品集真实受众的观看模式和心态， 
 上篇链接： 面试官是如何审视你的作品集？·上篇 
 而今天我们聚焦在 B 端项目界面展示部分这个重灾区，来讨论它存在的主要矛盾，以及如何优化。 
 界面展示的问题分析 
 作品集项目包装中的界面展示部分，之所以是重灾区，就是因为多数设计师根本不知道应该展示什么，只是根据其他人的案例随意得堆积页面和文案，造成了以下这些常见而又致命的问题。 
 问题1：页面展示尺寸过小 
 这是最直接也最严重的问题，很多新人做项目时就是无脑做 1920*1080 的界面，然后再将他置入到作品集画布里排版，要正常排版就得缩小界面的尺寸，而很多设计师还为了所谓的“分析”展示，进一步压缩界面展示的空间，让界面的宽度占比甚至不足一半。 
 雪上加霜的是，作品集观看者还会缩小作品集预览的窗口，让展示界面的实际尺寸变得更小。 最后的结果，就是界面里有什么内容根本就看不清。 
 前面我们也提过，作品集受众耐心是越来越差的，这种展示方式本身就形成负面的影响，如果不是包装质量过硬就更不会放大去查看页面内容。大概率就是划完一个项目以后也不知道你到底设计了什么东西，懒得去放大考究，直接关掉看下一份。 
 问题2：界面太亮全是留白 
 这是一个在真实项目当中不是问题但在展示环境会变成问题的问题，原因就是部分项目使用全白框架和背景，而界面中的内容又不多， 导致观看体验就是“白花花一片”。 
 这会导致观看者很难聚焦注意力到界面中，尤其界面本身留白多，但是包装又复杂，且旁边干扰的文案信息又非常多的情况。 
 问题3：界面之间差异太小 
 B 端项目往往界面之间差异并不大，做来做去都是差不多的表格和表单， 导致放出来的页面非常相似 。作为旁观者来说，根本无法区分这些界面之间的差异，只会因为这种重复的展示而感到厌烦。 
 对于看得人来说，项目这么无聊且重复就是你的问题，不需要额外进行谅解和投入关注，还是去听你的辩（jiao）解（bian）。 
 问题4：界面前后展示顺序混乱 
 即使界面展示尺寸、布局、样式都不错，也不代表观看者就能满意。还有个很严重的问题， 就是展示页面的顺序是缺乏逻辑联系的 。即前后展示的页面所属的模块、功能毫无关系，非常的跳跃，比如上个页面还是功能 A 的列表，下个页面就是功能 B 的表单，再下面一个页面又突然变成功能 C 的详情。 
 B 端项目本身就比较复杂，不是同行往往很难理解这些页面的作用和关系，展示做的缺乏联系性就只会让观看者更困惑更迷茫。 
 问题5：分析的废话文学占比太大 
 因为对分析的错误理解，很多同学会强行给页面添加大量的文案讲解，很多文案信息又没有实际的营养和价值，而为了放这些文案，可能又占用了界面的展示空间，形成双向打击。 
 在界面展示部分，观看的顺序一定是先界面再文案，无法满足这种要求或强行制造出浏览阻力，只会加速让观看者放弃这份作品集。 
 以上几个问题都是目前新手端 B 端作品集中最常见的问题，说到底就是大家不把作品集当成是一套需要被人“看下去”的文件，只是因为要做而做的岗位申请表单（填完提交上去审核）。 
 界面展示的设计思路 
 要解决上面的问题，就得重头规划和调整，而下面就是我们给大家整理出来的设计建议。 
 步骤1：提前安排展示顺序 
 第一个步骤也是最关键的步骤，那就是安排在包装中要展示的界面有哪些，以及相应的展示顺序，可以使用思维导图或文档工具做梳理。 
 建议以展示多个功能模块为目标去组织，比如一个医院后台，展示首页、病人模块、医生模块、药品模块、订单模块等主要功能模块。每个模块下级再选出最重要且有联系的页面出来，比如管理表格页、新建表单页、详情查看页这样。 
 项目展示的空间是有限的，不是无限延长的，所以展示的模块和页面数要做控制。一定要选择最能代表你当前项目功能、特色的部分，舍弃掉次要的部分节省空间。 
 步骤2：优化界面展示尺寸 
 第二步就是置入的界面尺寸一定要做优化，如果你原来的项目做的是 1080P，那么建议使用1400 或 1280 宽的画布重新调整一遍要展示的界面。如果原来的界面中功能非常多，你可以适当的去缩减一部分内容来满足界面展示的需要。 
 作为观看者来说 B 端界面内容有减少，且没有影响核心功能和服务，那么压根就不会察觉也不会在意，远远没有内容太小看不清的影响大。 
 而在具体展示中，主要页面的展示的宽度不要小于作品集画布的 2/3，才能确保界面能被看清。只有相对次要且内容少的页面，属于给主要页面做陪衬的类型，才能缩小尺寸看个大概即可。 
 步骤3：丰富界面视觉样式 
 如果置入页面后，发现前后的界面样式非常单调又一致，那是需要自己主动去调整一部分界面设计的样式了。 
 一方面是前面提到的全局泛白的问题，强烈建议在全局框架和内容背景上制造色彩对比，提高视觉的丰富性。 
 第二个要点就是即使是表格页或表单页，也可以通过不同的布局版式来设计，或是自己添加一些模块内容来丰富视觉效果。这是 B 端界面设计的基本常识，光抱怨原项目要求低界面需求一致是没用的。 
 步骤4：梳理展示辅助文案 
 最后就是要重点强调的文案应用逻辑，在界面展示模块中，文案信息最重要的作用是和观众解释 —— 界面的作用以及隐藏的细节 ，而不是堆积设计思路和专业术语强行升华它们。 
 也就是说文案不能完全没有，但只要能介绍清楚你的模块功能、界面作用、关键细节（如组件状态变化）即可。这样即可以让这个界面展示变得更直观易懂，也可以给界面预留出更多的空间出来，获得更好的展示效果。 
 界面展示区域的文案是次要的角色，不要反客为主把这些内容变成自己驾驭不了的主角。把精力更多投入在界面设计和排版上，不仅能获得更好的观看评价，也能节省自己更多的时间。 
 在 B 端项目包装中，界面展示就要明确围绕舒适、流畅、美观的视觉效果去构建并设计，不是说设计思路和想法不重要，只是这些东西是留在面试环节里输出的不是用这么点页面和文案解释得清楚的。 
 还有一点，就是谈 B 端项目只看功能和思路，不重视视觉表现的说法： 
 就像说现代年轻人去相亲只重视人品和智商一样荒诞…… 
 结尾 
 项目包装这件事没有大家想得那么复杂，真的建议你们自己做完以后过个几天缓缓，然后自己从上到下看一遍再读一遍，觉得哪里不对劲的那就是真不对劲，用更简单的做法往往比你们以为的高级和复杂有效。
```

```
 AI工具的发展回顾 
 AI 行业的爆发始于2022年 ChatGPT 和 Midjounery 的发布，到现在已经快4年了，整个发展过程用突飞猛进、日新月异形容一点都不过分。 
 但不管它过去还是现在，发展得有多迅猛，都依旧处于行业早期的开荒和发展阶段，离真正成熟的理想形态还有很长的一段路要走，这是我们首先要达成的共识。 
 而之所以有这样的结论，原因就是 AI 的发展速度是超过现实世界匹配它的速度的，这造成了一种经济和机会成本上的巨大阻力。 
 简单来说，我们可以把 AI 技术的应用拆分成上、中、下游三个层，上层是 AI 的核心，主要是算法、大模型，中游是负责运行AI模型的硬件和支撑，即 AI 芯片、光模块 CPO、高速内存 HBM、服务器液冷、特高压电路等，下游即 AI 具体的应用工具，包括 AI 软件、硬件、智能体等。 
 这是一个非常庞大且复杂的产业链，它们之间是需要相互耦合，才能顺利应用到具体的使用场景中。 
 简单来说，就是 AI 模型厂商需要根据技术路线和成本选择硬件并购买算力中心，以及消耗海量的电力来训练自己的模型并维持它的正常运转。虽然这些背后的基建作为普通用户可能感受不深，但头部厂商的竞争已经到了白热化的阶段。 
 比如大模型的开发必然要关注硬件规格和架构，大洋彼岸除了英伟达外，还陆续推出了谷歌 TPU、微软 Maia、英特尔 Gaudi 等芯片，它们代表了不同的技术方案，应用在了不同的模型上。 
 而因为贸易和地缘政治的因素，国内是无法进口这些高端芯片，所以我们只能自研生产以满足国产大模型的需求。如阿里在自研并应用自家的真武系列芯片， DeepSeek 近期更新缓慢的重要原因就是因为要适配华为的昇腾系列芯片等。 
 之所以要提这点，是因为 AI 大模型的发展也适用马太效应，赢家通吃。在接近性能和水平下谁家应用的算力、电力成本最低，谁就具备了最大的优势。 
 而每家大模型厂商都要面临技术路线的选择，并投入海量的成本孤注一掷做基建，很难走回头路。这种发展路径注定会让行业发展充满波折和动荡，会有很多企业倒在半路上，连带着它们提供的产品和服务一起逐渐消亡。 
 比如文生图的 OG Midjounery，已经无法跟上大厂的技术和训练规模掉队，在今天已经无人问津，而一年多前网上还到处都是用 MJ 生成UI界面设计，UI 设计师要下岗的帖子。 
 强如 OpenAI，也要应对 Google、X AI、Anthropic、Seedance 的冲击和成本管理失控（GPT5 训练量需要消耗一个中等规模城市1年用电量），也在近期放弃旗下的文生视频工具 Sora。 
 模型和硬件端都充满了巨大的不确定性，因为技术的发展是需要“试”出来的，总有前浪会被拍死在沙滩上。至于谁会被拍死在沙滩上就不是站在今天的视角上能回答的。 
 再回到我们今天的主角 Anthropic，这是一家非常务实的 AI 公司，专注于开发有商业应用场景的大模型，是写作、编程等领域中的最佳选择。与之相对的则是类似 ChatGPT、Gemini 这类全能多模态大模型。 
 因为其模型可用性极高，所以从去年开始付费用户量暴增，在今年初预估年化收益已经超过300亿美，不仅用户付费率远高于 OpenAI，且运行成本也远比它低。 
 而这家公司想要年底要上市（估值8000亿美金），产品矩阵的拓展就成为非常重要的资本策略。通过发布 Claude Design，能很好的和 Claude Code 实现联动，完成使用 AI 开发软件的闭环。 
 至于现阶段 Claude Design 够不够用不重要，毕竟故事和逻辑是成立的，作为初版它也有后续升级迭代和优化的空间。在我们目前的试用中，它的生成效果也就那样，但消耗的 Token 量（你最好确保自己有个足够鼓的钱包）…… 
 Claude Design 有新上市的光环风头正盛，那是否还记发布也还不久的Google Stitch、OpenClaw、FigmaMake、Rive、Pencil 等等。 
 再往前是否还记得使用 StableDiffusion 训练 Lora 和 Checkpoint 的工作流？ 
 工具一直在更新，即使今天还很粗糙。但用发展的眼光看问题的话，终有一天它生成的质量会足够高足够智能足够可控，甚至通过脑机接口 BMI 实现所想即所得的水平（又到了做梦的时候）。 
 但是，那天是什么时候？ 
 以及，和你现在设计的界面有什么关系？ 
 作为设计师一定要搞清楚的逻辑，就是 AI 生成界面和图像的过程，只是一种工具的发展和进步，提升了设计师的设计效率。就像人类发明了电脑和软件，取代了尺规作图，同样也大幅度降低了设计的效率和成本，但设计师这个岗位并没有消失。 
 因为使用工具的人之间的差异，会直接影响最终的输出效果，就像用同一个 AI 生成工具你可能就是做不出网上高赞帖子的案例，都用美图秀秀不用 PS 去 P 图你也 P 不过有一定专业积累的摄影师。 
 而且任何工具使用都有门槛，就一个简单的 Claude Design、Google Stich 安装和使用都需要学习和适应，生成结果还要做大量的调节和校准，这些就不是工作量，老板和产品经理也一定能闲的每天学习新工具并且直接自己干？ 
 我相信有很多公司现在都很热衷于拥抱 AI，强行推进 AI 的工作流，从企业战略和发展的角度来讲是能够理解的，但在实际执行层面上却不会像想的那么美好。其中最大的问题就是前面提到的整个产业的不确定性，产品还都有各自的局限。 
 每个团队在搭建 AI 工作流的时候需要做技术的选型，以及围绕自己的实际生产场景优化工具的使用方式。往往为了一个简单的工作结果就要耗费大量的精力去搭建工作流，而这个工作流很可能是一次性的，因为之后的工作场景发生改变这套工作流就用不了了。 
 有些同学可能心态好点觉得这种过程可以收获宝贵的经验，但最不幸的，就是工具的升级和更新（甚至关闭）可以直接颠覆掉原来的做法，让原先的成果毫无意义。 
 在我朋友和学员中普遍得到的反馈，就是在稍微复杂的专业场景中，强行适配 AI 的结果往往只有增加工作量，原先目标的降本增效，实际导致了更长的工作时间和更晚下班，从去年到现在在不同 AI 工具中做研究疲于奔命。 
 换个角度思考，其实就是工作过程变成今天用 PS，明天用 Figma，后天用 Excel 或 PPT 进行设计，每一阵子换个工具设计相同的东西，而它们又不能解决设计过程中最复杂的决策部分。 
 也就是我们一直在使用充满不确定性的工具去寻求相对确定的结果。 
 这就是今天 AI 工具在 UI 行业中应用的真相，不管我们怎么用逻辑还是格局去判断未来 AI 要实现一切，但起码不是今天和最近，就是变得更强了，也依旧需要有人去操作和控制。 
 而我们现在要做的是关注最终输出的结果，而不是和别人炫耀 Figma 用的好，还是 Skech、XD、即时、Pixso 用的好。等到市场竞争格局基本确认，通用的工具也普及以后，再去掌握和学习也不会低人一等（门槛不会高到哪里去）。 
 相信很多同学关注到所谓的 AI 直接生成代码和产品的能力，觉得以后不仅不需要设计也不需要开发。所以最后再留个开放性的问题给大家思考： 
 为什么你们觉得那些开发出来的小破产品，投入到市场上有人买单？如果不行，那到底缺的是什么？ 
 结尾 
 其实这两年一直有为什么我不去做 AI 相关教程的提问，说到底就是因为自己试用完以后实在没办法得出它们具有普适性的结论。做教程只能强行圈定一个范围和场景去讲，而这些场景基本都毫无实战价值。
```

## 6. 面试官是如何审视你的作品集？·上篇

```
原创 酸梅干超人 2026-04-16 22:34 上海 
 做作品集最大的问题不在于怎么做，而在于怎么 “说服” 作品集的观众。如果不能围绕这个目标去执行，我们就只能朝着错误的方向推进，事倍功半。 
 所以在分享作品集制作思路之前，我们首先要讨论的，就是作品集是怎么被 “查看” 的。先确定最终的目标和规律，再开始学习怎么实践。 
 作品集筛选的基本步骤 
 项目包装/展示，本质上就是用图文结合的模式，更生动、直观、“优雅”地展示你的工作成果以及设计能力。 
 也就是说，项目包装的主旨，是为了 —— 更有效得展示项目 。想要达成这个目标，是由观看者来决定的，只有他们觉得好才是真的好。 
 而作品集的主要观看对象是招聘企业的成员，包括企业主、人事、产品经理、程序员、在职设计师等等， 而不是设计平台里的其它设计师或你的设计群群友 。 
 换句话说，作品集实际面向的是具有真实工作经验，甚至有丰富项目经验的人，要求高眼光毒。但很多新人对待作品集就像对待大学期末作业或毕业论文，以为只要做出来能应付过去就行了，这显然无法满足实际的要求。 
 所以我要从招聘者的角度，来分享一遍观看作品集的顺序和要求有哪些。 
 步骤1：作品集初筛 
 首先我们知道现在招聘市场是买方市场，供远远大于求，一个招聘岗位往往会收到很多简历/作品集投递，我们肯定没有耐心一份一份认真看过去。所以要做的第一件事就是初筛，排除掉质量差的并筛选出质量最好的几份。 
 怎么初筛？就是打开作品集文件后快速翻页扫过去，只要扫过前几页，就能对作品集质量怎么样有个大致的判断。这里的质量主要指的是作品集整体视觉和审美水平的优劣。往往只要看到封面或者前几页就可以做判断，太差的直接关了不用往下看下去。 
 作品集得第一印象远比大家想的重要，从封面到前几页的设计质量只要一垮，然后就没有然后了…… 
 步骤2：作品质量检查 
 被选出来的几份不代表全部过关了，还需要再进一步筛选，减少面试人数提高招聘效率。所以接下来就要进入更细致的检查环节。 
 在这个阶段，主要检查的对象就是界面设计水平，也就是展示出来的界面的视觉质量。通常我们会直接拖动进度条到界面展示的部分（忽略文字分析），聚焦在界面案例进行审视。 
 视觉质量好就是好，差就是差，和包装、分析内容没有关系，所以可以非常快的判断对方的真实设计水平，并筛选出最优的几份，进入到后面的面试邀请环节。 
 如果几份作品集界面设计水平基本一致，那就会从视觉层进一步深入，查看界面中的产品功能、布局框架、交互逻辑。要注意的是，这是通过静态界面图来分析你的设计思路，而不是看你旁边的文案是怎么解释的。 
 对于有丰富项目经验的观看者来说，只看界面就能很快判断你的设计意图以及发现界面中存在的问题点，看旁边的分析和介绍文案反而会影响查看的效率。 
 如果界面功能、交互存在很多明显的错误（比如漏步骤），或是让人难以理解摸不着头脑，就会被排除，并筛选出最终得几份。 
 步骤3：项目思路检查 
 前面先看作品集的包装，然后看作品的质量，那么很多人关心的项目分析、文案内容就不用看了吗？这个问题是最耐人寻味的。 
 首先前两个阶段中，观看者是基本不太会看分析和文案内容，原因很简单 —— 没必要。一方面大多数分析文案质量过差滥竽充数，另一方面看起来也很浪费时间和精力。先通过包装和视觉筛选就够了，因为这是必要条件。 
 真正会看项目分析和文案的场景只有两种情况，第一种是作品集从包装到界面的质量都极其优异，吸引了观看者在一开始就认真观看下去。 
 另一种情况，就是在正式进入面试环节之前，面试者要在面试阶段进行提问考验应聘对象，除了常规问题外， 更多的面试问题会从作品集中挖掘 。也就是先认真看你项目介绍和分析，从中找出疑惑点或分歧，再在面试阶段提出。 
 但这个步骤不是必须的，因为前面查看界面阶段就可以形成大量的问题了，面试官偷懒一点的可以等到面试让对方自己现场讲解，再从中找出问题来提问（发难）。 
 也因为面试过程中一定会密集出现和项目设计思路有关的问题，所以让大家意识到项目分析的重要性。但多数人给出的错误的应对方式，那就是在作品包装内容中添加大量的项目分析和设计思路的文案。 
 不是说作品集包装里不能加分析文案内容，但它一定要满足下面这些条件： 
 作品集包装排版审美质量过关 
 展示项目的界面设计质量过关 
 文案措辞简练清晰不讲套话 
 不满足这些条件，即使你花费了再大的力气甚至和 AI 对线三天三夜生成的分析文案也没用，这些言之无物的文字只会拉低作品集整体的评价造成恶劣的负面影响。 
 作品集观众的特征总结 
 作品集就像做一个完整的项目，你需要先总结用户的特征才能制定决策并展开设计，下面我们就再总结一些作品集观众的特征，帮助大家更好地思考作品集应该怎么做。 
 1.招聘精力分配的欠缺 
 首先要说的问题是最重要也最容易被忽视的问题，那就是观看作品集的人大多也是员工（牛马），他们是有本职工作要做的，往往工作做到一半还要接收从 HR 那边推来的一连串作品集。在这种场景下会有闲情逸致慢慢浏览作品集和研读里面的内容吗？ 
 可以很肯定的说多数相关负责人看这些东西都是不耐烦的，初筛囫囵吞枣是一种常态。只有自己时间多，且招聘岗位对自己的关联足够大才会重视起来。 
 所以如果作品集不能很好的展示重点，且不能很流畅的让人观看，那甚至都没办法被看完就会被关闭。 
 2.项目实战经验丰富 
 前面也说过招聘相关负责人往往实战经验很丰富，所以对应聘者应该具备哪些能力有很清晰的认识。越是专业的招聘者，对专业技能的要求就越严格，可以很轻易的看出展示项目中存在的问题。 
 越是想要卖弄，避实就虚的做法就越容易被鄙视，或招致反感。比如对一些很基础的交互或布局大夸特吹，或是引用很多理论和模型但和设计毫无关联。越想要展示这些东西，在他们看来就显得越浮夸和心虚。和经验丰富的专业人员卖弄基础知识，就像一个小孩和中年人讨论人生与婚姻…… 
 3.互联网黑话/套话厌恶 
 招聘者作为牛马，平时肯定也要应付领导或者输出报告、准备述职等，在工作中应用黑话套话是必然的，但不代表他们本身就喜欢那么做。 
 所以工作里应付归应付，自己招聘的时候就更不喜欢再看到这样的内容。而很多初级设计师总喜欢把这些黑话看得很高深专业，比如什么抓手、破局、链路、锚点、触点、异化、渗透等等，但凡有点工作经验都会对这些词汇免疫。尤其是分析内容逻辑性不足的话，影响更恶劣。 
 4.专业性要求不一致 
 不同招聘环境和岗位要求，招聘者的侧重点就不同，有的更重视交互能力，有得重视视觉能力，有得重视交付协作能力。 
 有时候作品集没通过不一定完全是因为综合质量不足，而是在特定的领域没达到这家的预期没有匹配上而已。所以要综合分析作品集的侧重点和局限性，不要认为它可以通吃所有情况，也不要因为不配对的失败而感到泄气。 
 5.作品集观看经验较多 
 有招聘经验的负责人都看过非常多的作品集，尤其是设计师的话，看过的作品集案例就更多了。所以会对作品集中出现的一些常见设计套路、视觉元素了如指掌。 
 而很多新手会为了偷懒去网上找别的作品集模板，且按新手匮乏的见识与经验，找到和使用的都是最热门最常见的那些（找模板都是偷懒），所以打开扫一眼就能知道用的是谁的，以及出处在哪里，这在稍微专业一点的招聘环境中是绝对过不了的。 
 总结起来，作品集输出一方面是需要专业性的支撑，另一方面就是要拒绝敷衍和偷懒的心态，想靠抄近道的方式去面对越来越严格的招聘环境是行不通的。 
 结尾 
 同时强烈建议还在准备做作品集的同学，多咨询一些有招聘经验的朋友，问问他们从一开始筛选作品集到最终面试都是怎么看的，以及中间的心态是什么样的，会给你们更好的佐证。
```

## 9. 交互设计必备基础知识，一篇讲清

- 日期: 2026-04-08 22:34
- 链接: https://mp.weixin.qq.com/s?__biz=MzI0Njc5NzM5OQ==&mid=2247506202&idx=1&sn=487ce1b9363bacc835b76da6510aeaf8

```
原创 酸梅干超人 2026-04-08 22:34 上海 
 最近课程进入交互阶段，着重围绕交互的具体实操和应用做解释，但没有太多涉及 基础知识的扫盲和解释 ，所以今天重新做一遍整理，分享和交互有关的基础信息。 
 交互设计的基本认识 
 学习交互就先认识一遍交互设计是什么，它和 UI 界面设计，以及 UX 体验设计有什么区别。 
 UI 界面设计最容易理解，就是把 UI 界面最终视觉效果设计出来的过程，包括排版布局、色彩填充、图标设计等具体操作。UX 体验设计则是以提升用户体验为目标的设计过程，囊括了和设计相关的一切分析与实操步骤，既包含了界面设计也包含了交互设计。 
 而交互设计，则是制定界面使用方法和功能操作流程的过程，比如制定外卖下单的操作流程、任务提醒的设置步骤、AI 对话的使用方法等等。 
 如果只看介绍不够直观，想要更深刻地认识什么是交互的话，就可以使用不同应用去实现同一个目标，比如预订车票、下单外卖、美化照片，感受它们之间操作上的差异，而这种差异就是交互上的差异，也是交互设计的结果。 
 想要区分界面设计、体验设计、交互设计最简单的方法，就是认识设计的出发点。如果仅仅是基于产品需求把界面视觉做出来，那就是 UI 设计。如果目标是为了追求更好的用户体验，且把界面视觉作为提升体验的手段之一去完成，那就是体验设计。如果是为了制定产品界面或功能的操作方法，那就是交互设计。 
 虽然设计方式有区分，但它们并不是完全孤立的，而是相互之间都有很大的重叠与影响。首先体验设计同时包含视觉设计与交互设计，在正常的设计流程中，会先完成交互设计再做界面设计，交互是视觉的前置条件。 
 交互设计的结果必然会影响用户体验也会影响界面设计，而界面设计必然也要考虑交互操作与用户体验。总结起来就是做交互的时候不会只为了操作顺手就让界面布局变得特别违和，只做界面视觉时也不应该只顾着好看就完全不管操作的逻辑和可用性。 
 用一张图来概括它们之间的关系： 
 交互设计不是界面设计，所以设计的具体对象和交付内容也不同。交互设计主要设计的对象是线框图/原型图，并通过添加连线和备注文案来表达更具体的交互信息与逻辑。 
 很多新人和外行会把交互设计等同于制作可交互原型，比如使用 Axure 制作线框图并添加交互，或是使用 Figma 创建交互连线和添加动画效果等等。这些大多属于无用功，因为交互设计要交付的是具体方案和明细内容，怎么做看起来直观且高效就怎么做，而可交互原型很难满足这些核心要求。 
 在真实的项目中不一定会添加一个独立的流程节点给交互设计，就是因为在交互比较简单的需求中，完成界面视觉设计的同时就能“顺便”把交互方式也定了，这样效率更高。 
 只有在产品功能、逻辑非常复杂，且交互方式影响重大时，才会独立展开交互设计，并在确定交互方案以后再推进界面的视觉设计。这么做才能更好的减少改稿次数，提高设计的整体推进效率。 
 交互设计在不同的团队和不同的项目中会有不同的要求，可能是显性也可能是隐性的，但它必然是每个 UI 设计师工作中都必然会涉及且做出决策的内容之一。 
 交互的相关名词扫盲 
 简单理解完交互以后，下一步我们就围绕交互中最常见的一些名词来做解析，帮助大家更好的扫盲交互的基础认知。 
 人机交互 Human–Computer Interaction 
 是人类和机器或软件进行互动的过程总称，比如开关机、输入文本、控制鼠标、打开应用等等都是人机交互。软件界面的交互是人机交互的一部分，是它的子分类之一。 
 交互设计 Interaction Design 
 如果没带“人机”前缀，则通常指的是对软件界面交互方式的制定过程，也就是上文解释的内容。 
 交互界面 User Interface 
 本质上就是 UI 界面，是可以被交互并实现相关反馈的可视界面。而一张静态的图片或是正在播放的视频图像就不是交互界面。 
 交互原型 Interactive Prototype 
 分为静态交互原型和动态交互原型，静态交互原型指的就是用于表达交互方式的普通线框/原型图，而动态交互原型就是为前者添加交互并可以上手进行操作。 
 交互文档 Interaction Specification 
 记录项目相关交互信息和逻辑明细的文档，方便团队内部进行传递和留档记录。 
 交互流程 User Flow 
 用户在软件内完成某个目标或使用某个功能所经历的步骤，比如从首页完成外卖的点单或是预定酒店/机票等。 
 交互事件 Interaction 
 交互事件是用户进行单次交互操作并获得对应反馈的整个过程。包含了交互方式、交互行为、交互反馈、交互动效等下级内容。 
 在 Figma 中，就是以事件  Interaction 为单位添加交互效果。 
 交互方式 Interaction Trigger 
 界面中元素被触发的方式类型，在不同客户端和系统会有不同的触发方式，比如手机上以手指交互为主包括点击、长按、重压、拖拽等，PC 网页端则以鼠标交互为主包括点击、双击、拖拽、快捷键等等。 
 在 Figma 中交互面板的  Trigger 就用于设置交互方式。 
 交互操作 User Action 
 用户在使用界面过程中应用的具体操作，同样也包含点击、长按、双击等操作类型。它和交互方式的差异，在于交互方式是软件内置好的规则，而交互操作是用户的行为，比如界面中的停止按钮要长按但用户单点没有触发，就是交互方式和交互操作错配，两者不能等同起来。 
 交互反馈 System Feedback 
 是用户执行某个操作后呈现的结果，比如打开页面、关闭弹窗、完成支付、发送消息等等。 
 在 Figma 中交互面板的 Action 就用于设置交互反馈的类型。 
 交互动效 UI Motion 
 是系统对交互反馈进行呈现的动画效果，比如新页面从下方推入、弹窗从上方滑入，气泡从点击位置弹出等等动画效果。 
 交互动效即 Figma 中交互面板的 Animation 选项设置。 
 交互体验 Interaction Experience 
 交互体验是用户体验中的一部分，是用户在使用产品功能过程中形成的体验，只针对产品使用和操作的部分，不包含如定价、客服、售后等其它因素。 
 交互可用性 Usability 
 指的是设计出来的交互流程或事件能否被用户正常使用，且使用起来的难易度如何，是用于评价交互设计方案好坏的主要标准之一。 
 结尾 
 交互是 UI 设计师最重要的产出内容之一，也是无法被 AI 取代的部分。尤其在 B 端领域，交互的权重远大于视觉和动效等基础能力。
```

## 10. 讲讲UI设计中B端和C端的职业选择

- 日期: 2026-04-02 22:34
- 链接: https://mp.weixin.qq.com/s?__biz=MzI0Njc5NzM5OQ==&mid=2247506186&idx=1&sn=4e9d1e58f5105edde800a90914a2ec15

```
原创 酸梅干超人 2026-04-02 22:34 上海 
 最近遇到越来越多针对 C 端和 B 端选择的问题，尤其是去大厂面试的学员，有很多投的是 C 端但实际面试的岗位却是 B 端，所以这次简单解释这两端的选择问题，以及如何进行前期的准备工作。 
 B/C 端工作职能认识 
 B 端和 C 端设计虽然都是 UI 设计，但在岗位职能和技能要求上却不是完全一致的，所以它们既是同一个职业类型，也是不同的岗位。 
 在主流的认识中，C 端就是做移动端 UI 界面的，B 端就是做网页管理系统的，但这只说对了一半，并不能概括所有情况。 
 C 端主要负责的是面向终端用户/消费者的产品界面设计，以移动端 APP、小程序为主导，还包括运营活动设计、C 端网页设计、消费型数码产品界面设计等。 
 B 端主要负责的是面向商业用户/打工人的产品界面设计，以 PC 端网页管理系统为主，但也会包含移动端产品设计、数字大屏设计、商用定制产品界面设计等。 
 面向场景、产品、用户类型不同，也就导致实际工作内容上存在差异，总结起来主要有以下三点： 
 设计规范、技巧、思路不同 
 交互形式和复杂度不同 
 设计流程和关注点不一致 
 差异点1：设计实操 
 第一个差异就体现在设计实操上，虽然 C 端和 B 端都使用 Figma 做设计，但是界面使用的设计规范不同，结构不同，参数不同，这是需要有一定的学习和训练进行适应的。 
 同时 B 端界面空间远比 C 端大，可容纳的信息也更多，更注重横向的排版。且在界面适配方面，C 端面向的不同手机屏幕规格差异不大，适配简单，而 B 端界面可以被任意缩放（调整窗口大小）就导致要重点考虑响应式和自动布局规则。 
 再到更细节的视觉层面，C 端产品面向的消费市场往往需要呈现出更丰富多样（花哨）的视觉效果，包括渐变、遮罩、异形轮廓、设计字体、IP 形象等等，这就需要用到不同的设计技法和创意。而B端界面相对严简约，往往只应用简单的软件最基础的样式功能。 
 总结起来，在设计实操方面两者使用的设计软件相同，但设计规范不同，C 端更注重视觉样式的表现力与创意，而 B 端更注重版式的合理性与适配性。 
 差异点2：交互操作 
 第二点就是两者交互形式有较大差异，首先就是使用的设备不同，手机使用手指进行点按，界面小但交互面积大，所以可交互元素上限就低。而电脑端使用鼠标指针进行点击，界面大且指针交互更精确，所以可交互元素上限就高。 
 同时，B 端面向生产力环境，功能和操作要求远远比 C 端复杂，也就导致了 B 端设计过程中需要耗费更多精力在交互上，确保功能的交互严谨有效且能被用户正常使用。 
 总结起来，交互方面两者交互形式不同，C 端交互相对简单且成熟案例参考较多，B 端交互相对更复杂且难以找到参考依赖设计师的发挥，门槛更高。 
 差异点3：设计流程 
 在项目设计流程规划中，C 端产品主要由用户驱动迭代，所以会更重视对用户的理解，可以添加丰富的用户调研与分析步骤，为提升用户体验做足准备。而 B 端产品则由业务驱动，首先是实现业务的功能点，再考虑用户体验，所以会花费更多时间在业务的分析上。 
 所以在 UI 设计实操外，C 端的前期分析主要围绕体验做分析，而 B 端主要围绕业务做分析。体验分析只是细节多但并不复杂，而业务分析的难度则由行业决定，很多 B 端行业的业务逻辑极其复杂，比如医药、航天、船舶、物流等，这构成了 B 端设计中最大的难点与门槛。 
 以上三点就是 C 端设计和 B 端设计的主要差异，这种差异已经可以构筑起两个岗位之间的壁垒，虽然不算太高，但也需要经过一定的学习和训练才能胜任另一端的工作。 
 B/C 端岗位发展分析 
 了解岗位之间的职能差异， 下一步就要谈谈很多同学关心的发展前景的问题了 ，也是促职业选择的核心因素。 
 如果从宏观层面来说，C 端岗位的权重和上限是更高的，因为 C 端项目的规模普遍更大，且 C 端更关注体验与视觉的表现，所以 C 端 UI 设计师有更高的发展空间和薪资上限。与之对应的，就是 C 端职业要求也更高，门槛更高。 
 而 B 端岗位相对来说权重和上限较低，因为 B 端项目规模和盈利能力很难和 C 端相提并论，且 B 端更注重业务的实现，很多场景下只要“能用”并不是特别在意设计的好坏。所以 B 端在入门上要求和门槛就更低。 
 总结起来，就是 C 端上限更高门槛更高，而 B 端上限较低门槛也低。这个结论并不是告诉大家 C 端就一定比 B 端优越，因为高出的那部分上限只存在于职业中高级阶段，对职业初期和新人来说没有太大的影响。 
 要从哪个方向作为起点一方面看个人的兴趣，另一方面也要根据实际的求职情况处理，下面我们就围绕这两点做一个简单的说明。 
 岗位与个人兴趣 
 前面分析过 B 端和 C 端设计在岗位职能上的差异。C 端更重视觉、技法、创意要求，对于喜欢跟随视觉前沿和发挥创造力输出美观设计产物的同学来说显然是更合适的。 
 而 B 端更重视业务、逻辑、规范，不需要很复杂的视觉技法和创造力，相对 C 端来说更“死板”，适合不喜欢创作且更喜欢逻辑推理的同学。 
 更深一点去总结， C 端设计看似自由，本质是要面对市场与用户的双重 “不确定性” ，设计师得持续输出新的方案来试探不断变化的市场环境与用户偏好。这个过程是充满“玄学”的，不是光努力或从专业角度上认为优秀的产出就能带来良好的结果。 
 而 B 端设计看似死板， 但它要实现的目标和成果的检验是非常 “明确的” ，只要设计师能给出合理的设计方案，说服管理层并配合开发实现出来，就能完成设计目标。 
 所以喜欢哪个方向需要由个人性格决定，喜欢面对未知挑战征服用户的话就会喜欢 C 端，喜欢确定性能用专业能力实现工作目标的话就会更喜欢 B 端。 
 求职场景与际遇 
 虽然有时候我们会先做出选择，但实际求职过程中不一定会按我们的意志展开。 
 比如求职过程投递的是其中一个岗位，结果只受到另一个岗位的面试，或是入职是目标岗位，结果实际工作做的却是另一端的工作，这都是非常普遍的情况。 
 这里就必须要结合大环境再做一点讨论，那就是 C 端岗位一般都有做 B 端的需求，而 B 端岗位做 C 端的需求就相对较少。 
 因为专门做 B 端项目的团队，很少会去输出移动端，即使做了也是简化版，所以需求量并不大。但任何成熟企业还是 C 端产品都会有 B 端系统和平台的使用需求，小一点的公司可以直接买 SaaS 服务，大一点的公司往往需要自研开发。 
 比如游戏公司会有自己的游戏管理后台，社交应用会有自己的审核平台，电商应用会有自己的 ERP 系统。这些设计需求不一定会独立招聘 B 端的设计师而是直接让目前的 C 端设计师完成，不仅效率更高沟通成本也更低。 
 尤其是目前大厂产品招聘的 “产品设计师”，往往就要负责一个固定功能模块客户端和后台管理系统的一体化设计服务，也就是 C 端和 B 端都要做。 
 还有种情况，就是面向校招的场景，如果是 B 端岗位开 HC，但是学生群体里很少有 B 端实习还是设计经历的，所以也就会从 C 端作品集里挑人来面 。 
 比如最近有个做 C 端作品集的学员面上了腾讯，结果去的岗位是游戏中台的 B 端设计师。 
 这要表明一点就是，虽然两个岗位之间是有一定界限的，但企业招聘初级岗位来说这个界限往往并不会划分得非常清楚（认为你都得会）。同时在实际工作场景里，B 端设计师不一定会做 C 端的产品，但 C 端设计师一定会有 B 端设计的需求。
```
