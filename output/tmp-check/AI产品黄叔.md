## 2. 五一我没开电脑，但我的 Agent 团队没放假 | 如何用Hermes Agent移动办公

- 日期: 2026-05-06 12:12
- 链接: https://mp.weixin.qq.com/s?__biz=MjM5ODU1MzQzOQ==&mid=2451429544&idx=1&sn=5f8208456b628c3d96c6d51b180b4753

```
原创 AI产品黄叔 2026-05-06 12:12 北京 
 五一去新疆之前，我还是把 MacBook Air 塞进了包里。过去很多年，我都有一种内容创业者的条件反射：出门可以少带一件衣服，但电脑不能不带。 
 结果这次五一，从天山天池、赛里木湖到那拉提，我每天不是在路上，就是在景区，回到酒店基本累到直接睡。假期结束后我才发现一件挺离谱的事： 
 我的 MacBook Air 一次都没打开过！！！ 
 但，我的工作并没有完全停。 
 微信群摘要还在生成，内容选题还在筛，口播稿还在改，竖版配图还在出，每日复盘还在跑，甚至几个不同的 Agent 已经开始在同一个飞书群里互相协作。 
 当时我脑子里冒出来的不是“移动办公真方便”。 
 更准确地说，是另一种感觉： 
 🌟 我不是在手机上办公。 
 我是在手机上指挥一群 Agent 办公。 
 所以这篇不写安装教程。 
 网上已经有不少教程了，而且说实话，真要讲配置，一篇文章也讲不完。 
 我更想记录的是这次旅行给我的一个测试： 
 人不在电脑前，只剩一部手机，一个人加一群 Agent 的内容系统，到底还能不能慢慢转起来？ 
 能。 
 但不是那种科幻片式的自动化。 
 01 手机没有变强，它只是接上了 Agent 
 先交代一下我的基本链路，免得后面读起来太玄。 
 前台是飞书。 
 中间是 Hermes Agent。 
 背后接的是 Codex，自带 GPT-Image-2 最近最火的生图能力。 
 我在手机上看到的只是一个飞书聊天框。 
 但干活的不是手机，是后面的 Agent。 
 这件事让我对“移动办公”这个词有点改观。 
 以前说移动办公，通常是人在高铁上、咖啡馆里、酒店房间里打开电脑，继续敲字。地点变了，活还是自己干。 
 这次不太一样。 
 我在手机上不写长文，不排版，也不整理资料。手机只负责发一句判断，或者补一句要求。 
 剩下的，交给 Agent 先跑。 
 说白了，手机还是那个手机。 
 只是它后面接了一组能干活的人，或者说，像人的东西。 
 02 后来我不敢再把所有事塞给一个 Agent 
 这套东西不是一开始就规划好的。 
 最早我也只有一个个人助理 Agent。 
 待办、选题、文章、图片、群消息、资料分析，什么都往里丢。 
 刚开始很爽。 
 因为你终于有了一个随叫随到的“人”。 
 但时间一长，我发现它开始变得奇怪。 
 它什么都知道一点，什么都想插一手。今天帮我做日程，明天又要理解少儿 AI 选题，后天还得整理群消息。 
 看起来全能，实际很容易变成我脑子里的第二个垃圾堆。 
 上下文会串，技能也会串。 
 它一会儿帮你处理日程，一会儿帮你写少儿 AI 选题，一会儿又去整理微信群信息。看起来很全能，实际容易变成老板脑子里的第二个垃圾堆。 
 所以我后来把它们拆开。 
 Agent 更像什么岗位 主要负责什么 
 个人助理 Agent 日常运营中枢 信息、待办、复盘、资料整理 
 内容总监 Agent 内容负责人 选题、脚本、配图、方法论沉淀 
 蓝皮书 Agent 专项研究员 长周期资料与专题内容 
 Agent 群聊 临时项目组 上下文同步与跨角色协作 
 拆开之后，我才觉得它有点像一个小团队了。 
 03 个人助理先替我接住杂事 
 这是我最早创建的 Agent。 
 我还给她取了一个名字，叫夏夏Hermas。 
 为什么叫 Hermas？ 
 因为那会我还不熟 Hermes，甚至把名字都起错了。后来为了让自己印象深刻，就一直没改。 
 这个 Agent 没有负责什么特别酷的事。 
 它主要接那些我很容易拖掉、但拖久了会出问题的东西。 
 微信群信息。 
 每日待办。 
 长资料整理。 
 临时冒出来的调研、海报、问卷、复盘。 
 这些事单独看都不大，却它们最磨人。 
 微信群摘要：我不用再人肉刷屏 
 我们每个人都在很多微信群里。 
 问题是，群聊里既有高价值讨论，也有大量闲聊、转发和情绪。 
 如果全靠人肉刷，我一天很容易被信息流拖走两三个小时。 
 所以我让 Hermes Agent 每天定时处理群消息，把和 AI 有关的高质量讨论挑出来，提炼成摘要，再自动生成图片。 
 我每天只要快速扫一眼卡片，就知道昨天有哪些值得回看的信息。 
 有价值的，再回到对应群里找源头。 
 以前我刷群，最痛苦的不是信息多，而是不知道哪一句有用。 
 现在至少先有人帮我筛一遍。 
 不完美，但够用了。 
 图片生成直接走 Codex 内置的 GPT-Image-2，我也不用再切一个生图工具。少切一次工具，人就少烦一次。 
 事件梳理：把碎片文章压成一张脉络图 
 另一个常用场景，是把一堆文章压成一个事件脉络。 
 每天能看到很多公众号文章。 
 单篇看都还行，但几篇放在一起，费劲的是搞清楚：它们到底在讲同一件事的哪个侧面？时间线怎么走？谁在争什么？我能从哪个角度写？ 
 最近有个教育事件比较火，我把相关链接都丢给它。 
 它把时间线、关键节点、不同观点和可写角度整理出来，还顺手生成了图。 
 我再看这件事的时候，就不用从一堆文章里重新扒线索。 
 每日复盘：把 Todo 从许愿池里捞出来 
 还有一个用法，我现在越来越离不开：每天对一下目标。 
 我会把 todo 丢给它，让它帮我拆事情、排时间，有些事还会顺手设个提醒。 
 这听起来很小。 
 但独立创作者最麻烦的地方，不是没想法，是没有人帮你清算行动。 
 Todo 如果没人清算，很快就会变成许愿池。 
 尤其我这种 P 人，一天冒出十几个想法，看起来很忙，晚上复盘发现都只是脑内开工。 
 所以我需要一个 Agent 每天问我：这件事到底推进了吗？还是你只是又开了个新坑？ 
 第二天的自动复盘也挺有用。 
 它会把前一天的事情拉回到几个长期目标里看：内容增长、少儿 AI、社群、课程、Agent 系统。 
 普通聊天机器人回答问题。 
 这个助理至少会提醒我：别又被新想法带跑了。 
 播客转 PDF：9 小时内容变成可阅读资产 
 还有一个任务，我没想到它能跑完：播客转 PDF。 
 最近我在听南老师的播客，一期 9 个多小时，内容很密。 
 音频当然可以听。 
 但对我来说，如果一段内容以后还要反复用，它最好能变成可检索、可标注、能放进知识库的东西。 
 所以我就问它：能不能整理成一份方便阅读的 PDF？ 
 过去这类任务我用 Claude Code 跑过几次，都不太顺。 
 这次 Hermes Agent 把 32 万字处理完了。 
 后面我又让它读取 shownotes 里的时间戳和图片，再塞进 PDF。 
 这一步跑通之后，我有点意外。 
 因为它已经不是“帮我写一段文字”了。 
 它是在把一段很长、很散的一次性内容，整理成以后还可以继续用的资料。 
 我关心的其实是这个。 
 家长调研：从一个想法到问卷和海报 
 最近我也在想一件更私人的事：怎么给我 9 岁的女儿做 AI 启蒙。 
 它先是一个爸爸的问题。 
 顺着这个问题，我想做一次少儿 AI 家长调研，看看其他家长到底在焦虑什么、期待什么、误解什么。 
 以前做这类事，要拆成好几步：想问卷、写说明、做海报、改文案、发出去。 
 这次我基本是口头交代给个人助理 Agent。 
 它帮我出问卷，也帮我做海报。 
 因为整个过程都在飞书里，来回改起来比较省事。 
 类似的事还有不少。 
 写到这里，我对个人助理 Agent 的定位反而更清楚了。 
 它不是替我完成某个很惊艳的大项目。 
 它主要替我接住那些碎片工作。 
 少一点漏事，少一点反复切工具，少一点脑子被信息流拖着走。 
 04 内容总监先从新账号开始练手 
 第二个角色，是内容总监 Agent。 
 我最近在起一个新号，方向是少儿 AI。 
 这件事背后有个很私人的动机： 我想陪自己的 9 岁女儿更好地理解 AI，也想看看普通家庭到底该怎么面对下一代 AI 教育。 
 如果你也有孩子，可以关注，多多交流视频号：黄叔陪娃学AI 
 从工作流角度看，这个新号刚好适合拿来测试： 
 一个内容方向从 0 到 1，Agent 能陪我搭到什么程度？ 
 新账号不是副业，是一次内容系统测试 
 所以我单独建了一个内容总监 Agent。 
 它不管我的日常杂事，只管内容。 
 选题、观点、脚本、配图、风格、方法论，都放到它这里。 
 我现在越来越觉得，某一次“帮我写篇稿”其实没那么值钱。 
 更麻烦也更值钱的是：它能不能慢慢懂我的判断。 
 我怎么选题？ 
 什么样的开头我会觉得太虚？ 
 哪种脚本像我，哪种一看就是 AI 味？ 
 哪种图适合这个账号，哪种像培训班海报？ 
 这些东西能留下来，内容总监 Agent 才不会每次都像第一天入职。 
 每日选题：从早晚报里筛少儿 AI 的信息差 
 它先从每日选题开始。 
 我让它每天读 Mac mini 上跑出来的早晚报，再从里面挑少儿 AI 方向可能能写的题。 
 这里最难的不是找热点。 
 AI 圈每天都有热点，但很多只是大人之间的自嗨。家长看了不知道和自己有什么关系，孩子也用不上。 
 它要帮我判断：哪些东西能翻译成家长听得懂、孩子真的能尝试的内容。 
 比如孩子用 AI 做作品、建站、做游戏，这些不是拿来蹭的素材。 
 我更关心的是它们背后那条线：以后孩子的竞争力，可能不只是多刷一套题，而是能不能从真实问题出发，借 AI 做出东西。 
 这条链路现在还粗糙，内容源和选题质量都有不少空间。但它已经能跑起来。对新账号来说，先有一个能跑的版本，比憋一个完美系统现实得多。 
 内容打磨：从选题到脚本，再到竖版配图 
 第二件事，是打磨内容。 
 我会围绕一个选题，和内容总监 Agent 来回聊。 
 先定观点。 
 再调结构。 
 再改逐字稿。 
 最后让它根据定下来的脚本，生成几张竖版图，后面可以插进短视频里。 
 这个过程和“让 AI 直接写稿”差别很大。 
 我给方向，它给一堆可能性。 
 我再挑、删、改、定稿。 
 它很少一次到位，但能把我从空白页里拽出来。 
 现在加了“去 AI 味”的逻辑之后，逐字稿还是偏理工男。 
 但已经能用了。 
 对我来说，这个阶段不用装成 100 分。 
 85 分先发出去，看真实反馈，再改下一版。 
 方法论迭代：判断标准要留下来 
 我也在不断改视频脚本的方法论。 
 比如开头要不要先给家长一个具体场景，什么时候可以反常识，什么时候要给判断标准，什么时候不能急着给答案。 
 这些东西如果只躺在聊天记录里，很快就找不到了。 
 写进 skill、方法论和知识库，下次 Agent 才有东西可读。 
 Hermes Agent 和 Skill 接在一起后，我用起来还挺顺手。 
 不用每次重新写一大段 Prompt。 
 我可以边用边改，边改边沉淀。 
 这件事让我越来越确定： 
 如果判断标准留不下来，Agent 每次都像临时工。 
 如果判断标准能留下来，它才会慢慢变成一个熟手。 
 05 把几个 Agent 拉进一个群之后 
 后来我还做了一件有点好玩的事： 
 把几个 Agent 拉进同一个飞书群。 
 一开始只是觉得好玩。 
 但用了几次之后，我发现这不只是“多几个机器人聊天”。 
 不同 Agent 有不同上下文、不同分工、不同技能。放到一个群里之后，至少有两种场景还挺有用。 
 全员同步：让不同岗位先对齐上下文 
 第一种，是同步上下文。 
 有些问题我也不知道该交给谁，就在群里 @ 所有人。 
 个人助理从目标和待办角度看。 
 内容总监从选题和传播角度看。 
 专项 Agent 从资料和研究角度看。 
 有点像把几个人叫到会议室里，先听一圈意见。 
 指定协作：让一个 Agent 帮另一个 Agent 排障 
 第二种，是让它们互相接活。 
 比如我先问个人助理一个信息，再让内容总监把它改成脚本。 
 还有一次内容总监 Agent 报错，我就在群里让个人助理帮它排查。 
 这件事听起来有点绕。 
 但它让我第一次感觉到，Agent 不是只能一对一聊天。它们可以在同一个工作现场里传递上下文。 
 当然，Agent 群聊我现在用得还不多。 
 还有个很现实的问题：经常要 @，我老忘。 
 但我大概能看见那个方向了。 
 未来的个人工作台，可能不是一个超级聊天框。 
 它更像一个小办公室，里面有几个分工不同的 Agent。 
 06 这次旅行给我的信号 
 4 月底，受苏杰老师邀请，我去杭州做了一次龙虾加 Skill 的线下工作坊。 
 当时我就想试一件事：能不能不碰电脑，直接帮大家完成整套配置？ 
 后来有些朋友现场没带电脑，结果也能用手机完成龙虾和 Skill 的配置。 
 那次我已经有点感觉了。 
 但工作坊毕竟还是工作坊。 
 这次五一在新疆，更接近日常生活里的测试。 
 人在路上，电脑没开，手机在手边。 
 我的内容系统还能不能继续转？ 
 速度不快，但能转。 
 我不想把这件事写成数字游民爽文。 
 不是“人在新疆，AI 替我赚钱”。 
 没那么夸张，也没那么轻松。 
 现在的 Agent 团队问题还很多。 
 会误解上下文。 
 会写得像咨询报告。 
 会因为权限和 API 卡住。 
 如果你的工作流本来就乱，它还会帮你乱得更快。 
 所以这次体验之后，我反而没那么兴奋，心里多了一个更具体的判断： 
 一人公司想跑起来，不能只靠“多几个 Agent”。 
 流程得先说清楚。 
 选题、素材、结构、初稿、审稿、配图、发布、归档，每一步怎么走，不能全靠临场发挥。
```

```
 Claude Code 是一个 Agent。

 Skill 是什么？一份存在文件系统里的 Markdown。Agent 需要的时候把它加载进自己的工作记忆，获得 "知道怎么做" 的知识——加载之后，它会持续影响 Agent 的判断和决策。

 第一个证据：Skill 天然可以包含脚本。

 my-skill/ ├── SKILL.md           # 知识：怎么做 ├── reference.md       # 补充资料 └── scripts/     └── helper.py      # 执行：CLI 操作 第一次注意到这个的时候我有点意外。 
 第二个证据：Skill 可以注入 CLI 的执行结果。 
 官方有一种 !`command` 语法。Shell 命令先跑，输出直接灌进 Skill 内容里。比如你写个 PR 审查 Skill： 
 - 变更文件列表: !`gh pr diff --name-only` - PR 评论: !`gh pr view --comments` Agent 拿到的时候，这些已经是 gh CLI 跑完的真实数据了。CLI 的结果，直接变成了 Skill 的上下文。什么意思？CLI 喂 Skill，Skill 消化 CLI。 
 第三个证据：Skill 可以给 CLI 开绿灯。 
 在 Skill 的 frontmatter 里写一行 allowed-tools: Bash(git *) ，Agent 跑这个 Skill 时就能自由调 git，不用每次弹窗确认。相当于 Skill 跟 AI 说：这几把刀我替你磨好了，放心用。 
 三个设计放在一起看，方向很清楚： 
 1 Skill 可以 包含 CLI 操作（scripts/） 
 2 Skill 可以 消化 CLI 输出（!`command`） 
 3 Skill 可以 授权 CLI 执行（allowed-tools） 
 每一个设计都在让 Skill 和 CLI 更深度地咬合。如果官方认为它们是上下级，不会做这种双向嵌套的设计。

 简单判断口诀： 手动三次，就该写成说明书。 
 就像你连点三次同一家外卖，第三次就该把它存进收藏夹了。我做那个信息图 Skill 就是这个逻辑。前两次手动调 CLI 跑通流程，第三次直接封装。

 Skill 做判断，CLI 做执行，左右手交替传接：
```

```
 Context 是唯一重要的事情。 Context 不在你的小红书里，不在你的即刻里。 Context 在你的微信里，飞书里，电脑里。 但其实那些也远远不是全部。 最重要的 Context 其实是在你的心里。 是你对世界的认知，对关系的感知，对事物的判断。 那是你独一无二的灵魂。

 01 
 PART 
 你的 AI 为什么总写不出你要的东西？ 
 你大概经历过这个场景：打开 ChatGPT 或 Claude，输入一段 Prompt—— "帮我写一篇关于 xxx 的公众号文章，要有深度，有案例，语言生动一点……"

 问题到底出在哪？ 
 不是模型不够好。Opus 4.6、GPT 5.4，能力都够强了。也不是 Prompt 不够精，你加了约束、给了示例、甚至塞了几千字参考材料。 
 问题是：你的 AI 没有足够的 Context，也没有你的审美。 
 它不知道你上周写过类似的文章、风格是什么样的。不知道你的读者是有行业经验但不懂代码的人。不知道你收藏过哪些好文章、你觉得什么样的表达叫"好"。不知道你对这个话题的真实判断是什么——哪些观点你认同、哪些是废话。 
 它在一个真空环境里凭空创作。 一个再聪明的人被关在没有任何信息的房间里，也只能说出正确但空洞的废话。

 适合已经有稳定内容输出习惯的人。 
 如果你一周写不了一篇，问题不在于工具链而在于要不要写、写什么。 
 这套方法解决的是 "已经决定要写，怎么写得更快更好" 。 
 另外文中会用到命令行工具（CLI），不会用也没关系——核心思维一样，结尾会给降级方案。

 以前写新文章要在脑子里翻："我以前写过类似话题吗？当时怎么说的？那个数据在哪？" 效率低还经常漏。
```
