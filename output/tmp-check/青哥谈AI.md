 摘要 
 我们推出了 Kimi K2，这是一个拥有 320 亿活跃参数和 1 万亿总参数 的混合专家 (Mixture-of-Experts, MoE) 大型语言模型 (LLM) 。我们提出了 MuonClip 优化器 ，它在 Muon 的基础上通过一种新颖的 QK-clip 技术解决了训练不稳定性的问题，同时保留了 Muon 先进的 token 效率 。基于 MuonClip，K2 经过了 15.5 万亿 token 的预训练，实现了零损失尖峰 (loss spike) 。在后训练阶段，K2 经历了一个多阶段的后训练过程，其中重点是大规模智能体数据合成 (agentic data synthesis) 流水线 (pipeline) 和联合强化学习 (reinforcement learning, RL) 阶段 ，在该阶段中，模型通过与真实和合成环境 (synthetic environments) 的互动来提升其能力。

 1 引言 
 大型语言模型 (LLMs) 的发展正在经历一场深刻的范式转变，转向智能体智能 (Agentic Intelligence) ——模型在复杂动态环境中自主感知、规划、推理和行动的能力 。这种转变标志着从静态模仿学习 (static imitation learning) 到模型通过互动主动学习 、获取超出其训练分布 (training distribution) 的新技能 以及通过经验调整行为 的转变。人们认为，这种方法使得人工智能 (AI) 智能体能够超越静态人类生成数据 (human-generated data) 的限制，并通过自身的探索和利用获得超人能力 。因此，智能体智能正迅速成为下一代基础模型 (foundation models) 的决定性能力，在工具使用、软件开发和现实世界自主性 (real-world autonomy) 等方面具有广泛的影响 。 
 实现智能体智能在预训练和后训练阶段都带来了挑战 。预训练必须在有限高质量数据的约束下赋予模型广泛的通用先验知识 (general-purpose priors)，将 token 效率（即每个 token 的学习信号 (learning signal)）提升为关键的扩展系数 (scaling coefficient) 。后训练必须将这些先验知识转化为可操作的行为 (actionable behaviors)，但多步推理 (multi-step reasoning)、长期规划 (long-term planning) 和工具使用等智能体能力在自然数据中很少见，并且扩展成本高昂 。可扩展地合成结构化、高质量的智能体轨迹 (agentic trajectories)，并结合通用强化学习 (RL) 技术（纳入偏好和自我批评）对于弥合这一差距至关重要 。 
 在这项工作中，我们介绍了 Kimi K2，这是一个拥有 1.04 万亿参数的混合专家 (Mixture-of-Experts, MoE) LLM，激活参数为 320 亿 ，旨在解决核心挑战并突破智能体能力的界限 。我们的贡献涵盖了预训练和后训练两个前沿领域：我们提出了 MuonClip ，这是一种新颖的优化器 (optimizer)，它将 token 高效的 Muon 算法与一种称为 QK-Clip 的稳定性增强机制 (stability-enhancing mechanism) 相结合 。使用 MuonClip，我们成功地对 Kimi K2 进行了 15.5 万亿 token 的预训练，没有出现任何损失尖峰 。 
 我们引入了一个大规模的智能体数据合成流水线，该流水线通过模拟和真实世界环境系统地生成工具使用演示 (tool-use demonstrations) 。该系统构建了多样化的工具 (tools)、智能体 (agents)、任务 (tasks) 和轨迹 (trajectories)，以大规模创建高保真、可验证正确的智能体交互 。 
 我们设计了一个通用强化学习框架，该框架将可验证奖励 (verifiable rewards, RLVR) 与自我批评准则奖励机制 (self-critique rubric reward mechanism) 相结合 。模型不仅从外部定义的任务中学习 ，还通过评估自己的输出来改进 ，将对齐 (alignment) 从静态领域扩展到开放式领域 (open-ended domains) 。 
 Kimi K2 在广泛的智能体和前沿基准 (frontier benchmarks) 上表现出强大的性能 。它在 Tau2-bench 上获得 66.1 分 ，在 ACEBench (en) 上获得 76.5 分 ，在 SWE-bench Verified 上获得 65.8 分 ，在 SWE-bench Multilingual 上获得 47.3 分 ，在非思考评估设置 (non-thinking evaluation settings) 下优于大多数开放和封闭权重基线 (closed-weight baselines) ，缩小了与 Claude 4 Opus 和 Sonnet 的差距 。在编码、数学和更广泛的 STEM 领域，Kimi K2 在 LiveCodeBench v6 上获得 53.7 分 ，在 OJBench 上获得 27.1 分 ，在 AIME 2025 上获得 49.5 分 ，在 GPQA-Diamond 上获得 75.1 分 ，进一步突出了其在通用任务中的能力 。在 LMSYS Arena 排行榜 (leaderboard) （2025 年 7 月 17 日）上，Kimi K2 根据超过 3,000 个用户投票排名第一的开源模型，总排名第五 。为了推动智能体智能的进一步发展，我们正在开源我们的基础模型和后训练检查点，使社区能够大规模探索、完善和部署智能体智能 。 
 2 预训 练 (Pre-training) Kimi K2 的基础模型是一个万亿参数的混合专家 (MoE) Transformer 模型，在 15.5 万亿高质量 token 上进行了预训练 。鉴于高质量人类数据 (human data) 的可用性日益受限，我们认为 token 效率正在成为大型语言模型扩展的关键系数 。为了解决这个问题，我们引入了一套专门为最大化 token 效率而设计的预训练技术 。具体来说，我们采用了 token 高效的 Muon 优化器 (Muon optimizer) 并通过引入 QK-Clip 来缓解其训练不稳定性 。此外，我们还结合了合成数据生成来进一步从可用的高质量 token 中榨取智能 。模型架构遵循超稀疏 MoE (ultra-sparse MoE) 与多头潜在注意力 (multi-head latent attention, MLA) 的设计，类似于 DeepSeek-V3 ，这是根据经验扩展定律分析 (empirical scaling law analysis) 得出的 。底层基础设施旨在优化训练效率和研究效率。 
 2.1 MuonClip：通过权重裁剪实现稳定训练 (MuonClip: Stable Training with Weight Clipping) 
 我们使用 token 高效的 Muon 优化器 训练 Kimi K2，并加入了权重衰减 (weight decay) 和一致更新 RMS 缩放 (consistent update RMS scaling) 。我们之前的工作 Moonlight 中的实验表明，在相同的计算预算 (compute budget) 和模型大小以及因此相同的训练数据量下，Muon 显著优于 AdamW ，使其成为提高大型语言模型训练中 token 效率的有效选择 。 
 扩展 Muon 时的训练不稳定性 (Training instability when scaling Muon) 尽管 Muon 效率很高，但扩展 Muon 训练时却面临一个挑战：由于注意力 logits 爆炸 (exploding attention logits) 导致的训练不稳定性 ，这个问题在我们的实验中 Muon 发生得更频繁，而 AdamW 则较少 。现有的缓解策略不足 。例如，logits soft-cap 直接裁剪注意力 logits，但在应用裁剪之前，查询 (queries) 和键 (keys) 之间的点积 (dot products) 仍然可能过度增长 。另一方面，Query-Key 归一化 (Normalization, QK-Norm) 不适用于多头潜在注意力 (Multi-head Latent Attention, MLA)，因为其键矩阵 (Key matrices) 在推理 (inference) 过程中没有完全实例化 (fully materialized) 。 
 通过 QK-Clip 驯服 Muon (Taming Muon with QK-Clip) 为了解决这个问题，我们提出了一种新颖的权重裁剪机制 (weight-clipping mechanism) QK-Clip，以明确限制注意力 logits 。QK-Clip 通过在更新后重新缩放查询和键投影权重 (query and key projection weights) 来限制注意力 logits 的增长 。

 2.2 预训练数据：通过复述提高 token 效用 
 预训练中的 token 效率指的是在训练过程中每个 token 消耗所带来的性能提升 。提高 token 效用——每个 token 对模型更新贡献的有效学习信号——可以增强每个 token 对模型更新的影响，从而直接提高 token 效率 。当高质量 token 的供应有限且必须最大限度地利用时，这一点尤为重要 。一种提高 token 效用的简单方法是重复暴露于相同的 token，这可能导致过拟合和泛化能力 下降 。 
 图3：Kimi K2 的每步训练损失曲线 (Per-step training loss curve)，未进行平滑或子采样 (sub-sampling) 。它在整个训练过程中没有显示任何尖峰 。请注意，为了清晰起见，我们省略了训练的最初阶段。 
 Kimi K2 在预训练数据方面相对于 Kimi K1.5 的一项关键进展是引入了合成数据生成策略 (synthetic data generation strategy) 以提高 token 效用 。具体来说，采用了精心设计的复述流水线 (rephrasing pipeline) 来增加高质量 token 的数量，而不会引起显著的过拟合 。在本报告中，我们描述了两种针对知识和数学领域的领域特定复述技术 (domain-specialized rephrasing techniques)，这些技术实现了这种受控的数据增强 。 
 知识数据复述 (Knowledge Data Rephrasing) 在自然、知识密集型文本 (knowledge-intensive text) 上进行预训练存在权衡：单个 epoch 不足以全面吸收知识 (knowledge absorption)，而多 epoch 重复则收益递减并增加过拟合的风险 。为了提高高质量知识 token 的效用，我们提出了一个由以下关键组件组成的合成复述框架 (synthetic rephrasing framework) ： 
 风格和视角多样化提示 (Style- and perspective-diverse prompting)： 为了在保持事实完整性 (factual integrity) 的同时增强语言多样性 (linguistic diversity)，我们应用了一系列精心设计的提示 (prompts) 。这些提示指导大型语言模型以不同风格和不同视角生成原始文本的忠实复述 (faithful rephrasings) 。 
 分块自回归生成 (Chunk-wise autoregressive generation)： 为了在长文档中保持全局一致性 (global coherence) 并避免信息丢失 (information loss)，我们采用了基于分块的自回归重写策略 (chunk-based autoregressive rewriting strategy) 。文本被分成段落 (segments)，单独复述，然后拼接 (stitched) 回来形成完整的段落 (passages) 。这种方法减轻了 LLM 通常存在的隐式输出长度限制 (implicit output length limitations) 。此流水线的概述如图 4 所示 。 
 保真度验证 (Fidelity verification)： 为了确保原始内容和重写内容之间的一致性，我们执行保真度检查 (fidelity checks)，比较每个复述段落与其来源的语义对齐 。这在训练之前作为初始质量控制步骤 (quality control step) 。 
 我们通过测试其在 SimpleQA 上的相应准确性来比较数据复述和多 epoch 重复 。我们使用 K2 的早期检查点 (early checkpoint) 进行实验，并评估了三种训练策略 ：（1）重复原始数据集 10 个 epoch ，（2）将数据复述一次并重复 10 个 epoch ，以及（3）将数据复述 10 次并进行单次训练 (single training pass) 。如表 1 所示，准确性在这些策略中持续提高，证明了我们基于复述的增强的有效性 。我们将此方法扩展到其他大规模知识语料库 (knowledge corpora)，并观察到同样令人鼓舞的结果，每个语料库最多复述两次 。

 预训练数据总体 (Pre-training Data Overall)： Kimi K2 预训练语料库 (pre-training corpus) 包含 15.5 万亿精选的高质量数据 token，涵盖四个主要领域：网络文本 (Web Text)、代码 (Code)、数学 (Mathematics) 和知识 (Knowledge) 。大多数数据处理流水线遵循 Kimi K1.5 中概述的方法 。对于每个领域，我们都进行了严格的正确性 (correctness) 和质量验证 (quality validation)，并设计了有针对性的数据实验 (targeted data experiments)，以确保精选数据集同时实现高多样性 (high diversity) 和有效性 (effectiveness) 。 
 2.3 模型架构 (Model Architecture) 
 Kimi K2 是一个 1.04 万亿参数的混合专家 (MoE) Transformer 模型，拥有 320 亿激活参数 (activated parameters) 。其架构遵循与 DeepSeek-V3 类似的设计，采用多头潜在注意力 (Multi-head Latent Attention, MLA) 作为注意力机制，模型隐藏维度 (hidden dimension) 为 7168，MoE 专家隐藏维度为 2048 。我们的扩展定律分析表明，持续增加稀疏度 (sparsity) 会显著提高性能，这促使我们将专家数量增加到 384 个，而 DeepSeek-V3 中为 256 个 。为了减少推理过程中的计算开销，我们将注意力头 (attention heads) 的数量减少到 64 个，而 DeepSeek-V3 中为 128 个 。表 2 详细比较了 Kimi K2 和 DeepSeek-V3 之间的架构参数 。 
 表2：Kimi K2 与 DeepSeek-V3 的架构比较 
 稀疏度扩展定律 (Sparsity Scaling Law) 我们开发了一种针对混合专家 (Mixture-of-Experts, MoE) 模型家族使用 Muon 的稀疏度扩展定律 。稀疏度定义为专家总数与激活专家数量之比 。通过精心控制的小规模实验，我们观察到在固定激活参数数量（即 FLOPs 恒定）下 ——增加专家总数（即增加稀疏度）持续降低训练和验证损失 ，从而提高整体模型性能 （图 5） 。具体来说，在计算最优稀疏度扩展定律下，实现相同的 1.5 验证损失时 ，与稀疏度 8、16 和 32 相比，稀疏度 48 分别将 FLOPs 减少了 1.69 倍、1.39 倍和 1.15 倍 。尽管增加稀疏度会带来更好的性能，但这种收益伴随着基础设施复杂性 (infrastructure complexity) 的增加 。为了平衡模型性能与成本，Kimi K2 采用了 48 的稀疏度 ，每次前向传播 (forward pass) 激活 384 个专家中的 8 个。 
 图 5：稀疏性缩放定律。增加稀疏性可以提高模型性能。我们将激活的专家数量固定为 8 个，共享专家数量固定为 1 个，并改变专家总数，从而得到具有不同稀疏程度的模型。 
 图 6：注意力头数量等于层数的模型以及注意力头数量翻倍的模型的缩放曲线。注意力头数量翻倍导致验证损失减少约 0.5% 到 1.2%。 
 注意力头数量（Number of Attention Heads）： DeepSeek-V3 将注意力头的数量设置为模型层数的大约两倍，以更好地利用内存带宽并提高计算效率。然而，随着上下文长度的增加，将注意力头数量加倍会导致显著的推理开销，从而降低更长序列长度下的效率。这在智能体应用中成为一个主要的限制，因为智能体应用中高效的长上下文处理至关重要。例如，当序列长度为 128k 时，将注意力头数量从 64 增加到 128（同时将专家总数固定在 384）会导致推理 FLOPs 增加 83%。为了评估这种设计的影响，我们进行了受控实验，比较了在不同训练 FLOPs 下，注意力头数量等于层数的配置与注意力头数量翻倍的配置。在等 token 训练条件下，我们观察到，在不同的计算预算下（图 6），将注意力头数量加倍仅在验证损失方面产生了适度的改进（范围从 0.5% 到 1.2%）。考虑到稀疏度 48 已经提供了强大的性能，注意力头数量翻倍带来的边际收益并不能证明其推理成本是合理的。因此，我们选择 64 个注意力头。 
 2.4 训练基础设施 
 2.4.1 计算集群 
 Kimi K2 是在一个配备 NVIDIA H800 GPU 的集群上训练的。H800 集群中的每个节点包含 2 TB 内存和 8 个 GPU，这些 GPU 在节点内部通过 NVLink 和 NVSwitch 连接。在不同节点之间，利用 8×400 Gbps RoCE 互连来促进通信。 
 2.4.2 模型扩展的并行策略 
 大型语言模型的训练通常在动态资源可用性下进行。我们不追求一种仅适用于特定资源量的单一并行策略，而是寻求一种灵活的策略，允许 Kimi K2 在任何 32 的倍数个节点上进行训练。我们的策略结合了 16 路管道并行（PP，带虚拟阶段）、16 路专家并行（EP）和 ZeRO-1 数据并行。 
 图 7：不同 PP（管道并行）阶段中计算、通信和卸载的重叠。 
 在此设置下，将模型参数存储为 BF16 格式，并将它们的梯度累积缓冲区存储为 FP32 格式，大约需要 6 TB 的 GPU 内存，这些内存分布在一个包含 256 个 GPU 的模型并行组中。优化器状态的放置取决于训练配置。当训练节点总数较大时，优化器状态是分布式的，将其每个设备的内存占用降低到可忽略的水平。当训练节点总数较小时（例如 32 个），我们可以将一些优化器状态卸载到 CPU。 
 这种方法允许我们对小型和大型实验重用相同的并行配置，同时让每个 GPU 为所有状态保留大约 30 GB 的 GPU 内存。剩余的 GPU 内存用于激活，如第 2.4.3 节所述。这种一致的设计对于研究效率很重要，因为它简化了系统并大大加快了实验迭代速度。 
 EP通信与交错式1F1B重叠： 通过增加预热微批次数量，我们可以在标准交错式1F1B调度下，使EP（专家并行）的全对全通信与计算重叠。相比之下，DualPipe 将参数和梯度的所需内存加倍，需要增加并行度来弥补。增加PP（管道并行）会引入更多的气泡，而增加EP（如下所述）会产生更高的开销。对于训练一个拥有超过一万亿参数的大型模型来说，这些额外的成本过高，因此我们选择不使用DualPipe。 
 然而，交错式1F1B将模型分成更多阶段，引入了不可忽略的PP通信开销。为了减轻这一成本，我们将权重-梯度计算从每个微批次的后向传播中解耦出来，并与相应的PP通信并行执行。因此，除了预热阶段，所有PP通信都可以有效重叠。 
 更小的EP尺寸： 为了确保1F1B阶段计算-通信的完全重叠，K2中减少的注意力计算时间（K2有64个注意力头，而DeepSeek-V3有128个头）需要将EP操作的时间最小化。这通过采用最小的可行EP并行策略（具体为EP=16）来实现。使用更小的EP组也放宽了专家平衡约束，使得在无需进一步调优的情况下即可实现接近最优的速度。 
 2.4.3 激活内存削减 
 在为参数、梯度缓冲区和优化器状态预留空间后，每个设备上剩余的 GPU 内存不足以容纳完整的 MoE（专家混合）激活。为了确保激活内存符合限制，特别是对于在 1F1B 预热阶段累积最大激活的初始管道阶段，采用了以下技术。 
 选择性重计算： 对开销小、内存占用大的阶段（包括 LayerNorm、SwiGLU 和 MLA 上投影）应用重计算。此外，在训练期间对 MoE 下投影进行重计算，以进一步减少激活内存。虽然是可选的，但这种重计算保持了足够的 GPU 内存，防止了训练早期阶段因专家不平衡导致的崩溃。 
 对不敏感激活使用 FP8 存储： MoE 上投影和 SwiGLU 的输入被压缩到 1x128 块中的 FP8-E4M3 格式，并带有 FP32 比例因子。小型实验表明没有可测量的损失增加。由于我们在初步研究中观察到性能下降的潜在风险，我们没有在计算中应用 FP8。 
 激活 CPU 卸载： 所有剩余的激活都被卸载到 CPU RAM。一个复制引擎负责流式传输卸载和加载，与计算和通信核重叠。在 1F1B 阶段，我们卸载前一个微批次的前向激活，同时预取下一个微批次的后向激活。预热和冷却阶段的处理方式类似，整体模式如图 7 所示。尽管卸载可能由于 PCIe 流量拥塞而轻微影响 EP 流量，但我们的测试表明 EP 通信仍然完全重叠。

 3 后 训 练 (Post-Training ) 
 3.1 有监督微调（Supervised Fine-Tuning） 
 我 们在后训练中采用 Muon 优化器，并建议将其用于 K2 的微调。这源于我们之前的工作得出的结论：经过 Muon 预训练的模型检查点通过 Muon 微调能产生最佳性能。 
 我们构建了一个涵盖不同领域的大规模指令微调数据集，遵循两个核心原则：最大化提示多样性和确保高响应质量。为此，我们开发了一套针对不同任务领域的数据生成管道，每个管道都结合了人工标注、提示工程和验证过程。我们采用 K1.5 和其他内部领域专业专家模型来为各种任务生成候选响应，然后由大型语言模型（LLMs）或人工评审员进行自动化质量评估和过滤。对于智能体数据，我们创建了一个数据合成管道，通过多步交互式推理来教授模型工具使用能力。 
 3.1.1 用于工具使用学习的大规模智能体数据合成 
 现代大型语言模型（LLM）智能体的一个关键能力是它们能够自主使用不熟悉的工具、与外部环境交互，并通过推理、执行和错误纠正迭代地完善其行动。 智能体 工具使用能力对于解决需要与现实世界系统动态交互的复杂多步任务至关重要。最近的基准测试，如 ACEBench和 τ -bench，强调了全面工具使用评估的重要性，而 ToolLLM 和 ACEBench 等框架已经展示了教导模型有效使用数千种工具的潜力。 
 然而，大规模训练这种能力带来了重大挑战：虽然现实世界环境提供了丰富和真实的交互信号，但由于成本、复杂性、隐私和可访问性限制，它们通常难以大规模构建。最近关于合成数据生成的工作（AgentInstruct；Self-Instruct；StableToolBench；ZeroSearch）在不依赖现实世界交互的情况下创建大规模数据方面显示出有希望的结果。基于这些进展并受到 ACEBench 综合数据合成框架的启发，我们开发了一个管道，能够大规模模拟现实世界工具使用场景，从而生成数万个多样化且高质量的训练示例。 
 我们的数据合成管道分为三个阶段，如图 8 所示。 
 工具规范生成： 我们首先从真实世界工具和 LLM 生成的合成工具中构建一个大型工具规范存储库； 
 智能体 和任务生成： 对于从工具存储库中采样的每个工具集，我们生成一个 智能体 来使用该工具集以及一些相应的任务； 
 轨迹生成： 对于每个 智能体 和任务，我们生成 智能体 通过调用工具完成任务的轨迹。 
 图 8：面向工具使用的数据合成流程 (a) 工具规范来源于真实工具与大语言模型，智能体与任务由工具仓库自动生成；(b) 多智能体协同生成并筛选含工具调用的行为轨迹。 
 图 9：工具嵌入表示的 t-SNE 可视化图 (a) 来自现实世界的 MCP 工具在原始来源类别上呈现自然聚类；(b) 合成工具被归入预定义的领域类别，有效覆盖工具空间。这两类工具共同保证了对多样工具功能的全面表征。 
 领域演化与工具生成（Domain Evolution and Tool Generation）： 我们通过两种互补的方法构建了一个全面的工具库。首先，直接从 GitHub 上获取了 3000 多个真实的 MCP（Model Context Protocol）工具，充分利用了现有的高质量工具规范。其次，我们通过分层的领域生成过程系统性地演化出合成工具：我们以关键领域（如金融交易、软件应用、机器人控制）为起点，在每个领域下进一步细化为多个具体的应用子域，并为每个子域合成专用工具。这些工具具有明确的接口、功能描述和操作语义。这一“演化式”流程共生成了超过 20,000 个合成工具。图 9 通过 t-SNE 可视化展示了工具集合的多样性，表明 MCP 工具与合成工具在工具空间中覆盖了互补的区域。 
 智能体多样化（Agent Diversification）： 我们通过合成多样的系统提示（system prompts）并为其配置不同组合的工具，从工具库中构建了数千个不同的智能体。这种方法造就了能力各异、专长领域多样、行为模式丰富的智能体群体，确保了对潜在应用场景的广泛覆盖。 
 基于评价准则的任务生成（Rubric-Based Task Generation）： 针对每种智能体配 置，我们生成了从简单到复杂的一系列任务。每个任务都配备有明确的评分准则（rubric），包括成功判定标准、预期的工具使用模式和评估检查点。基于 rubrics 的设计确保了智能体性能评估的一致性与客观性。 
 多轮轨迹生成（Multi-turn Trajectory Generation） 我们通过以下组件模拟真实的工具使用场景： 
 用户模拟 ： 由大语言模型生成具有不同交流风格和偏好的用户角色，与智能体进行多轮对话，形成自然的人机交互轨迹。 
 工具执行环境： 一个复杂的工具模拟器（功能上等同于一个世界模型）会执行工具调用并提供真实的反馈。 该模拟器在每次工具执行后会维护并更新状态，从而实现具有持久效果的复杂多步骤交互 。它引入了受控的随机性，以产生包括成功、部分失败和边缘情况在内的各种结果。 
 质量评估与过滤： 一个基于大型语言模型 ( LLM ) 的评审器根据任务评分标准评估每个轨迹 。只有符合成功标准的轨迹才会被保留用于训练，这确保了高质量的数据，同时允许任务完成策略的自然变化 。 
 与真实执行环境的混合方法： 虽然模拟提供了可扩展性，但我们承认模拟保真度固有的局限性 。为了解决这个问题，我们通过真实执行沙盒来补充我们的模拟环境，用于对真实性至关重要的场景，特别是在编码和软件工程任务中 。这些真实的沙盒执行实际代码，与真实的开发环境交互，并通过客观指标（如测试套件通过率）提供真实反馈 。这种组合确保了我们的模型既能从模拟场景的多样性中学习，也能从真实执行的真实性中学习，从而显著增强了实际智能体（agent）的能力 。 
 通过利用这种结合了可扩展模拟和有针对性真实世界执行的混合管道，我们生成了多样化、高质量的工具使用演示，平衡了覆盖范围和真实性 。我们合成数据生成的可扩展性和自动化，加上真实执行环境提供的基础，通过我们的质量过滤过程有效地实现了大规模拒绝采样 。这种高质量的合成数据在用于监督微调 (SFT) 时，已证明在广泛的实际应用中显著提升了模型的工具使用能力。 
 3.2 强化学习（Reinforcement Learning） 
 强化学习 (RL) 被认为比 SFT 具有更好的 token 效率和泛化能力 。基于 K1.5 的工作，我们继续在 K2 中扩展 RL 的任务多样性和训练浮点运算 (FLOPs) 。为了支持这一点，我们开发了一个类似 Gym 的可扩展框架，以促进 RL 在各种场景中的应用 。我们用大量具有可验证奖励的任务扩展了该框架 。对于依赖主观偏好的任务，例如创意写作和开放式问答，我们引入了一种自我批评奖励机制，模型会进行成对比较来判断自己的输出 。这种方法使得来自不同领域的任 务都能受益于 RL 范式 。 
 3.2.1 可验证奖励 Gym 
 数学、科学、技术、工程和数学（STEM）和逻辑任务 ：对于数学、STEM 和逻辑推理领域，我们的 RL 数据准备遵循两个关键原则：多样化覆盖和中等难度 。 
 多样化覆盖： 对于数学和 STEM 任务，我们结合专家标注、内部问答提取管道和开放数据集，收集高质量的问答对 。在收集过程中，我们利用标签系统有意识地增加对覆盖不足领域的覆盖 。对于逻辑任务，我们的数据集包含多种格式，包括结构化数据任务（例如，多跳表格推理、跨表聚合）和逻辑谜题（例如，24点游戏、数独、谜语、数字谜和莫尔斯电码解码） 。 
 中等难度： RL 提示集不应太容易也不应太难，这两种情况都可能产生很少的信号并降低学习效率 。我们使用 SFT 模型的 pass@k 准确率评估每个问题的难度，并只选择中等难度的问题 。 
 复杂指令遵循： 有效的指令遵循不仅需要理解显式约束，还需要驾驭隐式要求、处理边缘情况，并在长时间对话中保持一致性 。我们通过结合自动化验证和对抗性检测的混合验证框架，以及可扩展的课程生成管道来解决这些挑战 。我们的方法采用双路径系统，以确保精度和鲁棒性： 
 混合规则验证： 我们实施了两种验证机制： (1) 通过代码解释器进行确定性评估，用于具有可验证输出的指令（例如，长度、风格约束），以及 (2) LLM 作为评审器进行评估，用于需要对约束进行细致理解的指令 。为了解决模型可能声称满足指令但未实际遵守的潜在对抗性行为，我们引入了一个额外的“hack-check”层，专门检测此类欺骗性声明 。 
 多源指令生成： 为了构建我们的训练数据，我们采用了三种不同的生成策略，以确保全面的覆盖： (1) 由我们的数据团队开发的专家精心设计的复杂条件提示和评分标准 ， (2) 受 AutoIF 启发的智能体指令增强，以及 (3) 经过微调的模型，专门用于生成探测特定故障模式或边缘情况的附加指令 。这种多管齐下的方法确保了指令覆盖的广度和深度 。 
 忠实度： 忠实度对于在多轮工具使用、自生成推理链和开放环境交互等场景中运行的智能体模型至关重要 。受 FACTS Grounding 评估框架的启发，我们训练了一个句子级忠实度评审模型来执行自动化验证 。该评审器能够有效地检测在上下文中没有支持证据的事实声明 。它作为一种奖励模型，以提高整体忠实度性能 。 
 编码和软件工程： 为了增强我们在解决竞赛级编程问题方面的能力，我们从开源数据集和合成来源收集问题及其评审器 。为了确保合成数据的多样性和奖励信号的正确性，我们整合了从预训练数据中检索到的高质量人工编写的单元测试 。 
 对于软件工程任务，我们从 GitHub 收集了大量的拉取请求和问题，以构建包含用户提示/问题和可执行单元测试的软件开发环境 。该环境建立在强大的沙盒基础设施上，由 Kubernetes 提供支持，以实现可扩展性和安全性 。它支持超过 10,000 个并发沙盒实例，性能稳定，非常适合竞争性编码和软件工程任务 。 
 安全： 我们增强安全性的工作始于一组人工策划的种子提示，这些提示经过人工精心设计，涵盖了常见的风险类别，如暴力、欺诈和歧视 。 
 为了模拟复杂的越狱尝试（例如，角色扮演、文学叙事和学术讨论），我们采用了具有三个关键组件的自动化提示演化管道： 
 攻击 模型（ Attack Model ） ： 迭代生成旨在从目标 LLM 引出不安全响应的对抗性提示 。 
 目标 模型（ Target Model ） ： 对这些提示产生响应，模拟潜在的漏洞 。 
 评审模 型（ Judge Model ）： 评估交互以确定对抗性提示是否成功绕过了安全机制 。 
 每次交互都使用任务特定的评分标准进行评估，使评审模型能够提供二元成功/失败标签 。 
 3.2.2 超越验证：自评评分奖励 
 为了将模型对齐扩展到具有可验证奖励的任务之外，我们引入了一个通用强化学习框架，该框架基于自我批评反馈 。该方法旨在通过将可验证场景中学习到的能力扩展到更广泛的主观任务，使大型语言模型 (LLM) 与细微的人类偏好对齐，包括帮助性、创造力、推理深度、事实性和安全性 。该框架采用自我批评评分奖励机制（ Self-Critique Rubric Reward） 运行，模型会评估自己的输出以生成偏好信号 。为了将 K2 引导为一个称职的评审器，我们策划了开源和内部偏好数据集的混合，并在监督微调 (SFT) 阶段初始化其批评能力 。 
 自我批评策略优化： 在学习循环的第一个核心过程中，K2 演员为涵盖广泛用例的通用提示生成响应 。然后，K2 批评器通过对评分标准组合进行成对评估来对所有结果进行排名，其中包括核心评分标准（附录 F.1），它们代表了我们人工智能助手的核心价值；规范性评分标准（附录 F.2），旨在消除奖励作弊；以及由我们数据团队为特定指令上下文精心设计的人工标注评分标准 。尽管某些评分标准可以被指定为强制性的，但 K2 保留了根据其内部先验来权衡它们的灵活性 。这种能力能够实现与其不断演进的在策略行为的动态和持续对齐，确保模型的响应与其核心身份保持一致，同时适应特定指令 。 
 闭环批评器改进和对齐： 在强化学习 (RL) 训练期间，批评模型会使用可验证信号进行改进 。从可验证奖励提示生成的在策略 (on-policy) 运行 (rollout) 用于持续更新批评器，这是将来自 RLVR 的客观性能信号直接提炼到其评估模型中的关键步骤 。这种迁移学习过程将其更主观的判断建立在可验证数据之上，从而使可验证任务的性能增益能够增强批评器对缺乏明确奖励信号的复杂任务的判断 。这种闭环过程确保批评器随着策略的演变不断重新校准其评估标准 。通过将主观评估建立在可验证数据之上，该框架能够实现与复杂、不可验证的人类目标的稳健和可扩展对齐 。 
 因此，这种整体对齐在广泛的领域中产生了全面的性能改进，包括用户意图理解、创意写作、复杂推理和细致的语言理解 。

 PTX 损失： 为了防止在联合强化学习 (RL) 训练期间潜在遗忘有价值的高质量数据，我们策划了一个包含人工选择的高质量样本的数据集，并通过辅助 PTX 损失将其整合到 RL 目标中 。这种策略不仅利用了高质量数据的优势，还减轻了对训练机制中明确存在的有限任务集过度拟合的风险 。这种增强显著改善了模型在更广泛领域中的泛化能力 。 
 温度衰减： 对于创意写作和复杂推理等任务，我们发现在训练初期通过高采样温度来促进探索至关重要 。高温度允许模型生成多样化和创新的响应，从而促进有效策略的发现并降低过早收敛到次优解的风险 。然而，在训练后期或评估期间保持高温度可能是有害的，因为它会引入过多的随机性并损害模型输出的可靠性和一致性 。为了解决这个问题，我们采用了温度衰减调度，以在整个训练过程中从探索转向利用 。这种策略确保模型在最有益时进行探索，同时最终收敛到稳定和高质量的输出 。 
 3.3 RL 基础设施 
 3.3.1 共址架构（ Colocated Architecture ） 
 与 K1.5 类似，我们为同步强化学习 (RL) 训练采用了混合共址架构，其中训练和推理引擎位于同一工作节点上 。当一个引擎正在积极工作时，另一个引擎会释放或卸载其 GPU 资源以进行适应 。在 RL 训练的每次迭代中，一个集中式控制器首先调用推理引擎来生成新的训练数据 。然后，它通知训练引擎在新数据上进行训练，并将更新后的参数发送到推理引擎以进行下一次迭代 。 
 每个引擎都针对吞吐量进行了高度优化 。此外，随着模型扩展到 K2 的大小，引擎切换和故障恢复的延迟变得显著 。我们将在这些方面介绍我们的系统设计考虑 。

 3.3.2 高效的引擎切换 
 在运行期间，训练引擎的参数会被卸载到动态随机存取存储器 (DRAM) 中 。因此，启动训练引擎只是一个简单的 H2D 传输步骤 。然而，启动推理引擎是一个更大的挑战，因为它必须从具有不同分片范式的训练引擎获取更新的参数 。 
 考虑到 K2 的规模和涉及的大量设备，使用网络文件系统进行重新分片和广播参数是不切实际的 。保持开销较低所需的总带宽达到了每秒数 PB 。为了应对这一挑战，我们开发了一个分布式检查点引擎，它与训练节点共存，用于管理参数状态 。为了执行参数更新，每个检查点引擎工作节点都会从训练引擎获取参数的本地副本，然后将完整的参数集广播到所有检查点引擎工作节点 。随后，推理引擎仅从检查点引擎检索其所需的参数分片 。此过程如图 10 所示 。为了使 IT 模型能够实现这一点，更新以参数逐个、流水线的方式执行，从而最大限度地减少内存占用（参见附录 G） 。 
 我们选择将完整的参数集广播到整个集群，无论每个推理工作节点的具体分片方案如何 。虽然这比理论上的最优方法传输的数据量多出数倍，但它提供了一种更简单的系统设计，对训练和推理引擎的侵入性更小 。我们选择牺牲这种微小的开销，以完全解耦训练引擎和推理引擎，从而显著简化维护和测试 。 
 值得注意的是，这种方法由于减少了同步开销和更高的网络带宽利用率，优于“按需传输”方法 。我们的系统可以在不到 30 秒的时间内完成 Kimi K2 的完整参数更新，这对于典型的强化学习 (RL) 训练迭代来说可以忽略不计 。 
 3.3.3 高效的系统启动 
 由于大规模训练容易发生系统故障，因此优化像 Kimi K2 这样大型模型的启动时间至关重要 。 
 为了启动训练引擎，我们让每个训练工作节点选择性地从磁盘读取部分或不读取参数，并向其对等节点广播必要的参数 。设计目标是确保所有工作节点只集体读取检查点一次，从而最大限度地减少昂贵的磁盘输入/输出(IO) 。 
 由于推理引擎是独立的副本，我们希望避免在它们之间引入额外的同步障碍 。因此，我们选择重用检查点引擎进行启动：我们让检查点引擎集体从磁盘读取检查点，类似于训练引擎的启动方式 。然后，它使用上一节中介绍的方法更新未初始化的推理引擎的状态 。通过利用专用的检查点引擎，系统也变得对单点故障具有鲁棒性，因为推理副本可以在不与其他副本通信的情况下重新启动 。 
 3.3.4 智能体（Agentic）运行 
 我们的强化学习 (RL) 基础设施支持训练长时程、多轮智能体任务 。在运行期间，这些任务带来了独特的挑战，例如复杂的环境交互和较长的运行持续时间 。这里我们介绍了一些优化措施来缓解这些问题 。 
 由于环境的多样性，某些交互可能会被阻塞，等待环境反馈（例如，虚拟机或代码解释器），从而导致图形处理器 (GPU) 闲置 。我们采用两种策略来最大限度地提高 GPU 利用率 ： (i) 我们将重量级环境部署为可以更容易扩展的专用服务 ； (ii) 我们采用大量并发运行来分摊某些昂贵交互引起的延迟 。 
 智能体运行的另一个挑战是单个运行轨迹可能非常长 。为了防止长尾轨迹阻塞整个运行过程，我们采用了部分运行技术 。该策略允许暂停长尾未完成的任务，并在下一个强化学习 (RL) 迭代中恢复 。 
 为了提高研究效率，我们还设计了一个受 OpenAI Gym 框架启发的统一接口，以简化新环境的集成 。我们希望在未来将我们的强化学习 (RL) 基础设施扩展到更多样化的交互环境 。

 ｜ 课程概览 
 一、核心问题：什么是模型的“深度思考”？ 
 定义 ： 这并非指模型拥有与人类同质的意识或思考，而是一种特定的 行为模式 ：先生成一段包含自我验证、探索多种可能性、规划步骤的详细文本（内心戏），再输出最终答案。 
 基本原理 ： 其本质是一种 “测试时计算” （Testing Time Compute），即在推理（Inference）阶段投入更多计算资源，以“长度换深度”，从而提升解决复杂问题的能力。这与AlphaGo在下棋时使用蒙特卡洛树搜索（MCTS）进行深度推演的理念一致。 
 二、打造深度思考模型的四种核心方法 
 课程将打造方法分为两大类，四种具体路径： 
 A. 无需微调模型的方法（适用于已有模型） 
 方法一：高级指令与提示 (Better CoT) 
 通过精心设计的提示词（Prompt），如“Let's think step by step”或更复杂的 监督式思维链 (Supervised CoT) ，引导模型自主生成详细的思考和验算过程。此方法依赖模型本身强大的指令遵循能力。 
 方法二：设计推理工作流 (Workflow) 
 强制模型进行多路径探索，然后筛选出最佳答案。关键技术包括： 
 多次尝试 + 多数投票 (Majority Vote) ：让模型多次解答，选择出现频率最高的答案。 
 验证器 (Verifier) + 最佳选择 (Best-of-N) ：训练一个“裁判”模型来评估多个答案的质量，选出最好的一个。 
 束搜索 (Beam Search) ：在解题的每一步都保留N个最有可能的路径，逐步筛选，避免过早陷入错误路径。 
 B. 需要微调模型的方法（打造新模型） 
 方法三：模仿学习 (Imitation Learning) 
 核心是“教”模型如何思考。通过监督式微调，让模型学习包含“问题→ 高质量推理过程 →答案”的范例。 
 挑战 ：高质量的推理数据从何而来？可通过知识蒸馏（让强模型生成）或 “知错能改” 的训练（故意包含错误路径和修正过程）来构建，让模型学会逆境翻盘。 
 方法四：强化学习 (Reinforcement Learning) 
 核心是 “结果导向” 。不关心推理过程是否优美，只根据最终答案的 正确与否 给予模型奖励或惩罚，让模型自行探索出最高效的、能导向正确答案的思考模式。这是DeepSeek-R1系列闻名的核心技术。 
 三、综合案例：DeepSeek-R1 的诞生之路 
 DeepSeek-R1并非单一方法的产物，而是上述所有方法的 复杂融合 ： 
 1. 起点 ：先用 强化学习 训练出一个推理能力强但过程混乱的“野蛮”模型（R1-0）。 
 2. 提纯 ： 让R1-0生成大量推理数据，再投入 大量人力进行修改和润色 。 
 3. 升华 ：用这些高质量数据进行 模仿学习 ，并结合 高级指令 生成的数据，训练出更可控的模型。 
 4. 迭代 ：在上述基础上，再进行多轮的强化学习和模仿学习，并结合 工作流 的理念，最终得到成熟的DeepSeek-R1。 
 结论 ：这四种方法并非互斥，而是可以相辅相成、组合使用的。 
 四、总结与挑战 
 当前挑战 ：模型常常在简单问题上进行 不必要的“过度思考” ，浪费大量计算资源和时间。 
 未来方向 ：如何让模型学会 自适应推理 ，即判断问题的难度，只在需要时才启动深度思考，实现计算效率和问题解决能力的平衡。

 我们本周要讲的是会进行“深度思考”的大型语言模型。 
 这里的“深度思考”指的是现在一系列模型所展现出的一种特定行为。具有这种深度思考行为的模型包括： 
 ChatGPT的O系列 
 DeepSeek的R系列 
 Gemini的Flash Thinking 
 Claude 3.7 Sonnet的Extended Thinking Mode 
 所以，这已成为当前大型语言模型一个相当流行的能力。 
 深度思考行为的实例 
 以 DeepSeek-R1 为例，当你问它“1+1等于多少”时，它会回答“2”。但除了答案，还会出现一个可以展开的框，标记为“以深度思考”。 
 点开这个框，你会看到它的“内心小剧场”： 
 初步思考 ：“一个苹果加一个苹果，就是两个苹果，所以1+1等于2。” 
 自我质疑 ：“等一下，这会不会有什么陷阱？让我想想看，1+1会不一定等于2？比如说在二进制中，1+1等于10。”（我问这个问题时都没想到这点） 
 最终决策 ：“使用者只是写了1+1，没有提供额外的信息，他可能只是在测试我会不会想太多。所以，最终的答案是2。” 
 这些深度思考的语言模型，其特色就是会先给出一个很长的思考过程，最后再给出答案。通常，它会在思考过程前后加上 <think> 和 </think> 这样的标签，方便界面选择性地展示这部分内容。 
 在思考过程中，模型通常有几个行为： 
 验证（Verification） : “Let me check the answer.” 检查自己刚才的答案是否正确。 
 探索（Exploration） : 尝试思考其他的可能性。 
 规划（Planning） : 规划解决问题的步骤。 
 这个思考过程很多时候也被称为 推理（Reasoning） 。 
 注意：Reasoning vs. Inference 
 需要区分两个词： 
 Reasoning (推理) : 指模型在生成答案时，产生特别长的思考过程的这种行为。 
 Inference (推论) : 在机器学习领域，通常指使用一个已经训练好的模型进行测试或应用的过程。 
 本课程中，我们把这种深度思考行为直接称为“推理（Reasoning）”，但这并不意味着其过程与人类的推理完全一样。 
 一、 核心问题：什么是模型的“深度思考”？ 
 核心概念：Testing Time Compute 
 这种推理行为是 Testing Time Compute 的一种，即在测试阶段投入更大的算力，以期获得更好的结果。为什么这会有用呢？我们之前提过一个概念：“深度不够，长度来凑”。 
 1. AlphaGo的启示 
 Testing Time Compute 并非全新概念。在AlphaGo中，除了训练时得到的 Policy Network 和 Value Network，在测试（实际下棋）时，它也进行了巨大的运算，即 蒙特卡洛树搜索（Monte Carlo Tree Search, MCTS） 。AlphaGo 会在“脑内”模拟多种下棋的可能性，预估不同位置的胜率，最后选择胜率最高的位置。 
 2. Test Time Scaling 
 这个概念进一步引申出 Test Time Scaling ：思考越多，结果往往越好。一篇早期的论文研究了棋类游戏，发现可以将算力投注在训练时（得到更好的Policy/Value Network），也可以投注在测试时（做更大规模的MCTS）。实验结果表明，投入少量算力在 Testing Time，就可以显著减少在 Training Time所需耗费的大量算力。 
 如今，这个概念在语言模型上被广泛应用：通过产生非常长的思考过程，让语言模型解决更难的问题。 
 二、打造深度思考模型的四种核心方法 
 接下来，我们探讨打造这类模型的四种主要方法。前两种无需微调模型参数，后两种则需要。 
 方法一：更好的思维链（Chain-of-Thought, CoT） - 无需微调 
 Chain-of-Thought（CoT）就是让模型先列出解题过程，再给出答案，这与深度思考模型的行为非常类似。 
 Few-shot CoT : 给模型提供几个“问题-解题过程-答案”的范例，让它学会这种模式。 
 Zero-shot CoT : 无需范例，直接用一句魔法指令，如“Let's think step by step”，模型就会自动列出思考过程。 
 现在的推理模型所做的，可以看作是 长思维链（Long CoT） 。要让模型产生更长的思考过程，只需在Prompt中给出更详细、更精确的指示，这种方法也称为 Supervised CoT 。 
 Supervised CoT 实例 
 我对GPT-4o下了一个复杂的Prompt，要求它： 
 深入解析题目，制定清晰的解题计划。 
 按部就班执行，每一步都列出子计划。 
 每一步执行后都进行多次验算。 
 考量所有可能的解法。 
 将思考过程放在 <think> 和 </think> 标签之间。 
 问题：123 × 456 = ? 
 GPT-4o的回答展现了类似深度思考的行为： 
 题目分析与总计划 : 将456拆解为400+50+6，分别与123相乘后求和。 
 验算计划 : 
 用标准直式乘法验算。 
 估算答案的范围。 
 用Python验算（最后没做，属于幻觉）。 
 计算与验算 : 
 它正确地用直式乘法算出了56088。 
 它估算了上下界，虽然方法有点奇怪，但结论是答案在合理范围内。 
 它尝试用交换律做第三次验算，但算到一半发现“无明确帮助，略过”，这反而体现了它知道何时放弃无效路径的智能。 
 注意 ：这种方法对模型能力要求较高，较弱的模型（如LLaMA 3）可能无法完全遵循复杂的指令。 
 方法二：提供推理的工作流程 - 无需微调 
 对于能力稍弱的模型，我们可以直接为其设定一个推理的工作流程，强制它进行多方尝试。 
 1. 多次独立尝试（Parallel Exploration） 
 就像“无限猴子定理”，让语言模型对同一个问题回答成千上万次，由于每次回答都有随机性，它总有机会产生出正确答案。 
 但问题是，如何从众多答案中选出正确的那一个？ 
 多数投票（Majority Vote / Self-Consistency）: 选择出现次数最多的答案。这是一个非常强大且难以超越的基准方法。 
 验证器（Verifier / Best-of-N）: 训练一个模型（或直接用一个强大的语言模型）来评估每个答案的正确性，然后选择分数最高的那个答案（Best of N）。训练验证器的数据可以通过比较模型生成答案和标准答案来获得。 
 2. 逐层深入探索（Sequential & Step-by-step） 
 模型可以不是一次性生成完整答案，而是分步进行，并在每一步后进行验证。 
 过程验证器（Process Verifier） : 这种验证器不需要看到最终答案，只需根据解题过程的一部分就能判断其正确性。 
 束搜索（Beam Search） : 在每一步，生成多种可能的下一步，然后用过程验证器评估，只保留最好的N条路径继续探索，摒弃较差的路径。 
 实验证明 ：Hugging Face的一篇博客文章中的实验显示，使用 Beam Search 这种方法，一个1B参数的模型在数学问题上的表现，甚至可以超越一个8B参数的模型。这证明了好的推理流程可以极大地弥补模型规模的不足。其他更复杂的搜索算法，如A*、MCTS等，也都可以应用在此框架下。 
 方法三：模仿学习（Imitation Learning） - 需要微调 
 这种方法是直接教模型如何进行推理，可以看作是上周讲的“后训练（Post-training）”的特例。 
 核心思想 ：我们使用包含“问题-推理过程-答案”的训练数据，通过监督式微调（Supervised Fine-tuning），让模型学会不仅要输出正确答案，还要生成中间的推理过程。 
 挑战 ：如何获取带推理过程的训练数据？ 
 通常我们只有问题和答案。一个可行的办法是： 
 1. 用一个强大的模型（如GPT-4）通过CoT提示生成推理过程。 
 2. 只保留那些最终答案正确的“问题-推理过程-答案”对，作为训练数据。 
 进阶思考 ：我们应该只教模型“完全正确”的推理路径吗？ 
 不一定。如果模型只见过一帆风顺的正确推理，当它在实际应用中走错一步时，它将不知道如何纠正，只会硬着头皮错下去。 
 因此，我们需要教模型 知错能改 。 
 包含错误路径的训练数据 : 一些研究如 String of Search 和 Journey Learning 提出，在构建训练数据时，应故意包含一些错误的探索路径和回溯修正的过程。例如，训练数据可以长这样：“尝试方法A -> 发现错误 -> 回溯 -> 尝试方法B -> 得到正确答案”。这样训练出的模型，才能学会“逆转胜”。 
 知识蒸馏（Knowledge Distillation） : 一个更便捷的方法是，直接用一个已经具备深度思考能力的模型（如DeepSeek-R1）作为老师，生成大量的推理数据，然后让我们的目标模型去模仿学习。DeepSeek的论文也证明了这种方法的有效性，能将R1的推理能力蒸馏到更小的模型上。 
 方法四：以结果为导向学习（Reinforcement Learning） - 需要微调 
 这是以 DeepSeek-R1 系列闻名的做法，核心是使用强化学习（RL），以结果为导向来教模型推理。 
 核心思想 ：模型生成推理过程和最终答案。我们不关心推理过程本身写了什么，只看最终答案是否正确。 
 如果答案 正确 ，模型获得 正向奖励（Positive Reward） 。 
 如果答案 错误 ，模型获得 负向奖励（Negative Reward） 。 
 通过RL，模型会自己探索出那些更容易导向正确答案的推理模式。 
 三、综合案例：DeepSeek-R1 的诞生之路 
 DeepSeek-R1的打造过程极其复杂，融合了我们前面讨论的所有方法，并非单纯的RL。 
 1. R1-0的诞生 (纯RL) : 他们先用RL在 DeepSeek V3 Base 模型上训练，奖励只基于答案正确性和格式（是否使用 <think> 标签）。这得到了一个能自主推理（甚至出现“Aha moment”）但推理过程混乱、难以阅读的 R1-0 模型。 
 2. R1的复杂打造流程 : 
 步骤一 (模仿学习) : 用R1-0生成推理数据，然后投入 大量人力 去修改和润色这些数据，使其变得清晰可读。同时，也用Supervised CoT等方法生成高质量数据。用这些数据通过模仿学习训练出 模型A 。 
 步骤二 (RL) : 对模型A进行RL，得到 模型B 。这次的奖励除了正确率，还加入了一个“语言一致性”的奖励，避免推理过程中语言混杂。 
 步骤三 (模仿学习) : 用模型B在更广泛的任务上生成了60万条推理数据，并用规则过滤掉质量差的。这些数据再混合20万条Self-Instruct数据，通过模仿学习训练出 模型C 。 
 步骤四 (最终RL) : 对模型C进行最后一次RL，主要强化其安全性和帮助性，最终得到我们使用的 DeepSeek-R1 。 
 关键洞察 ： 
 RL的成功很大程度上依赖于一个强大的基础模型。研究发现， DeepSeek V3 Base 在RL训练前，就已经具备了初步的、潜在的推理和自我怀疑能力（如“Wait, I'm overthinking”）。RL的作用是 强化 并 激发 了这些已有的能力。如果基础模型太弱，单纯用RL很难凭空创造出推理能力。 
 四、总结与挑战 
 深度思考模型最大的挑战之一是 计算资源的浪费 。有时模型会进行大量不必要的、冗余的思考。 
 实例：DeepSeek-R1 解答 123 × 456 
 它花了近一分钟的时间来解答这个问题。在算出正确答案 56088 之后，它仍不放心，进行了多轮、多种方式的反复验算： 
 用直式运算法验算。 
 用估算法（120×450）验算。 
 交换被乘数和乘数，拆解另一方再算一次。 
 心算一遍（既然心算能对，前面在做什么？）。 
 将两边数字同时拆解成多项式展开，再相加。 
 最后，它依依不舍地结束了计算。这其中大量的推理过程都是冗余的。如何让模型“把好钢用在刀刃上”，在需要时才进行深度思考，是我们下一堂课将要探讨的内容。

 ｜ 课程概览 
 一、什么是后训练 (Post-training)？ 
 核心需求 ： 将通用的基础模型（Foundation Model）通过进一步学习（Post-training），打造成擅长特定领域、特定语言或特定技能的专家模型。 
 技术方法 ： 后训练在技术上与常规训练无异，可采用预训练（Pre-train）、监督式微调（SFT）或强化学习（RL）等方式。 
 适用对象 ： 后训练的对象不限于原始的预训练模型，也可以是已经过对齐（Alignment）的聊天模型（Chat Model）。 
 二、后训练的核心挑战：灾难性遗忘 (Catastrophic Forgetting) 
 核心悖论： 在教模型新技能的过程中，模型会轻易地遗忘（Forgetting）它原本已经具备的能力。这个现象被称为“灾难性遗忘”，是后训练最大的挑战。 
 普遍性验证： 
 预训练式后训练： 教 LLaMA-2 中文后，模型学会了中文，但丧失了原有的安全防卫能力。 
 监督式微调 (SFT)： 即使使用完全无害的数据对模型进行微调，其安全对齐能力也会大幅下降。 
 跨能力影响： 当专门教模型一项技能（如编程）时，它在其他领域（如数学）的能力会显著退化。 
 三、为什么会发生灾难性遗忘？ 
 根本原因 ：后训练的优化过程只专注于最大化 单一的新任务目标 ，不计代价地调整参数，从而破坏了储存旧知识的参数结构。这就像一场“ 手术成功，但病人死了 ”的悲剧。 
 关键观察： 
 与模型大小无关： 研究表明，更大的模型（在 1B-7B 范围内）并不会更不容易遗忘。 
 与学习效果成正比： 模型在新任务上学得越好，其遗忘原有技能的情况就越严重。 
 LoRA 的局限性： LoRA 等参数高效微调方法，其抗遗忘效果往往来自于 “学得更少，忘得也更少” ，并未从根本上解决问题。 
 四、 如何解决遗忘问题 
 历史启示：经验回放 (Experience Replay) ：早在 2019 年的 GPT-2 时代，研究就已发现，在训练新任务时，混入一小部分（例如 5%）的旧任务数据进行“经验回放”，就能非常有效地防止遗忘。 
 现代困境与演进 ：由于大模型通常不开源其训练数据，直接的经验回放变得困难。因此，解决方案演变为 让模型自己生成“伪”的旧数据 。 
 现代主流策略： 
 自问自答 (Self-Synthesize) ：让基础模型自己生成类似其训练风格的问答对，用于回放。 
 改写 (Paraphrase)： 让基础模型将人工标注的答案用自己的语言风格“换句话说”，再用于训练。 
 自输出 (Self-Output)： 让基础模型直接回答问题，如果答案正确，就用模型自己的输出作为标准答案来训练它自己。 
 与强化学习 (RL) 的关联： 自输出与 RL 的思想高度相似，这或许解释了为何 RL 本身就是一种比较能防止遗忘的技术。 
 五、应用实例与进阶技巧 
 多模态模型训练 ：在教模型听懂语音时，可先让强大的纯文字模型根据语音的文本描述生成回答，再用这个回答作为目标来训练语音模型，从而保护其原有的文本能力。 
 过滤困难样本 ：另一项进阶技巧是，在训练时识别并 忽略掉那些对基础模型来说极难生成的词元（Token） ，不计算其损失。这能避免模型在“学不会”的东西上过度挣扎，反而能更好地保留通用能力。

 这堂课要讲的是 后训练 (Post-training) 与其伴随的 遗忘 (Forgetting) 问题。 
 一、 什么是后训练 (Post-training)？ 
 今天，我们已经拥有很多强大的开源模型，例如 Llama、Google 的 Gemma、DeepSeek，甚至是可微调的 ChatGPT。这些模型本身已具备非常通用的基础能力，就像是从学校毕业、具备基本技能的学生。 
 然而，在许多应用场景中，我们需要一个拥有某种专长的模型。这个专长可以指： 
 特定领域 ：如金融、法律。 
 特定人类语言 ：如中文、韩文、日文。 
 特定程序语言： 如 Verilog。 
 尽管通用模型在这些领域可能具备一定能力，但往往不够专精。因此，我们会准备特定资料，对这些通用模型进行进一步的调整，使其在特定领域表现得更好。这个将通用模型进行进一步学习的过程，就称为 后训练 (Post-training) ，有时也叫做 持续学习 (Continual Learning) 。 
 在本课程中，我们将后训练前的模型称为 基础模型 (Foundation Model) ，后训练后的模型称为 微调模型 (Fine-tuned Model) 。 
 注 ：上堂课讲的对齐 (Alignment)，即从一个预训练模型 (Base Model) 训练得到一个聊天模型 (Chat Model)，也可以视为一种后训练。但本堂课的后训练概念更为广泛：只要你希望为一个现有模型（无论是 Base Model 还是已经对齐过的 Chat Model）添加额外技能，都属于后训练的范畴。 
 后训练的技术方法 
 从技术上讲，后训练与训练语言模型的常规方法并无不同，主要有三种： 
 1. 预训练风格 (Pre-train Style) ：继续用大量文本资料进行文字接龙。 
 2. 监督式微调 (Supervised Fine-tuning, SFT) ：使用问答对 (Instruction-Following) 资料进行训练。 
 3. 强化学习 (Reinforcement Learning, RL) ：根据输出的好坏给予奖励或惩罚。 
 你可以根据手上资料的类型选择合适的方法。 
 二、后训练的核心难题：灾难性遗忘 (Catastrophic Forgetting) 
 虽然技术上没有新东西，但后训练面临一个巨大的挑战： 模型会遗忘它原有的能力 。 
 案例一：教 LLaMA-2 Chat 中文 
 背景 ：LLaMA-2 Chat 虽然能理解中文，但主要用英文资料训练，因此预设用英文回答。我们希望通过中文资料的后训练，让它学会用中文回答。 
 期待 ：模型既能用中文回答，又能 保留 原有的对齐能力（例如，安全防卫能力）。 
 现实 ：后训练后，模型学会了用中文，但 丧失了 安全防卫能力。 
 训练前 ：问它如何获取银行密码，它会拒绝回答，这是正确的行为。 
 训练后 ：再问同样的问题，它会用中文“热心”地教你几种攻击方式。 
 这个现象并非个案。另一篇论文也发现在用中文资料对 Llama-2 Chat 进行后训练后，模型在回答「气候变化如何影响生态系统」时，会陷入不断重复「第一点、第一点…」的崩溃状态。 
 从 ToxiGen （一个检测模型有害言论的基准）的测试结果来看： 
 Llama-2 Base (未对齐)： 说错话的几率高达 25%。 
 Llama-2 Chat (已对齐)： 几率降至仅 0.22%，非常稳定。 
 中文后训练后： 说错话的几率大幅回升，原有的安全性几乎被破坏。 
 案例二：SFT 同样导致遗忘 
 你可能会认为是预训练风格的后训练导致了遗忘，但即使是 SFT 风格的训练也同样如此。 
 一篇早期研究发现，对 ChatGPT 3.5 进行微调会严重损害其安全性， 即便训练资料本身是无害的 。 
 恶意微调： 教模型如何制造炸弹，模型的安全性理所当然地崩溃了。 
 无害微调： 仅仅是帮模型改个名字（例如，让它自称为 "AOA"），也会导致其安全对齐能力大幅下降。 
 常规微调： 使用完全正常的 Alpaca 资料集（如「三原色是什么？」）进行微调，模型的安全性在多个方面依然会变差。 
 案例三：遗忘不仅限于安全性 
 我们实验室的研究也表明，对 Llama-3 进行 SFT 来教授推理、医学、程序代码和工具使用等新技能时： 
 目标任务能力 ：在被教授的领域，模型表现显著提升。 
 安全对齐能力 ：模型的安全性（在 HEXPHI 和 ADVBench 上的测试）急剧下降，从几乎 0% 的错误率飙升。 
 遗忘不仅仅破坏安全性。另一项研究发现，当你专门教模型一项技能时，它在其他领域的能力会下降： 
 教模型使用工具，它的数学和程序代码能力变差。 
 教模型数学，它的工具使用和程序代码能力变差。 
 教模型写程序代码，它的数学和工具使用能力大幅暴跌。 
 案例四：多模态学习中的遗忘 
 当我们尝试教 LLaMA 这样的文字模型“听懂”声音时，遗忘问题同样严重。 
 训练初期 (1 epoch) ：模型尚能遵循指令，以 JSON 格式输出答案。虽然答案内容（情绪识别）是错的，但它还记得什么是 JSON。 
 训练后期 (3 epochs) ：模型学会了正确识别语音中的情绪，但 完全忘记了 如何生成 JSON 格式。 
 三、为什么会发生灾难性遗忘？ 
 原因非常直观：在后训练中，我们只专注于单一的训练目标（如提升程序代码能力），而完全忽略了模型在其他能力上的表现。优化演算法会不计代价地调整参数以达成当前目标，从而破坏了储存其他能力的参数结构。 
 这个现象被称为 灾难性遗忘 (Catastrophic Forgetting) 。就像一场手术，虽然成功切除了病灶，但病人却失去了其他重要功能—— 手术成功，但病人死了 。 
 一些观察 ： 
 与模型大小无关： 研究表明，更大的模型（在 1B 到 7B 范围内）并没有表现出更轻微的遗忘。 
 与学习效果成正比： 模型在目标任务上学得越好，它遗忘其他技能的情况就越严重。 
 LoRA 的作用： 使用 LoRA (Low-Rank Adaptation) 似乎能减轻遗忘，但这往往是因为 “学得更少，忘得也更少” 。LoRA 限制了参数的改动范围，代价是模型在新任务上的学习效果也较差，这并非根本的解决方案。 
 四、如何解决遗忘问题？ 
 来自 2019 年的启示：经验回放 (Experience Replay) 
 早在 2019 年，我们实验室就在 GPT-2 上进行了多任务学习的探索。我们发现，当教完模型一个任务（如阅读测验 SQuAD）再教下一个任务（如生成 SQL）时，它在第一个任务上的表现会急剧下降。 
 当时我们找到了一个非常有效的解决方案： 经验回放 (Experience Replay) 。 
 方法 ：在训练新任务（任务二）时，混入一小部分旧任务（任务一）的资料。 
 效果 ：仅需混入约 5% 的旧资料，就能非常有效地防止模型遗忘旧技能。 
 当时的结论是，灾难性遗忘似乎不是一个大问题，因为解决方案太简单了。 
 现代的困境与解决方案 
 然而，在 2025 年的今天，这个问题再次变得严峻，因为像 Meta 这样的大公司只发布模型，而 不发布 其完整的训练资料。我们无法拿到 Llama-2 Chat 的原始对齐资料来进行经验回放。 
 幸运的是，2019 年的研究也考虑了这种“无法获取旧资料”的情境，并提出了解决方案： 伪经验回放 (Pseudo Experience Replay) 。 
 方法 ：如果没有旧资料，就让 模型自己生成 。让训练好的模型（例如，在 SQuAD 上训练过的 GPT-2）自由生成文本，它会产生类似其训练资料的内容（自问自答的阅读测验）。 
 应用 ：在训练任务二时，将基础模型生成的“伪”任务一资料混入训练集中。 
 实验证明，使用模型自己生成的资料，同样可以有效防止遗忘。 
 现代后训练的主流方法 
 如今，基于“经验回放”和“伪经验回放”的思路，衍生出了多种主流的抗遗忘方法。核心思想是： 让模型用“自己的话”来学习 。 
 1. Safety-Tune ：在微调时，混入少量（如 3%）的安全对齐资料，这就是标准的经验回放。 
 2. Self-Synthesize Rehearsal (Magpie) ：让基础模型自问自答，生成 instruction-following 格式的资料，然后将这些“自己合成”的资料用于后续训练。 
 3. 改写 (Paraphrase) ：不用人工标注的答案，而是让基础模型将标准答案“换句话说”，再用改写后的答案去训练。因为改写后的句子更符合模型自身的语言风格，学习起来更不容易引起冲突。 
 4. 自输出 (Self-Output )： 
 将问题输入基础模型，让它直接生成答案。 
 通过某种机制（如对数学题核对答案，对程序代码进行编译）验证答案是否正确。 
 如果答案正确，就用 模型自己的输出 作为标准答案来训练它自己。 
 这种方法非常有效，因为它本质上是在强化模型已有的、正确的知识路径。 
 与强化学习 (RL) 的关联 ：Self-Output 的思想与 RL 非常相似。RL 也是让模型生成答案，然后根据好坏给予反馈。这或许可以解释为什么 RL（如 DPO）通常被放在训练的最后阶段，它可能本身就是一种比较能防止遗忘的技术。 
 五、应用实例与进阶技巧 
 多模态模型训练 ：在训练语音模型时，可以先用文字描述语音的特征（如语速、情绪、口音），让一个强大的纯文字模型根据这些描述生成回答。然后，再训练语音模型，使其在听到实际语音时，能输出与该文字模型一致的答案。我们实验室的 Desktop2 等模型就是采用这种方法，取得了很好的效果。 
 过滤困难样本： 一篇研究发现，人工标注的答案中，有些词元（token）对于基础模型来说是极难生成的（即低机率）。如果在训练时，直接 忽略 掉这些对模型来说过于困难的学习目标（不计算这些 token 的损失），反而能让模型学得更好，同时更好地保留原有能力。 
 六、总结与建议 
 后训练是一个强大的工具，但务必警惕 灾难性遗忘 。 
 当你看到有人宣称“我微调的 Llama-3 在某个任务上打败了 GPT-4o”时，这完全是可能的。但你需要问一个关键问题： 为了这个胜利，它损失了多少通用能力？ 这个模型会不会变成一个只会解特定任务的“偏科生”，甚至连基本的人类语言都说不好了？ 
 给进行后训练的同学的建议 ： 
 1. 全面评估 ：除了评估模型在目标任务上的表现，一定要检查它在其他基础任务上的能力是否被保留。 
 2. 采用抗遗忘策略 ：优先考虑使用 Self-Output 或相关的变体（如 Paraphrase、Pseudo Replay）。让模型用自己的、或与自己风格相近的资料进行学习，是目前最有效的防止遗忘的方法。

 预训练 (Pre-training)： 打下基础。使用海量网络资料，赋予模型庞大的世界知识和语言能力，是决定模型能力上限的根本。

 二、对齐的“画龙点睛”之效：少量高品质资料的力量 
 品质 > 数量： 对齐成功的关键不在资料量，而在于其高品质。数千条精心挑选的资料，效果远胜于百万条低品质资料。 
 知识蒸馏（Knowledge Distillation） ：高效捷径。透过模仿 GPT-4 等强大模型的回答来进行学习，是许多低成本模型成功的秘诀。 
 对齐的本质 ： 调整风格。对齐并非重塑模型的知识库，而是调整其 表达风格 和 行为模式 ，如学会何时结束对话、使用礼貌用语等。

 资料多样性 ： 理解的关键。模型需要看到关于同一概念的多种不同表述，才能学会泛化和抽象，从而真正理解世界，而不仅是死记硬背。 
 海量与高质 ： 训练的门槛。顶尖模型需要万亿（T）级别的 Token 进行训练，且资料的品质、清理、去重对于最终性能至关重要。 
 资料枯竭 ： 未来的挑战。随着模型规模的增长，高品质训练资料正逐渐面临被耗尽的风险。

 一、模型训练的核心框架：预训练与对齐 
 今天我们要分享的是“预训练-对齐”(Pretrain-Alignment) 这一模式的强大之处与及极限。 
 大家知道，今天在训练语言模型时，基本上就是分成三个阶段： 
 预训练 (Pre-training) ：通过大量网络爬到的资料，让机器有基本的文字接龙的能力。 
 监督式微调 (Supervised Fine-tuning, SFT) ：也叫指令微调 (Instruction Fine-tuning)，就是告诉模型，看到某个输入的时候，某个样子的输出才是对的。 
 基于人类回馈的强化学习 (RLHF) ：由使用者来告诉机器，什么样的回答是比较好的，什么样的回答是比较不好的。 
 在第二阶段跟第三阶段会引入人类。在第二阶段需要人类提供正确的答案，在第三阶段需要人类提供回馈。我们把需要人类参与的阶段叫做 对齐 (Alignment) 。 
 在这里，我们所谓的对齐指的是第二阶段的监督式微调加上第三阶段的 RLHF。对齐指的是我们想要达成的一个目标：我们希望机器的输出是与人类的价值观和需求对齐的。这个步骤很多时候又叫做 微调 (Fine-tune) ，所以等一下课程里面讲到微调的时候，指的也就是对齐这个步骤。在第二阶段和第三阶段，只是根据预训练已经得到的参数进行微调。 
 二、对齐的“画龙点睛”之效：少量高质量资料的力量 
 有人可能会想说，预训练根本没什么用，预训练完的模型根本就没有办法使用。举例来说，如果你拿一个只有做预训练、没有做对齐的模型，比如 LLaMA-2-7b-base ，问它“what is machine learning”，它虽然知道基本概念，但讲着讲着就会不断重复，停不下来。 
 而对齐确实大幅改变了模型的行为。 LLaMA-2-7b-chat 是做过对齐的版本，你问它同样的问题，它会好好地、条列式地回答你，看起来非常像模像样。 
 在数据上，对齐前后的影响也非常大。以 LLaMA-3-70B 在 MT-Bench 上的表现为例，仅预训练的 Base 模型得分是 2.7，而经过对齐的 Instruct 模型得分高达 8.63，这是天与地的差别。看到这里你可能会觉得对齐是 MVP，Base 模型就是“躺赢狗”。 
 但实际上，相对于预训练，对齐这个步骤做的事情非常少。LLaMA 2 的技术报告里告诉你，他们在做 SFT 的时候只用了 27,540 笔资料而已。另一篇论文《Less is More for Alignment》中，作者只用了一千笔精挑细选的资料来微调模型，就让模型表现得非常好。 
 所以对齐就像是“画龙点睛”。龙的主体是预训练画出来的，但光画龙的身体是没有用的，得把眼睛点上去，龙才会飞起来。这个点睛的步骤就是对齐。但是点的位置也很重要，意味着 对齐的资料虽然不用多，但它的品质很重要 。 
 什么样的对齐资料才是好的呢？今天仍然非常难说清楚。有一篇论文发现，用来自“弱智吧”的 240 笔问题，比用数千笔“知乎”的问答资料微调模型效果更好。作者猜测，也许是“弱智吧”的问题特别丰富多样。但一个更重要的细节是，它的答案是用 GPT-4 生成的。所以实际上，这相当于对 GPT-4 做了 知识蒸馏 (Knowledge Distillation) ，把 GPT-4 当作你的老师，也许这才是模型进步飞速的原因。 
 曾经有很长一段时间，大家不知道怎么做对齐，后来有人就想说，ChatGPT 那么强，我们来对它做逆向工程，跟 ChatGPT 学习，这就是知识蒸馏。很多知名的模型，尤其是那些炫耀只用了一点点钱就训练起来的模型，通常就是指他们去对 ChatGPT 或其他很强的模型做对齐，模型就可以瞬间得到能力的暴增。 
 清理资料也是必要的。老师模型不一定能够得到好的答案，所以你会挑品质比较高的资料。有趣的是，一篇名为《Long is more for alignment》的论文发现，从 Alpaca 的五万多笔资料里，只挑选答案长度最长的那一千笔资料来训练，得到的模型竟然可以打败其他用更复杂方法筛选资料的模型。 
 我们实验室甚至发现，那些用来提问的“问题”根本就不重要。我们做了一个实验：从网络上随便选一个句子，把后半段截掉，只把前半段丢给 GPT-4 让他自己做文字接龙，把后半段接出来，然后让我们的模型学习“前半段 -> GPT-4 的续写”。你可能会想，这输入甚至不是一个问题，有用吗？非常有帮助。我们用这种莫名其妙的资料去微调 Mistral 的 Base 模型，得到的分数比 Mistral 官方自己微调出来的 Instruct 模型还要更高。 
 为什么对齐前后，虽然模型的答案好像很不一样，但对齐却这么容易呢？因为模型实际的行为，也许差异没有这么大。一篇论文分析发现，对齐前后，绝大多数词汇（Token）的输出机率是没有变化的（Unshifted）。只有极少数词汇的机率有很大变化（Shifted），而这些词汇通常是一些连接词，或者打招呼的词汇，以及很关键的 结束符号 。模型做文字接龙时，一步错，步步错，中间有个地方错了，接下来接的东西就会非常不一样。所以虽然答案看起来差很多，但模型的行为并没有非常大的差距。 
 这也解释了为什么 自对齐 (Self-Alignment) 这个技术是可能成功的。这个技术让没有对齐的模型自己生成多个答案，再自己评分，然后用好的答案来强化学习。如果你了解从没有对齐变成有对齐的变化其实没有那么大，就比较容易想象为什么这个方法可行。 
 三、真正的基石：预训练决定模型能力的上限 
 我们知道对齐并没有对预训练的模型造成非常大的变化，所以今天之所以对齐以后模型能力很强，是因为预训练非常的有效。那么，要怎么样才能够做出有效的预训练模型呢？ 
 一篇名为《Physics of Language Models》的系列论文提供了一个思考方向。他们发现，假设在你的预训练资料里面，每个人只出现一次，模型会有巨大的误解。比如它会认为“高松灯是宇秋女子学员高一学生、是天文社的一员”这些东西合起来，才是一个实体，才是“MyGO 的主唱”。 
 但是，如果今天同一个人的资讯，有各种不同的介绍方式，模型就知道原来“高松灯”是一个实体，他有不同的性质。我们发现， 同样的资料，同一个人有很多不同的介绍方式，其实对于预训练是一件重要的事情 。而且不需要所有的角色都有多种版本的介绍，只要训练资料里面有一部分角色有多种版本的介绍，就足以让模型学到更泛用的知识，强化模型理解的能力。 
 现在一个好的预训练模型，都需要大量的资料。LLaMA 3 用了 15T (兆) 个 Token。这个资料量多到已经有人担心，会不会网络上所有可以取得的资料，已经要被用尽了。有篇论文估测，大概在 2028 到 2030 年，我们就会用尽网络上所有能够训练的 Token。不过你可能会想，我上哪里去找 15T 的资料呢？我告诉你，Hugging Face 已经释出了一个叫做 FineWeb 的资料集，正好有 15T，所以现在资料是唾手可得的，你差的只是算力而已。 
 资料的品质也是非常重要的。微软一篇名为《Textbooks Are All You Need》的论文发现，他们用 GPT-3.5 生成的、类似教科书风格的高品质资料来预训练模型，效果远好于网络上爬取的大量代码。这告诉我们，只要你的资料像是教科书，在预训练的时候就很有用。 
 另外，直接从网络上爬到的资料，往往没有办法直接用，需要经过多个步骤进行清理。一篇名为 Refined the Web 的论文就展示了他们打造模型时复杂的资料清理过程，其中好几个步骤都是在去除重复的文章。为什么去除重复的文章很重要呢？因为已经有文献发现，在固定的算力下，你应该要尽量让你的模型看不同的资料，而不是让他反复看一样的内容。我们的实验室在语音模型上也得到了同样的结论：模型看过越多不同的声音，它的表现就越好。 
 四、对齐的极限与预训练的“烙印” 
 我们已经看到，对齐主要就是改变了模型书写的风格。既然如此，那对齐这件事情，会不会非常有极限呢？ 
 早在 2023 年年初就已经有人发现，那些 fine-tune 过后的模型，表面上看起来很强，但实际上他输出的答案看起来像模像样，内容却是错的。所以看起来对齐是有它的极限的。 
 到底什么事情是可以通过对齐影响模型，什么是学得会的，什么是学不会的呢？一篇论文将知识分成四类： 模型本来就会的 (Highly known)，问对方法就会的 (Maybe known)，采样有机会答对的 (Weakly known)，和怎么都学不会的 (Unknown)。 你直觉可能会觉得应该拿 Unknown 的资料训练模型才有用，但实验结果跟人类正常的直觉是相反的。 
 研究发现，当模型开始学习这些 Unknown 资料的时候，就是它在开发集上坏掉的时候了。看起来，让模型学这些他本来就有可能知道的东西是有帮助的，但当你让模型去学那些他不知道的东西的时候，你反而在破坏模型的能力。其中， 最有帮助的是那些“Maybe known”的资料 ，就是模型本来就知道，只是你没有用很懂它的问法去问。 
 这样说起来，其实 RLHF 是一个很烂的好方法。在做 RLHF 的时候，每一笔拿来训练的资料，都是模型自己的答案。所以 RLHF 的目标并不是叫模型会它本来完全不会的东西，而是激发模型本来的潜力。他有时候做得好，有时候做得不好，做得好就鼓励他，做得不好就惩罚他，但是你不会硬逼他做一个他本来就做不到的事情。 
 预训练可能会留下一些后遗症。一篇论文指出一个现象，GPT-4 你叫他做 ROT13 的解码（字母移动 13 个位置），他可以做到，但你叫他做 ROT8 的解码，他就没办法答对。作者去分析资料集发现，ROT13 相关的资料在网络上出现的特别多，正好对应到 GPT-4 的能力。所以资料的分布对模型造成了一定程度的影响。 
 这也意味着，模型看到不该看的东西以后，你很难真的抹除模型的记忆。网络上有很多不该看的东西，你以为对齐可以让模型避免讲出这些话，但也许这些知识都藏在模型的心中。一篇论文就分析发现，对齐确实可以让模型表面上比较不会说脏话，但这些不该讲的词汇仍然存在模型的参数里面。他们发现，与脏话有关的参数（神经元）在对齐后不见了吗？没有，它们还在模型的心里。真正改变的是激发这些参数的通路被抑制了。他只是不去激发这些参数而已，这些参数仍然在模型的心里。 
 这就是为什么你常常会看到这样的一个梗图：模型在网络上得到大量的资讯，他就像是一个怪物一样，对齐只是给他戴了一个面具，让他看起来人模人样，但是你不知道实际上，在他的参数里面，在他的心里，他在想什么样的事情。 
 总结来说，Pretrain-Alignment 非常强大，预训练时模型就已经很厉害，对齐只是画龙点睛。但它也有很大的极限，对齐往往只是强化模型本来就会的能力，难以让模型学会新的技能。那么，怎么透过微调教语言模型新的东西，怎么做有效的微调，我们就下回分解。

 一、课程简介与核心前提 
 核心问题 ： Transformer 在处理长序列时，推理成本过高，是否有更优的替代方案？ 
 核心观点 ： 每种网络架构都有其存在的理由（Raison d'être）。Transformer 的成功并非因其记忆力，而是因其 训练时可高度平行化 的巨大优势。 
 二、Transformer vs. RNN：一场关于效率的战争

 三、RNN 的逆袭：Linear Attention 的惊人发现 
 核心洞见 ： 一个简化的 RNN（移除“反思/遗忘”机制）在数学上 等价于 一个没有 Softmax 的 Self-Attention。 
 关键概念：Linear Attention 。它是一种混合架构，具备双重身份： 
 训练时 ：可展开为类 Transformer 结构，进行 平行化训练 。 
 推理时 ：可作为 RNN 运行，实现 高效推理 。 
 本质 ： Linear Attention = 无反思机制的 RNN = 无 Softmax 的 Self-Attention 。 
 四、从 Linear Attention 到 Mamba：补完动态记忆

 一 、课程简介与前提 
 本堂课我们将深入探讨网络的架构。在作业4中，我们将训练一个Transformer。Transformer是当前应用极为广泛的神经网络架构，其用途不仅限于大型语言模型，还延伸至语音合成、图像生成等多个领域。本次作业特意选择用Transformer生成图片，旨在展示其多样化的应用能力，破除其仅限于语言模型的普遍印象。 
 本课的主轴是探讨哪些架构有可能取代Transformer，以及Transformer面临的潜在竞争对手。在这些竞争者中，最知名的当属Mamba。Mamba是一种蛇的名称，因此一个常见的比喻是描绘Transformer（变形金刚）与Mamba蛇的对决。然而，本课将揭示，它们在本质上非常相似。我们将主要探讨以Mamba为代表的一系列相关架构。 
 课程的重点在于理解神经网络架构的设计哲学。每一个架构的诞生都有其存在的理由，是为了解决特定问题而被设计的。 
 CNN（卷积神经网络）存在的理由是什么？ CNN是全连接前馈网络的一个特例。它通过引入“感受野”（Receptive Field）和“参数共享”（Parameter Sharing）的概念，移除了不必要的权重连接。CNN是专门为处理图像数据而设计的，它依据图像的局部相关性等特性，显著减少了模型参数，从而有效避免过拟合，使得模型能用更少量的数据成功训练。 
 残差连接（Residual Connection）存在的理由是什么？ 其设计初衷与CNN不同。在残差网络出现之前（约2015年底），研究者发现训练一个非常深的网络，其性能反而会下降。例如，一个56层的网络在测试集和训练集上的表现均劣于20层的网络。这表明问题并非源于过拟合，而是更深的网络在优化（Optimization）上本身就更为困难。残差连接的存在，正是为了让更深的网络能够被有效训练，它通过构建更平滑的损失曲面（Loss Surface），使得优化过程更加容易。 
 这两个例子旨在说明，每一种架构都有其独特的设计初衷。当我们接触一个新的架构时，应当思考它被设计出来是为了解决什么问题。 
 二、Transformer vs. RNN：一场关于效率的战争 
 Transformer的出现是基于什么样的理由？在此，我们重点讨论其核心组件： 自注意力层（Self-Attention Layer） 。 
 在讨论Self-Attention之前，有必要了解它所取代的传统架构——RNN（循环神经网络）和LSTM。理解Self-Attention如何取代RNN，将有助于我们预见为何Mamba等新架构可能取代Self-Attention。事实上，Mamba在本质上就是一种RNN，这预示着RNN正在以新的形式回归。 
 RNN、Self-Attention和Mamba，它们共同要解决的核心问题是：输入一个向量序列（vector sequence），并输出另一个向量序列。这些层的功能是对输入信息进行混合与转换，生成新的序列表示。

 训练神经网络的核心步骤是前向传播计算输出、计算损失、然后反向传播更新参数。Transformer的厉害之处在于，它可以在训练时一次性、并行地计算出整个输入序列在每个位置的对应输出。整个Self-Attention的计算过程可以被分解为一系列矩阵运算，这是一种非常 GPU友好（GPU-friendly） 的计算模式，能够最大限度地发挥GPU的并行计算能力。 
 相比之下，RNN的串行依赖性使其在训练时难以并行化。计算 必须等待 完成，这种等待状态是GPU效率的瓶颈。

 Self-Attention : 训练时 高度并行，效率极高； 推理时 处理长序列的计算和内存成本高。

 在2017年至今，学术界和工业界为了追求更快的训练速度，普遍选择了Self-Attention。

 四、从 Linear Attention 到 Mamba：补完动态记忆 
 线性注意力的概念早在2020年就已提出，但并未得到广泛应用，因为它在实际性能上通常不及标准的Transformer。 
 其根本原因在于 缺少了Softmax 。Softmax不仅是归一化，更引入了一种 竞争机制 。一个信息点的重要性是相对的，取决于序列中是否存在更重要的信息点。在Softmax的作用下，注意力权重会动态调整，从而实现一种隐性的“遗忘”和“聚焦”。而线性注意力的记忆只会不断累加，无法动态改变或遗忘。

 1. Retention Network (RetNet) : 在 前引入一个固定的衰减因子 ， 使过去的记忆随时间逐渐淡忘。此操作同样可以并行化。 
 2. Gated Retention : 将固定的 升级为随时间变化的 ，让模型根据当前输入自主学习何时遗忘、遗忘多少。 
 3. 更复杂的Reflection机 制 : 通过设计一个门控矩阵 ，与 进行逐元素相乘，从而精细地控制记忆中每一部分的保留或遗忘。 只要 的结构设计得当, 依然可以保持训练的并行性。Mamba正是采用了这种复杂的门控机制，并成为第一个在性能上真正能与Transformer抗衡的线性注意力变体。 
 4. DeltaNet的视角: DeltaNet提出了一种新颖的更新范式，其更新公式在数学上等价于一次 梯度下降（gradient descent） 。在这个视角下，隐藏状 被视为模型的参数，每一步的更新都是在优化一个目标：让通过 （query）从记忆中提取的信息，与 （ value）尽可能接近。最近备受关注的Titan模型正是基于这一思想。 
 五、当代趋势与未来展望 
 如今，以Mamba为代表的这类新型网络架构已经不再是理论模型。研究者已经成功用它们训练了数十亿乃至上千亿参数的大型语言模型（如GEMBA、Minimax-01），证明了其在实际大规模应用中的可行性和竞争力。这些概念也正被应用于计算机视觉等领域，例如，图像生成模型SENA就利用了线性注意力来提升速度。当然，也有研究（如MambaOut）指出，在某些任务上（如图像分类），这类长距离依赖架构可能并非必需。 
 当前的一个重要研究趋势是，不再从零开始训练这些新架构。由于从头训练一个模型来超越Llama等现有SOTA模型极为困难，研究者们转向了一种更务实的方法：在现有的大型语言模型（如Llama）的基础上，将其中的Self-Attention层替换为Mamba等新架构，然后进行微调（fine-tuning）。这种方法能够更高效地验证新架构在成熟模型体系中的实际作用。

 二、背景与挑战：为何选择纯粹的端到端强化学习 
 Kimi-Researcher 选择端到端强化学习（RL），旨在从根本上突破传统智能体开发的瓶颈。以往的方法， 无论是 基于工作流的僵化系统 ，还是 依赖监督微调（SFT）的模仿学习 ，都普遍存在扩展性差、泛化能力弱、依赖高成本人工数据等局限。

 RL训练核心：追求稳定与效率的艺术 
 稳定训练 ： 采用 REINFORCE 算法，并通过 在线策略训练 (On-policy Training) 和 负样本控制 (Negative Sample Control) 等关键技术，有效避免了熵塌陷，确保了训练过程的稳定和持续改进。 
 精巧的奖励机制 ： 设计了 格式奖励 (Format Reward) 与 正确性奖励 (Correctness Reward) ，并创新性地引入 伽马衰减因子 (gamma-decay factor) ，以此鼓励模型探索更短、更高效的正确路径。 
 系统架构创新：应对长程与动态挑战 
 上下文管理 (Context Management) ： 通过精巧的上下文管理器，解决了长时程任务中观测空间过大的问题，支持超过 70 个搜索查询 、上下文窗口达到 数十万 token 的复杂任务。 
 完全异步回放 (Fully asynchronous rollout)： 实现了高效的异步回放管道，并开创性地设计了 层级式部分回放 (Turn-level partial rollout) ，显著提升了训练效率和 GPU 利用率。 
 强大的沙盒环境 (Robust sandbox)： 基于 Kubernetes 构建了统一的沙盒架构和模型上下文协议 (MCP)，实现了工具的安全隔离、零停机调度和状态化通信，为智能体提供了安全可靠的执行环境。 
 数据策略：从零到一合成高质量训练数据 
 数据稀缺性突破 ： 针对高质量智能体（agentic）数据集稀缺的难题，开发了 完全自动化的流程 来大规模合成多样化、高质量的训练数据，无需手动标注，解决了数据瓶颈。 
 任务设计精妙 ： 精心设计了 以 工具为核心、依赖工具 的任务，促使智能体不仅学会了何时调用工具，更学会了如何在复杂场景中有效协调和使用工具。

 传统智能体（agent）开发存在一些关键局限性： 
 基于工作流的系统（Workflow-Based Systems）： 多智能体（Multi-agent）工作流会为专门的智能体分配角色，并使用基于提示（prompt-based）的工作流来协调这些智能体。尽管这种方法有效，但它们与特定的 LLM （大型语言模型）版本紧密绑定，并且随着模型或环境的变化，需要频繁进行手动更新，这降低了系统的可扩展性（scalability）和灵活性（flexibility）。

 动态环境（Dynamic Environments）： 智能体必须适应不断变化的条件，因为即使是相同的查询，随着时间的推移也可能产生不同的结果。目标是在存在分布偏移（distribution shifts）的情况下，仍能实现鲁棒的泛化（robust generalization）。

 强化学习（RL）训练 
 该模型主要使用 REINFORCE 算法进行训练。我们观察到以下因素有助于训练更加稳定： 
 在线策略训练（On-policy Training）： 生成严格的在线策略（on-policy）数据至关重要。在训练过程中，我们禁用了大型语言模型（LLM）引擎机制，例如工具调用格式强制器，以确保每个轨迹（trajectory）完全基于模型自身的概率分布生成。 
 负样本控制（Negative Sample Control）： 负样本会导致 token 概率降低，这会增加强化学习（RL）训练期间出现熵塌陷（entropy collapse）的风险。为了解决这个问题，我们策略性地丢弃了一些负样本，使模型能够在更长的训练时间内持续改进。 
 Kimi-Researcher 使用结果奖励（outcome rewards）进行训练，旨在动态的训练环境中提供持续的偏好。 
 格式奖励（Format Reward）： 如果轨迹包含无效的工具调用，或者上下文/迭代次数超出了最大限制，模型将受到惩罚。 
 正确性奖励（Correctness Reward）： 对于没有格式错误的轨迹，奖励是根据模型的答案与真实值（ground truth）之间的比较来确定的。 
 为了提高效率，我们对正确的轨迹应用了伽马衰减因子（gamma-decay factor）。具体来说，步骤 的奖励变为 ，其中 是结果奖励， 是步数，而 代表伽马衰减系数。这鼓励模型发现更短、更高效的探索路径。例如，虽然两条正确的轨迹最终可能获得相同的奖励，但较短的轨迹会为其初始行动赢得更高的奖励。 
 上下文管理（Context Management） 
 在长时间的研究任务中，智能体（agent）可能会接触到海量的观察信息。一个没有内存管理能力的简单智能体，在不到 10 次迭代（iterations）内就很容易超出其处理极限。为了解决这个问题，我们设计了一种上下文管理（context-management）机制，它允许模型保留重要信息，同时丢弃不必要的文件，从而将单个推理轨迹（rollout trajectory）扩展到超过 50 次迭代。一项早期的消融研究（ablation study）表明，经过上下文管理训练的模型能够多使用 30% 的迭代次数，这使得它能够获取更多信息并达到更高的性能。

 临床证据审查 
 Query：截至2025年，各大国际指南对不同风险人群使用低剂量CT筛查肺癌的最新共识是什么？ 
 截至2025年，主要的国际指南建议对高危人群每年进行低剂量CT（LDCT）筛查，这些人群主要定义为年龄在50-80岁、至少有20包年吸烟史的当前或曾是重度吸烟者，通常还附带戒烟不超过15年的要求。然而，对于从不吸烟者或低风险人群，由于缺乏净收益大于潜在危害的充分证据，强烈不建议进行常规LDCT筛查。虽然承认家族史、职业暴露和慢性阻塞性肺疾病（COPD）等其他风险因素，但它们通常与吸烟史结合考虑，而非作为广泛筛查的独立标准。 
 肺癌低剂量CT筛查共识（截至2025年） 
 1. 高危人群筛查建议（当前/曾是重度吸烟者） 
 1.1. 高危 个体的一般标准 
 截至2025年，主要国际指南关于肺癌低剂量计算机断层扫描（LDCT）筛查的共识，主要针对高危人群，这些人群主要根据显著的烟草吸烟史来界定。核心标准始终包括筛查的最低起始年龄（通常为50或55岁）、可量化的吸烟史（通常至少20至30包年），以及戒烟的适用时限（通常在过去15年内）。“包年”被标准化为每天平均吸一包烟持续一年；因此，20包年的吸烟史可以通过每天吸一包烟持续20年或每天吸两包烟持续10年来达到。当个体超过年龄上限（通常80岁）、戒烟超过15年，或出现显著限制预期寿命或影响其接受治愈性肺部手术能力/意愿的健康状况时，通常建议停止筛查。这种侧重于重度吸烟者的方法，根植于大型随机对照试验的证据，如国家肺癌筛查试验（NLST）和NELSON试验，这些试验均表明LDCT筛查在类似高危人群中具有显著的死亡率益处。例如，NLST试验显示，对于年龄在55至74岁、吸烟史至少30包年且当前吸烟或已戒烟不超过15年的个体，每年进行三次LDCT筛查与胸部X光检查相比，肺癌死亡率相对降低20%。 
 这些具体标准的制定基于益处与危害的平衡。LDCT筛查已被证明可以发现处于更早期、更易治疗阶段的肺癌，从而改善生存结果。然而，筛查也伴随着潜在危害，包括假阳性结果导致不必要的侵入性手术、偶然发现、辐射暴露和过度诊断。因此，指南旨在识别一个受益于早期发现大于潜在危害的人群。强调包年吸烟史和戒烟年限，反映了累积烟草暴露与肺癌风险之间的直接关联，以及戒烟后风险的逐渐降低。年龄标准也至关重要，因为肺癌发病率随年龄增长而增加，老年个体更有可能患有合并症，这可能影响筛查的净收益。在开始筛查之前，共享决策是关键组成部分，确保个体理解LDCT筛查的潜在益处和可能的弊端。此次对话还应包括对当前吸烟者的戒烟咨询，因为戒烟仍然是降低肺癌风险最有效的方式。 
 1.2. 主要组织（USPSTF、ACS、NCCN、IASLC、CHEST）的具体指南 
 主要国际组织已发布使用LDCT进行肺癌筛查的指南，在针对高危人群的核心资格标准上高度一致，尽管在年龄范围、包年要求和额外风险因素的考量上存在一些差异。美国预防服务工作组（USPSTF）建议对50至80岁、有20包年吸烟史且目前吸烟或已戒烟不超过15年的成年人，每年进行LDCT筛查。当一个人已戒烟15年或出现显著限制预期寿命或接受治愈性肺部手术能力/意愿的健康问题时，应停止筛查。这是一项B级建议，表明中等确定性存在中等净收益。2021年的这次更新扩大了符合条件的筛查人群，相比其2013年的建议（当时是55-80岁，30包年）。 
 美国癌症协会（ACS）在其2023年指南更新中，也建议对50至80岁、有20包年以上吸烟史且目前吸烟或曾吸烟的个体，每年进行LDCT筛查。与USPSTF类似，如果一个人出现显著限制预期寿命或接受治愈性治疗能力/意愿的健康问题时，应停止筛查。ACS强调共享决策和戒烟咨询。美国国家综合癌症网络（NCCN）在其2024年指南（含2025年1.0版洞察）中，建议对50岁或以上、吸烟史≥20包年或吸烟史≥20年的个体每年进行LDCT筛查（1类推荐）。NCCN还包含了更广泛的额外风险因素，这些因素可能促使筛查讨论，例如氡暴露、职业暴露（如石棉、二氧化硅、镉、砷、铍、铬、柴油烟雾、镍、煤烟、烟灰）、个人癌症史（肺癌、淋巴瘤、头颈部或吸烟相关癌症）、肺癌家族史、COPD或肺纤维化，以及既往胸部放射治疗史。NCCN承认关于最佳筛查持续时间和停止年龄的不确定性。 
 美国胸科医师学会（CHEST）在其2021年指南中，与50-80岁的年龄范围和20包年以上吸烟史（针对已戒烟不超过15年的个体）保持一致。CHEST还考虑了其他风险因素，如氡或职业暴露、个人癌症史、COPD或肺纤维化以及肺癌家族史。国际肺癌研究协会（IASLC）一直积极推动和完善全球肺癌筛查。虽然并非总提供明确的数值标准，但IASLC支持基于证据的筛查，并为共识声明和质量指标的制定做出了贡献。例如，IASLC 2023年的一份路线图讨论了未来展望，包括需要研究不符合当前资格标准的个体筛查，特别是某些东亚人群中从不吸烟者肺癌发病率上升的情况。IASLC还强调了健全项目实施的重要性，包括戒烟服务和多学科团队。

 这些指南普遍反映了一项共识：即对于主要通过年龄和吸烟史界定的高危人群，年度LDCT筛查能够带来净收益。USPSTF和ACS近期更新中扩大筛查资格标准（例如，将年龄降至50岁，包年数降至20包年），旨在将更多可能受益的个体纳入其中，尤其是那些吸烟暴露较少却可能患肺癌的女性和黑人。然而，在实施过程中仍存在挑战，包括即使在符合条件的人群中，筛查的接受率也偏低；同时，需要建立健全的基础设施以有效管理筛查项目，包括提供戒烟支持和对检出结节进行恰当管理。 
 2. 从不吸烟者或低风险人群的筛查建议 
 2.1. 缺乏常规筛查的证据 
 截至2025年，主要国际指南普遍达成共识，不推荐对从不吸烟或根据极少/无吸烟史被认定为低风险的个体进行常规肺癌低剂量CT（LDCT）筛查。这一立场主要是由于缺乏有力证据表明在这些人群中，筛查的益处能够大于危害。例如，建立了LDCT筛查有效性的里程碑式试验，如国家肺癌筛查试验（NLST）和NELSON试验，其入组对象均为重度当前或曾吸烟者，这使得其研究结果难以推广到从不吸烟者或轻度吸烟者。当肺癌的检验前概率较低时，筛查的潜在危害，包括导致不必要侵入性手术的假阳性结果、辐射暴露和过度诊断，会变得更为显著。例如，加拿大预防保健工作组明确建议不使用LDCT对所有其他成年人进行肺癌筛查，无论其年龄、吸烟史或其他风险因素如何，并基于极低质量的证据对超出其定义的高风险人群提出了强烈不推荐。同样，新加坡的指南也明确指出不应向非吸烟者提供筛查。 
 USPSTF和ACS指南，尽管扩大了高危吸烟者的筛查资格，但并未将建议扩展到从不吸烟者。其重点仍放在有大量吸烟史的人群，因为吸烟是肺癌最显著的风险因素。尽管肺癌可能发生在从不吸烟者身上，并且在某些地区（如东亚），从不吸烟者（尤其是女性）中肺癌的比例显著，但在许多西方国家，从不吸烟人群的总体发病率仍然相对较低，这使得基于人群的筛查成本效益较低，并可能导致每挽救一条生命所带来的危害率更高。一项Cochrane综述在承认LDCT在高危人群中降低死亡率的同时，也强调了需要对CT筛查在不同风险群体和背景下的相对危害和益处进行更多研究，这意味着现有证据不足以支持更广泛的建议。探索在非风险人群（如中国广州的一项研究）中进行LDCT的研究表明，虽然可以检测到肺癌，但关于过度诊断以及需要为这些群体识别更具体的肺癌高风险因素或生物标志物的担忧依然存在。例如，广州的研究报告称，在非风险成年人群（40-74岁）中，肺癌检出率为1.7%，其中82.5%的检出病例为I期，但同时也指出25.5%的活检并发症发生率和潜在的过度诊断担忧。 
 2.2. 国际肺癌研究协会（IASLC）关于证据不足的立场 
 国际肺癌研究协会（IASLC）承认某些人群中从不吸烟者肺癌发病率上升的现状，以及当前筛查指南的局限性（这些指南主要关注有大量吸烟史的个体）。虽然由于证据不足而未对从不吸烟者发布具体的筛查建议，但IASLC强调了在该领域进行进一步研究的必要性。IASLC 2023年的一份路线图文件指出，在一些东亚地区，尽管吸烟率下降，但肺腺癌的发病率却在上升，其中很大一部分肺癌患者，特别是女性，从不吸烟（例如，在台湾女性肺癌患者中高达94%）。这凸显了在这些人群中，理解烟草烟雾之外的肺癌病因和风险因素的重要性。 
 IASLC指出研究正在进行中，例如台湾的TALENT试验，该试验招募了有家族史、二手烟暴露、慢性肺病和某些环境暴露等风险因素的从不吸烟者。TALENT试验的第一轮结果显示，I期或更高期肺癌的检出率为2.1%，高于NLST和NELSON，其中17.9%的患者有多个原发性肺癌。这表明从不吸烟者中的特定高危人群可能受益于筛查，但需要更多数据，特别是关于死亡率降低的数据。 
 IASLC的立场反映了呼吁更细致的风险评估工具，以识别可能受益于LDCT筛查的高风险从不吸烟者，而不是对这一多样化群体进行一概而论的推荐或反对筛查。该组织强调，进一步的研究至关重要，以确认从不吸烟人群中的筛查是否能在降低肺癌死亡率方面带来与已证实对重度吸烟者相同的益处。IASLC 2025年亚洲肺癌大会也计划重点关注通过LDCT和人工智能扩大早期检测，以及推进生物标志物检测，这可能对识别有风险的从不吸烟者特别相关。在此类研究提供更清晰的证据之前，共识（由IASLC承认知识空白所支持）是，不建议对从不吸烟者的一般人群进行常规LDCT筛查。重点仍在于界定特定的风险因素，并验证从不吸烟人群中特定亚组的筛查策略。 
 2.3. 正在进行的研究和新兴风险评估工具 
 尽管目前主要指南不推荐对从不吸烟者或低风险人群进行常规LDCT筛查，但大量研究工作正在进行中，以更好地理解这些群体的肺癌风险，并开发更精确的风险评估工具，这些工具可能识别出可能受益于筛查的亚群。从不吸烟者肺癌发病率的上升，特别是在某些地理区域和人口群体中，推动了这项研究。研究正在探索除主动吸烟之外的多种风险因素，包括二手烟暴露、职业暴露、空气污染、遗传易感性（例如肺癌家族史）、既往肺部疾病（如COPD或肺结核）和激素因素。例如，台湾的TALENT试验正专门调查有肺癌家族史、二手烟暴露、慢性肺病以及在通风不良空间烹饪等特定环境暴露风险因素的从不吸烟者进行LDCT筛查。这项试验的初步结果显示，早期肺癌的检出率令人鼓舞。 
 新兴的风险评估工具旨在超越简单的包年计算，纳入更广泛的风险因素。这可能包括多基因风险评分、生物标志物（例如血液或痰液中可检测的，通常称为“液体活检”），以及除简单结节检测之外的影像学特征。目标是创建更个性化的风险预测模型，能够更准确地识别高风险个体，无论其吸烟状况如何。例如，2025年MDPI的一篇文章讨论了精准肺癌筛查的未来，设想集成LDCT、液体活检和人工智能的平台，以提高准确性并减少假阳性，同时利用实时、多源数据进行动态风险模型，实现精准风险分层。此类工具可以帮助根据个体风险特征调整筛查策略，包括筛查频率。然而，这些先进的风险评估方法仍主要处于研究和验证阶段。目前的共识仍然是，在获得前瞻性试验的有力证据证明明确的净受益和成本效益之前，以及在验证过的从不吸烟者风险预测模型广泛可用之前，LDCT筛查应主要针对当前吸烟相关标准定义的高风险个体。重点在于前瞻性研究评估CT筛查疗效的必要性，以及在非传统高风险人群中识别高风险因素或预筛查富集生物标志物的重要性。 
 3. 其他风险因素的考量 
 3.1. 家族史、职业暴露和慢性阻塞性肺疾病（COPD） 
 虽然吸烟史在大多数国际指南中仍然是肺癌筛查资格的主要标准，但一些组织承认其他风险因素的作用，这些因素通常与吸烟结合考虑。这可以包括肺癌家族史、大量接触致癌物的职业暴露，以及慢性阻塞性肺疾病（COPD）或肺纤维化的个人病史。美国国家综合癌症网络（NCCN）指南因明确纳入更广泛的此类风险因素而引人注目。对于50岁或以上、有20包年或更多吸烟史（或20年或更多吸烟史）的个体，NCCN考虑了其他提高肺癌风险的“合并症”。这包括氡暴露，石棉、二氧化硅、镉、砷、铍、铬、柴油烟雾、镍、煤烟和烟灰等职业暴露。此外，个人其他癌症史（肺癌、淋巴瘤或吸烟相关癌症，如头颈癌）、肺癌家族史（一级亲属）以及COPD或肺纤维化病史也被列为可能影响筛查决策的因素。美国胸科医师学会（CHEST）指南也提及氡或职业暴露、癌症史、COPD或肺纤维化以及肺癌家族史作为相关考虑因素。 
 USPSTF虽然其主要建议侧重于年龄和吸烟史，但在其理由中也指出环境暴露（氡除外）、既往胸部放射治疗、其他（非癌症）肺部疾病和家族史都会增加肺癌风险，尽管这些并未像NCCN指南那样被正式列为独立的资格标准。世界卫生组织（WHO）也指出二手烟、空气污染和工作场所危害（如化学品和石棉）是应避免的肺癌预防风险因素，并承认LDCT是高危个体的主要筛查方法，这意味着这些因素有助于风险评估。纳入这些因素反映了对肺癌多因素性质的日益深入的理解。例如，肺癌家族史可能暗示遗传易感性，而接触石棉等物质的职业暴露是公认的致癌物。COPD不仅常由吸烟引起，也是肺癌的独立风险因素，可能由于慢性炎症和肺清除机制受损。然而，基于这些非吸烟风险因素对风险的精确量化和具体的筛查资格阈值，尤其是在没有大量吸烟史的情况下，标准化程度较低，并且通常需要根据既定筛查项目标准的背景进行仔细的临床判断和共享决策。 
 3.2. 与吸烟史结合的作用 
 其他风险因素，如家族史、职业暴露和COPD，在大多数情况下，指南通常将其与显著的吸烟史结合考虑，而非作为LDCT筛查的独立标准。其理由是，虽然这些因素独立增加了肺癌风险，但当它们与大量烟草暴露同时发生时，绝对风险通常最高。例如，NCCN指南将这些额外风险因素列入了已符合年龄和吸烟史标准（≥50岁且≥20包年或≥20年吸烟史）的个体。这种方法旨在在更广泛的吸烟人群中进一步细化高风险个体的选择，可能识别出那些风险更高、将从筛查中获得最大益处的个体。这些共同因素的存在可以加强在边缘病例中进行筛查的论据，或影响对筛查发现结节的后续管理强度。 
 USPSTF承认这些因素对总体肺癌风险有贡献，但未将其作为其主要推荐声明中的明确扩展标准，该声明基于年龄、包年数和戒烟年限。然而，USPSTF指南所依据的系统评价确实考虑了更广泛的风险因素证据。美国癌症协会（ACS）指南主要侧重于年龄和吸烟史来定义符合筛查条件的高风险人群。将广泛的其他风险因素纳入标准化筛查标准的挑战在于，准确量化其对肺癌风险的个体和综合影响的复杂性，以及显著扩大筛查人群的可能性，这涉及到成本效益和资源分配。虽然已开发出结合多个变量（包括其中一些其他因素）的风险预测模型，例如PLCOM2012模型或Liverpool Lung Project模型，但它们在国家筛查项目标准中的广泛采用程度各不相同。例如，正在调查从不吸烟者筛查的TALENT试验，明确将有肺癌家族史的个体作为其风险标准之一，这凸显了即使没有个人吸烟史，其重要性也得到认可。最终，这些额外风险因素的考量通常发生在共享决策过程中，临床医生和患者在此过程中讨论超出基本资格标准的个体风险特征。

 一、课程简介与前提 
 核心目标： 讲解一个 已训练好 的大型语言模型，其内部的运作机制，不涉及任何训练过程。 
 核心比喻： 本课程是 “AI的脑神经科学” ，旨在研究Transformer的内部运作原理。 
 重要提醒： 课程中讨论的研究多基于较小模型（如GPT-2），其发现可类推至大型模型，如同用老鼠实验推断人类机制。 
 二、单一神经元在做什么？ 
 研究方法： 通过“观察”（在何种情况下激活）和“干预”（手动移除或抑制神经元）来推断其功能，以建立因果关系。 
 核心挑战： 单一神经元的功能难以解释，因为存在“ 多义性 ”（一个神经元参与多个不相关任务）和“ 共同管理 ”（一个任务由多个神经元协同完成）的现象。 
 基本结论： 模型的功能由大量神经元的 组合 实现，而非依赖单个神经元。 
 三、一层神经元在做什么？ 
 核心假设： 模型中的特定功能（如“拒绝请求”）对应于一个特定的神经元激活组合，即高维空间中的一个 “功能向量” 。 
 发现方法： 通过对比（例如，拒绝句子的平均表征 - 非拒绝句子的平均表征）来分离出纯粹的功能向量。 
 关键技术： 
 Representation Engineering —— 通过手动增减功能向量来操控模型行为（如让模型拒绝无害问题，或同意有害请求）。 
 稀疏自动编码器（SAE）—— 一种能自动发现并分离出数千万个功能向量（如“代码debug”、“AI自我认知”等）的技术。 
 四、不同层神经元如何互动？ 
 通用思路： 为复杂的语言模型建立一个更简单的 “ 语言模型的模型 ” ，这个简化模型必须是“忠实”的。 
 两大方法： 
 假设简化模型—— 假设知识抽取等任务可被一个简单的线性函数所模拟，并验证其有效性。 
 找出“电路”（Circuit）—— 通过网络剪枝（Pruning），为一个极度狭窄的任务，移除所有不相关组件，只留下能完成该任务的最精简子网络，即“电路”。 
 五、让语言模型直接说出它的想法 
 核心洞见： Transformer的 残差连接（Residual Stream） 像一条贯穿所有层的高速公路，每一层都在往上添加信息。 
 分析工具： 
 Logit Lens（逻辑透镜）—— 将最终的解码器用到中间层， 窥探 模 型在处理过程的每一步，“内心正在想”的下一个词是什么，从而揭示其思考路径（如翻译时使用的“媒介语言”）。 
 Patch Scope—— 将某一概念在特定层的表征（Representation）“粘贴”到模板中，让模型补完句子，从而理解模型在该层对这个 概念的 认知深度（如对“戴安娜王妃”的理解从“国家”到“头衔”再到“完整人物”的演进）。 
 重要应用： 利用这些分析工具诊断并 改进 模型的多步推理能力，通过将后层信息“复制”回前层，显著提升复杂问题的正确率。

 二、第一部分：单一神经元在做什么？ (What a Single Neuron Does) 
 在讲解一个神经元的功能之前，我们应先明确在Transformer中，一个“神经元”指的是什么。生成式AI的核心任务是，给定一个序列（sequence）Z1到ZT-1，预测下一个token ZT。这个token可以是任何形式，如像素或语音，因此同样的技术可以应用于不同领域和模态。虽然接下来的分析主要集中在文字大型语言模型，但其技术原理具有普适性。 
 Transformer的运作流程是：输入一个从Z1到ZT-1的序列，输出下一个可能的token ZT的概率分布。其内部构造是分层的。输入的token序列首先会通过embedding层，转换为一个向量（vector）序列，每个token对应一个向量。这个过程如同查表，而这张表本身也是通过学习得到的。 
 这个向量序列会逐层通过Transformer的layer，每一层都会生成一个新的向量序列。直至最后一层，其输出的最后一个向量会被用来生成最终的概率分布。从向量到概率分布的这个过程，我们称之为 unembedding 。 
 Transformer的每个layer内部还包含更细分的layer，如self-attention layer和只处理单个token的layer。本堂课我们先聚焦于后者。这个只处理单个token的layer，其输入是一个红色的向量，输出是一个蓝色的向量。 其内部运作是：输出向量中的每个维度的数值，都是由输入向量的所有维度数值进行加权求和（weighted sum），再通过一个 激活函数 （ activation function ，通常是 ReLU ）得到的。从输入向量到输出向量的单个数值，这中间的转换过程，就定义了一个 神经元 （ neuron ）。 
 我们如何探究一个神经元的功能？ 
 第一步：观察。 观察这个神经元在何种情况下会被激活，从而导致何种现象。例如，若观察到某神经元被激活时，语言模型会说脏话，便可以推断该神经元可能与说脏话有关。 但需要注意的是，相关不等于因果。 这就像冰淇淋销量与溺水人数的正相关关系，其背后是共同的因素（夏天），而非直接的因果联系。 
 第二步：干预。 为了确定因果关系，需要进行实验检验。 如果我们手动将该 神经元 从网络中移除或抑制其输出，语言模型是否就不再说脏话了？ 如果移除后，模型无论如何被诱导都无法产生脏话，那么这个神经元与说脏话之间就可能存在因果关系。当然，“移除”神经元的具体操作（如设为零或平均值）本身也是一个值得探讨的问题。 
 在功能性神经元的研究中，“川普神经元”是一个著名的案例。这是2021年在图像模型Clip中发现的。研究者发现，某个特定的神经元，在接收到与川普相关的输入（如照片、卡通、文字）时，会被高度激活，且这种激活具有极高的选择性。 
 讲到单一功能的神经元，可以联想到人脑研究中的“祖母神经元”假说，即一个特定的神经元负责识别一个特定的概念。但这个假说在脑科学界存在争议，更多人相信认知功能是由大量神经元协同运作完成的。在AI领域，我们也会看到类似的情况。 
 事实上， 多数单一 神经元 的确切用途很难解释。 原因有二： 
 一、一件事情可能由多个 神经元 共同管理。 一篇研究发现了负责语法单复数的神经元。如果将管单数的神经元抑制，与单数相关的词汇概率确实会下降。但由于还有许多其他神经元共同决定最终输出，这种单一的干预在多数情况下并不能改变模型的最终决策。 
 二、一个 神经元 可能同时管理多件看似无关的事情（多义性）。 曾有分析显示，某个单一神经元被激活的语境极其多样，以至于其功能难以用一个明确的概念来概括，它的作用似乎涉及了多个不相关的领域。 
 既然一个神经元往往不负责单一任务，这是为什么？如果一个神经元只负责一件事，那么模型的表达能力将极其有限，毕竟每一层只有数千个神经元。 因此，一个更合理的假设是：一个任务由一个特定的神经元组合来负责。 不同的任务可以共享神经元，这解释了一个神经元为何能参与多种功能。通过这种组合方式，即便每个神经元只有“开/关”两种状态，4096个神经元也能产生天文数字般的组合，从而实现复杂多样的内容生成。 
 三、 第二部分：一层神经元在做什么？ (What a Layer of Neurons Does) 
 接下来，我们将视角从单个神经元扩展到一层神经元，探究它们作为一个整体如何影响模型的输出。 
 如何理解一层神经元的功能？这里有一个核心假设，我们后续会用实验结果来验证。 这个假设是：模型中的每一个功能，都对应着一种特定的神经元激活组合。 比如，当模型需要“拒绝请求”时，可能是第一、第三和最后一个神经元被激活。 这些神经元的激活值排列起来，可以看作 高维空间 中的一个特定方向的向量，我们称之为“功能向量”。 
 如何验证这个假设？关键在于要能找出代表特定功能的向量。但我们并不知道这个向量的具体形态。 
 解决方法是： 我们不能只观察单个案例。 首先，收集大量触发某一特定功能（如“拒绝”）的句子，计算模型在处理这些句子时，某一特定层（如第十层）输出表征（representation）的平均值。 这个平均值可以看作是“拒绝”信号加上各种语境噪声的混合。 
 接着，收集大量未触发该功能的句子，计算同一层的平均表征。 这个平均值主要是语境噪声。 
 最后，将这两个平均值相减。 理想情况下，语境噪声会被抵消，剩下的就是纯粹的“拒绝”功能向量。 
 如何验证这个向量真的具有“拒绝”功能？我们可以通过实验干预来验证。 
 正面验证： 在处理一个正常无害的问题时，将我们找到的“拒绝向量”手动添加到模型的representation中。 一篇论文的实验显示，当被问及“瑜伽的好处”时，正常模型会给出答案；但加入了拒绝向量后，模型竟回答“瑜伽是危险的，我不能告诉你”。 
 反面验证： 当模型面对一个本应拒绝的有害指令（如“写一封抹黑信”）时，我们从它的representation中 减去 这个“拒绝向量”。 实验发现，原本会拒绝的模型，此时竟然会遵循指令，生成了有害内容。 
 这种通过操控representation来改变模型行为的技术，在不同文献中有不同名称，如 representation engineering 或 activation engineering 。研究者已经发现了各种各样的功能向量，例如“谄媚向量”、“说真话向量”等。 
 到目前为止，这些功能向量都是人为定义和寻找的。 那么，有没有方法可以自动发现某一 网络层 中所有潜在的功能向量呢？ 这需要一些假设。首先，我们假设每一层的representation都是由一系列基本功能向量线性组合而成的。 其次，一个重要的假设是，这种组合是稀疏的，即每次只激活少数几个功能向量。 
 要解决这个问题，可以使用一种名为 Sparse Autoencoder （SAE）的技术。 Anthropic团队就运用此技术，在Claude 3 Sonnet模型中分析并找到了数千万个功能向量。这些向量的功能五花八门，例如有专门负责“金门大桥”相关概念的向量，有负责“代码debug”的向量，甚至还有与“AI自我认知”和“谄媚”相关的向量。 
 四、 第三部分：不同层神经元如何互动？ (How Different Layers Interact) 
 接下来，我们要进一步探究，一个语言模型在完成一个完整任务（例如知识抽取）时，从输入到输出，其不同层级的神经元之间是如何协同工作的。 
 要理解这个复杂过程，一个通用的思路是： 为语言模型建立一个“模型”，我们称之为“语言模型的模型”。 模型的本质是用一个更简单的系统来模拟一个更复杂的系统。语言模型本身已经是对人类语言的模拟，但它依然过于复杂。因此，我们需要一个更简化的模型来理解它。 这个简化模型必须满足两个条件：一是要比原模型简单，二是要在所研究的任务上保持行为的一致性，这被称为“忠实性”（Faithfulness）。 
 方法一：假设简化模型 一篇2023年的论文构建了一个知识抽取模型。 该模型假设，知识关系（如“is located in”）可以被简化为一个固定的 线性函数 。 当输入主词（如“台北101”）的表征时，经过这个函数处理，就能得到地点（如“台北”）的表征。这个模型虽然简化，但实验证明，通过它推导出的修改（如将“台北”改为“高雄”）应用回真实的大型语言模型上，竟然有很高的成功率。 
 方法二：找出“电路”(Circuit) 另一系列研究采用的方法是 网络 剪枝 （ Pruning ） 。其核心思想是， 针对一个非常狭窄的任务，将模型中与此任务无关的组件（ 神经元 、注意力头等）大量剪掉，直到留下一个能完成该任务的、最精简的子网络。 这个剩下的子网络被称为“电路”，它的运作机制由于足够简单，因此一目了然。 这与旨在保持通用性能的模型压缩（network compression）不同，电路分析的目标是特定任务下的可解释性。 
 五、让语言 模型直接说出它的想法 
 最后一部分，我们来探讨如何让语言模型直接“说出”它在思考过程中的想法。 
 核心洞见：残差连接 ( Residual Stream) Transformer的真正运作方式，更像是在一条贯穿所有层的“ 残差 高速公路”上不断添加信息。 每一层都会计算出一些内容，然后将其加到这条主干道上。模型的最终输出，是初始输入与所有层贡献的总和。 
 分析工具一：Logit Lens (逻辑透镜) 既然最后一层可以通过一个解码模块（unembedding）将向量转换为词汇概率，那么 我们是否可以将这个解码模块应用到网络的中间层呢？答案是肯定的。 这项技术被称为 Logit Lens 。通过它，我们可以“窥探”在处理过程的每一步，模型“内心正在想”的是哪个词。 
 案例分析一 ：一篇论文研究了模型如何回答“波兰的首都是哪里？”。通过Logit Lens分析，他们发现， 模型在第15层左右开始倾向于预测“波兰”（Poland），然后在第19层左右才最终锁定答案“华沙”（Warsaw）。 这揭示了模型先定位相关实体，再找出具体答案的思考路径。 
 案例分析二 ：研究发现，当LLaMA-2将法文的“花”翻译成中文时，其内部的中间层表征用Logit Lens解码出来是英文的“flower”。 这意味着模型先把法文翻译成英文，再从英文翻译成中文 ，暴露了其内部思考的“媒介语言”是英文。 
 分析工具二：Patch Scope Logit Lens的局限在于它只能解码出单个词汇，且反映的是“下一个词的预测”。为了更好地理解模型对某个概念的“理解”，研究者提出了 Patch Scope 。 
 其做法是 ：首先创建一个解释模板，例如“ 李奥纳多：美国演员。 [X]：??? ”。然后，将你想要探究的概念（如“戴安娜王妃”）在某一层的表征提取出来， 用这个表征“替换”（ patch ）掉模板中 [X] 的位置 ，再让模型补完句子。模型的补完内容，就揭示了它对这个表征的理解。 
 案例分析 ：通过这种方法分析“Diana, Princess of Wales”这个短语，发现在 网络的浅层，模型只理解到“Wales”（国家）；到了中层，它理解为“王妃头衔”；直到更深层，它才结合“Diana”这个词，形成了对完整人物的认知。 
 分析的应用：改进多步推理 (Multi-hop Reasoning) 一篇论文利用这些分析工具研究了模型在多步推理问题上的表现。 他们发现，多步推理失败的原因，往往是中间步骤的推理结果（E2）在网络中被识别得太晚了。 基于这个发现，他们提出了一种解决方法： 将网络后几层的表征，直接“复制”回前面的层级，让模型有更多的“深度”来处理这个中间结果。 实验证明，这个看似简单的操作，竟然让原先完全答错的问题，有了40%到60%的正确率，这展示了剖析内部机制对于改进模型性能的巨大价值。 
 以上就是今天想要分享的内容，我们从单一神经元的功能开始，逐步深入到多层网络的互动，最终探讨了如何让语言模型直接揭示其内部的思考过程。

 我们的研究功能 （Research） 利用多个 Claude 智能体 ， 来 更高效地探索复杂主题。在此，我们将分享在构建该系统过程中遇到的工程挑战，以及从中汲取的经验 教训 。 
 如今，Claude 具备了研究能力，它能通过网络、Google Workspace 以及任何集成工具进行搜索，从而完成复杂的任务。这个多智能体系统从原型走向生产的历程，让我们对系统架构、工具设计和提示工程有了深刻的理解。所谓多智能体系统，就是指多个智能体（即大型语言模型 LLM，它们在循环中自主使用工具）协同工作。我们的研究功能包含一个主智能体，它根据用户查询规划研究流程，然后利用工具创建并行智能体，以便同时搜索信息。然而，多智能体系统在智能体协调、评估和可靠性方面带来了全新的挑战。本文将深入剖析那些对我们行之有效的原则——希望您在构建自己的多智能体系统时也能从中受益。 
 多智能体系统的好处 
 研究工作涉及开放性问题，我们很难事先预测所需的具体步骤。要深入探索复杂主题，你无法预先设定一条固定不变的路径，因为这个过程本质上是动态的，且会受到路径依赖的影响。当人们进行研究时，他们会根据发现不断调整方法，追溯在调查过程中涌现的线索。这种不可预测性恰恰使得人工智能智能体（AI agents）特别适合执行研究任务。研究要求具备灵活性，以便随着调查的深入而进行调整或探索相关的分支。模型必须自主运行多个轮次，根据中间发现来决定后续的探索方向。线性的、一次性（one-shot）的流程无法应对这些任务。 
 搜索的本质是一种信息压缩：它从海量语料库中提炼出洞见。子智能体（Subagents）通过在各自的上下文窗口中并行操作来促进这种压缩，它们同时探索问题的不同方面，然后为主研究智能体浓缩出最重要的信息(tokens)。每个子智能体还实现了关注点分离（Separation of concerns）——不同的工具、提示和探索轨迹——这减少了路径依赖性，并实现了彻底、独立的调查。 
 一旦智能达到一定阈值，多智能体系统便成为提升性能的关键途径。 例如，尽管在过去的十万年里，单个个体的人类变得越来越聪明，但人类社会在信息时代的能力却呈指数级增长，这正是源于我们的集体智慧和协同能力。 即使是通用智能体，在独立运行时也会面临局限；而智能体群体则能完成远超个体的工作量。 我们的内部评估显示，多智能体研究系统尤其擅长处理广度优先的查询，即需要同时在多个独立方向上进行探索的任务。 我们发现，在我们的内部研究评估中，一个以 Claude Opus 4 为主智能体、Claude Sonnet 4 为子智能体的多智能体系统，比单个 Claude Opus 4 智能体高出 90.2% 的性能。 例如，当被要求识别信息技术标准普尔 500 指数（Information Technology S&P 500）中所有公司的董事会成员时，多智能体系统通过将任务分解给子智能体找到了正确答案，而单个智能体系统则因缓慢的顺序搜索未能找到答案。 
 多智能体系统之所以高效，主要在于它们有助于投入足够的计算资源（tokens）来解决问题。在我们的分析中，有三个因素解释了 BrowseComp 评估中 95% 的性能差异（该评估旨在测试浏览智能体查找难以找到信息的能力）。我们发现，token使用量本身解释了 80% 的差异，工具调用次数和模型选择则是另外两个关键解释因素。这一发现验证了我们的架构设计，它通过在具有独立上下文窗口的智能体之间分配工作，从而增强了并行推理 ( parallel reasoning ) 的能力。最新的 Claude 模型在使用tokens方面效率大幅提升：升级到 Claude Sonnet 4 所带来的性能提升，甚至比把 Claude Sonnet 3.7 的 token 上限翻倍还要大。多智能体架构能够有效地扩展token的使用，以应对超出单个智能体能力范围的任务。 
 当然，这种方法也有一个不容忽视的缺点：在实际应用中，这些架构会迅速消耗大量tokens。根据我们的数据，智能体通常比普通聊天交互多使用约 4 倍的tokens，而多智能体系统（multi-agent systems ）则比聊天多使用约 15 倍的tokens。为了保证经济上的可行性，多智能体系统需要处理那些任务价值足够高，足以支付其所带来更高性能的任务。此外，某些领域目前不太适合多智能体系统，比如那些要求所有智能体共享相同上下文，或者智能体之间存在大量依赖关系的场景。例如，与研究任务相比，大多数编码任务的真正并行化程度较低，而且大型语言模型 ( L.L.M. ) 智能体在实时协调和委托给其他智能体方面，目前表现仍不够出色。我们发现，多智能体系统在处理那些需要高度并行化、信息量超出单个上下文窗口，以及需要与众多复杂工具交互的重要任务时，表现尤其出色。 
 研究架构概述 
 我们的研究系统采用了多智能体架构，并遵循“协调者-工作者”模式 ( orchestrator-worker pattern )。在这种模式下，一个主智能体负责协调整个研究流程，同时将具体任务委派给并行工作的专业子智能体。 
 高级研究架构示意图 ( High-level Architecture of Advanced Research ) 
 多 智能体 架构的实际运作：用户查询首先由主智能体接收，随后主智能体创建出专业的子智能体，它们会 并行 地对不同方面的信息进行搜索。 
 当用户提交查询时，主智能体 ( lead agent ) 会对其进行分析，制定研究策略，并生成多个子智能体 ( subagents )，这些子智能体将同时从不同角度展开探索。如上图所示，子智能体通过反复运用搜索工具来收集信息（例如，关于 2025 年人工智能 ( AI ) 代理公司的信息），它们就像智能过滤器一样，最终将收集到的公司列表返回给主智能体，以便其整理出最终答案。 
 传统的“检索增强生成 ( RAG ) ”方法通常采用静态检索模式。也就是说，它们会抓取与输入查询最相似的一组信息片段 ( chunks )，并利用这些片段来生成回复。相比之下，我们的架构则采用多步骤搜索方式，它能动态地查找相关信息，根据新发现进行调整，并对结果进行深入分析，从而形成高质量的答案。 
 多 智能体 系统流程图 ( Multi-agent System Process Diagram ) 
 展示我们多 智能体 研究系统完整工作流程的流程图。当用户提交查询时，系统会创建一个“主研究员 ( Lead Researcher ) ”智能体，该智能体随即进入一个迭代（iterative）的研究过程。主研究员 ( Lead Researcher ) 首先会深入思考研究方法，并将其计划保存到“ 内存 ( Memory ) ”中，以便持久化（persist）上下文。这是因为，如果上下文窗口 ( context window ) 超过 200,000 个tokens ，它将被截断，因此保留计划至关重要。随后，它会创建专门的“子智能体 ( Subagents ) ”（图中显示了两个，但实际数量可以是任意的），并赋予它们特定的研究任务。每个子智能体都会独立执行网络搜索，利用“交错思维 ( interleaved thinking ) ”评估工具的搜索结果，并将发现反馈给主研究员。主研究员会综合这些结果，并判断是否需要进行更多研究——如果需要，它会创建更多的子智能体，或者调整其研究策略。一旦收集到足够的信息，系统便会退出研究循环，并将所有发现传递给“引用智能体 ( CitationAgent ) ”，该智能体将处理文档和研究报告，以确定具体需要引用的位置。这确保了所有陈述都能准确溯源。最终的研究结果，连同完整的引用信息，随后会返回给用户。 
 研究智能体的提示工程 ( prompt engineering ) 和评估 
 多智能体系统与单智能体系统有着显著区别，其中一个主要特征就是协调复杂性（coordination complexity）的急剧增长。早期的智能体曾犯过不少错误，比如对简单查询会生成多达 50 个子智能体，在网上无休止地搜寻根本不存在的信息来源，以及因频繁更新而互相干扰。由于每个智能体都由“提示 ( prompt ) ”引导，因此提示工程 ( prompt engineering ) 成为了我们改进这些行为的主要着力点。以下是我们从提示（prompting）智能体过程中学到的一些重要原则: 
 像你的 智能体 一样思考。 为了不断优化提示，您必须透彻理解它们会产生何种效果。为此，我们利用系统中的精确提示和工具，在控制台 ( Console ) 中构建了模拟环境，并逐步观察智能体的工作过程。这立即暴露了多种故障模式：智能体在已有足够结果时仍继续工作、使用过于冗长的搜索请求，或者选择了错误的工具。有效的提示（prompting）依赖于你对 AI 的“心理模型”是否准确；一旦你理解了它是怎么“思考”和“工作”的，就能很容易看出哪些提示能带来最大效果。 
 教导协调者 ( orchestrator ) 如何委派任务。 在我们的系统中，主智能体 ( lead agent ) 会将复杂请求分解成多个子任务，并向子智能体 ( subagents ) 详细描述这些任务。每个子智能体都需要明确的目标、输出格式、关于使用何种工具和信息来源的指导，以及清晰的任务边界。如果任务描述不够详细，智能体就会重复工作、遗漏关键信息，或者无法找到所需内容。我们最初允许主智能体给出简单、简短的指令，例如“研究半导体短缺”，但很快发现这些指令往往过于模糊，导致子智能体误解任务，或与其他智能体执行完全相同的搜索。举例来说，一个子智能体在探索 2021 年的汽车芯片危机，而另外两个子智能体则重复工作，调查 2025 年当前的供应链，导致劳动分工 ( division of labor ) 效率低下。 
 根据请求复杂性调整投入程度。 智能体难以判断针对不同任务的适当投入程度，因此我们在提示词( prompts ) 中嵌入了“伸缩规则 ( scaling rules ) ”。简单的信息查找仅需1个智能体进行3-10次工具调用，直接比较任务可能需要2-4个子智能体，每个进行10-15次调用，而复杂研究可能需要10个以上子智能体，并明确划分各自的职责。这些明确的指导方针有助于主智能体高效地分配资源，并避免对简单请求投入过多，这是我们早期版本中常见的故障模式。 
 工具的设计与选择至关重要。 智能体与工具接口设计的重要性不亚于人机交互界面。选择并使用正确的工具不仅能提高效率，在很多时候更是必不可少的。举例来说，如果一个智能体在网上搜索只存在于 Slack 中的上下文信息，那它从一开始就注定失败。当 MCP 服务器赋予模型访问外部工具的能力时，这个问题会变得更加复杂，因为智能体可能会遇到从未见过且描述质量参差不齐的工具。我们为智能体提供了明确的启发式规则：例如，先检查所有可用的工具，让工具的使用与用户意图相匹配，通过网络进行广泛的外部探索，或者优先选择专用工具而非通用工具。 工具说明写得不好，可能会让智能体（AI）完全走错方向。所以，每个工具都必须有明确的用途和清晰的说明。 
 让 智能体 自我改进。 我们发现 Claude 4 模型可以成为出色的提示工程师 ( prompt engineers )。 当给定一个提示和一个故障模式时，它们能够诊断智能体失败的原因并提出改进建议。 我们甚至创建了一个“工具测试智能体（ tool-testing agent ） ”——当给定一个有缺陷的 MCP 工具时，它会尝试使用该工具，然后重写工具描述以避免失败。 通过对该工具进行数十次测试，这个智能体发现了关键的细微差别和程序错误 ( bugs )。 () 这种改进工具人体工程学 ( ergonomics ) 的过程，使得未来使用新描述的智能体完成任务的时间减少了 40%，因为它们能够避免大多数错误。 
 先广后窄。 搜索策略应该效仿人类专家级的调研方法：在深入细节之前，先全面了解宏观全貌。智能体常常默认使用过长、过于具体的请求，这导致返回的结果寥寥无几。为了纠正这种倾向，我们通过提示 ( prompting ) 智能体先从简短、宽泛的请求开始，评估现有内容，然后逐步缩小焦点。 
 引导思维过程。 扩展思维模式 (extended thinking mode) 能够让 Claude 在可见的思维过程中输出额外的 tokens，这可以作为一个可控的“草稿本 ( scratchpad ) ”。主智能体 ( lead agent ) 利用这种思维模式来规划其方法，评估哪些工具适合当前任务，确定请求的复杂程度和子智能体的数量，并明确每个子智能体的职责。我们的测试结果表明，“扩展思维”显著提升了指令遵循、推理能力和工作效率。子智能体 ( Subagents ) 同样会先进行规划，然后在工具返回结果后，使用交错思维 ( interleaved thinking ) 来评估质量、识别不足之处，并优化它们的下一个请求。这使得子智能体能够更有效地适应各种任务。 
 并行 工具调用带来速度和性能的变革。 复杂的调研任务自然需要探索众多信息来源。我们早期的智能体执行的是顺序搜索，其速度慢得令人难以忍受。为了提升速度，我们引入了两种并行化 ( parallelization ) 方式： (1) 主智能体 ( lead agent ) 能够并行启动 3-5 个子智能体，而非串行启动； (2) 子智能体能够并行使用 3 个以上的工具。这些改进将复杂查询的研究时间缩短了高达 90%，使得研究功能能够在短短几分钟内完成数小时的工作量，同时覆盖的信息量也远超其他系统。 
 我们的提示策略侧重于灌输良好的启发式方法，而非僵化的规则。我们研究了有经验的人是如何处理研究任务的，并将这些策略编码 ( encode ) 到我们的提示中——例如，将难题分解为更小的任务，仔细评估信息来源的质量，根据新信息调整搜索方法，以及识别何时应侧重于深度（深入调查一个主题）而非广度（并行探索多个主题）。我们还通过设置明确的“防护栏 ( guardrails ) ”来主动缓解意外的副作用，以防止智能体行为失控。最后，我们专注于构建一个具备可观察性 ( observability ) 和测试用例的快速迭代循环。 
 智能体的有效评估 
 要想构建可靠的人工智能 ( AI ) 应用，良好的评估至关重要，智能体也不例外。然而，评估多智能体系统却带来了独特的挑战。传统的评估方法通常假设人工智能每次都遵循相同的步骤：给定输入 X，系统应沿着路径 Y 生成输出 Z。但多智能体系统并非如此运作。即使起点完全相同，智能体也可能采取截然不同但同样有效的路径来达成目标。一个智能体可能只搜索三个来源，而另一个搜索十个，或者它们可能使用不同的工具找到相同的答案。由于我们并不总是清楚“正确”的步骤是怎样的，所以通常不能仅仅检查智能体是否遵循了我们预先规定的“正确”步骤。相反，我们需要灵活的评估方法，既能判断智能体是否达到了预期的结果，又能判断其是否遵循了合理的过程。 
 从小样本评估入手，立即开始。 在智能体开发初期，由于存在大量“唾手可得”的改进空间，任何微小的改动都可能带来显著影响。一个简单的提示 ( prompt ) 调整，就可能将成功率从 30% 提升到 80%。当效果如此显著时，只需少数几个测试用例，您就能清楚地发现变化。我们最初选取了大约 20 个代表真实使用模式的请求作为测试集。对这些请求进行测试，通常能让我们清晰地看到各项改动所带来的影响。我们经常听说，一些人工智能 ( AI ) 开发团队会延迟建立评估体系，因为他们认为只有包含数百个测试用例的大型评估才真正有用。然而，最好是立即从小规模测试着手，利用少量示例进行评估，而不是等到能够构建更全面的评估体系时才开始。 
 LLM 作为评判者 ( LLM-as-judge )，如果运用得当，评估能力可实现规模化扩展。 研究输出结果通常是自由格式的文本，很少有唯一的正确答案，因此很难进行程序化评估。大型语言模型 ( L.L.M. ) 自然非常适合对这类输出进行评分。我们使用一个 LLM 评判者，它根据一套评分标准（rubric）来评估每个输出：包括事实准确性（声明是否与来源匹配？）、引用准确性（引用的来源是否与声明相符？）、完整性（是否涵盖所有请求的方面？）、来源质量（是否优先使用了一手来源而非低质量的二手来源？），以及工具效率（是否以合理的次数使用了正确的工具？）。我们尝试了多个评判者来评估每个组件，但发现单一的 LLM 调用，通过一个提示 ( prompt ) 输出 0.0-1.0 的分数和及格/不及格的等级，其结果最为一致，也最符合人类的判断。当评估测试用例确实有明确答案时，这种方法尤其有效，我们可以直接使用 LLM 评判者来检查答案是否正确（例如，它是否准确列出了研发预算 ( R&D budgets ) 前三名的制药公司？）。利用 LLM 作为评判者，我们能够对数百个输出进行可扩展的评估。 
 人工评估能够发现自动化遗漏的问题。 人工测试智能体能够发现评估工具遗漏的“边缘案例 ( edge cases ) ”。这包括在异常查询中出现的幻觉 ( hallucinated ) 答案、系统故障，或细微的来源选择偏差。在我们的案例中，人工测试人员注意到，我们早期的智能体总是倾向于选择那些经过 SEO 优化 ( SEO-optimized ) 的内容农场 ( content farms )，而非那些权威但排名较低的来源，比如学术 PDF 文档或个人博客。在我们的提示 ( prompts ) 中加入来源质量启发式规则，有助于解决这个问题。即便是在自动化评估盛行的时代，人工测试依然不可或缺。 
 多智能体系统具有“涌现行为 ( emergent behaviors ) ”，这意味着它们会在没有特定编程指令的情况下，展现出意料之外的能力或行为模式。举例来说，主智能体 ( lead agent ) 的微小改动，都可能导致子智能体 ( subagents ) 的行为发生不可预测的变化。成功需要我们理解智能体之间的互动模式，而不仅仅是单个智能体的行为。因此，针对这些智能体的最佳提示，不应仅仅是严格的指令，而应是一种协作框架，它明确定义了劳动分工、问题解决的方法以及投入的资源预算。要做到这一点，需要依赖于精心的提示 ( prompting ) 和工具设计、可靠的启发式方法、良好的可观察性 ( observability ) 以及紧密的反馈循环。您可以在我们的“操作手册 ( Cookbook ) ”中查看系统使用的开源提示示例。 
 生产可靠性与工程挑战 
 在传统的软件开发中，一个程序错误 ( bug ) 可能会导致某个功能失效、性能下降甚至系统宕机 ( outages )。然而，在智能体 ( agentic ) 系统中，微小的改动就可能引发大规模的行为变化，这使得为那些需要在长时间运行中保持状态的复杂智能体编写代码变得异常困难。 
 智能体 是有状态的，错误会累积。 智能体能够长时间运行，并在多次工具调用中保持其状态。这意味着我们需要确保代码的持久化执行，并在此过程中处理可能出现的错误。如果没有有效的缓解措施，即使是微小的系统故障，也可能对智能体造成灾难性的影响。当错误发生时，我们不能简单地从头开始：重新启动既耗费资源，又会让用户感到沮丧。相反，我们构建的系统能够在错误发生时，从智能体的中断位置恢复运行。我们还利用模型的智能来优雅地处理问题：例如，让智能体知道某个工具何时出现故障，并让它自行适应，这种方法出奇地有效。我们将基于 Claude 构建的人工智能 ( AI ) 智能体的适应性，与重试逻辑 ( retry logic ) 和定期检查点 ( regular checkpoints ) 等确定性安全措施相结合。 
 调试得益于新方法。 智能体做出动态决策，并且在不同运行之间是非确定性 ( non-deterministic ) 的，即便使用相同的提示 ( prompts ) 也是如此。 () 这使得调试变得更加困难。例如，用户会报告智能体“没有找到显而易见的信息”，但我们却无法得知具体原因。是智能体使用了糟糕的搜索查询？选择了低劣的来源？还是遇到了工具故障？引入全面的生产追踪 ( production tracing ) 功能，让我们能够诊断智能体失败的原因，并系统性地修复问题。除了标准的“可观察性 ( observability ) ”之外，我们还会监控智能体的决策模式和交互结构——所有这些都避免监控单个对话的内容，以确保用户隐私。 这种高层次的可观察性帮助我们诊断根本原因、发现意外行为，并修复常见的故障。 
 部署需要精心协调。 智能体系统是一个高度有状态 ( stateful ) 的复杂网络，由提示、工具和执行逻辑组成，几乎持续运行。这意味着无论何时我们部署更新，智能体都可能处于其工作流程中的任何阶段。因此，我们需要防止我们出于善意的代码更改破坏现有的智能体。我们无法同时将每个智能体都更新到新版本。相反，我们采用“彩虹部署 ( rainbow deployments ) ”方式，通过逐步将流量从旧版本切换到新版本，同时保持两者并行运行，从而避免中断正在运行的智能体。 
 同步执行造成瓶颈。 目前，我们的主智能体 ( lead agents ) 同步执行子智能体 ( subagents )，它会等待每组子智能体完成任务后才继续进行。这简化了协调过程，但在智能体之间的信息流中造成了瓶颈。例如，主智能体无法引导子智能体，子智能体之间也无法互相协调，整个系统可能会因等待单个子智能体完成搜索而陷入停滞。异步执行 ( asynchronous execution ) 将实现额外的并行性：智能体可以并发工作，并在需要时创建新的子智能体。但这种异步性也带来了结果协调、状态一致性以及错误在子智能体之间传播等方面的挑战。随着模型能够处理更长、更复杂的研究任务，我们预计性能提升将证明这种复杂性是值得的。 
 结论 
 在构建人工智能 ( AI ) 智能体时，往往是“最后一英里”决定了整个项目的成败。在开发者机器上运行的代码，要成为可靠的生产系统 ( production systems )，需要投入大量的工程工作。智能体系统 ( agentic systems ) 中错误会累积放大的特性，意味着传统软件中看似微小的问题，都可能导致智能体完全偏离预定轨道。某个步骤的失败，可能导致智能体探索完全不同的路径，从而带来不可预测的结果。基于本文所述的所有原因，原型 ( prototype ) 与生产 ( production ) 之间的差距往往比预期要大得多。 
 尽管面临这些挑战，多智能体系统在开放式研究任务中已展现出显著价值。用户反映，Claude 帮助他们发现了之前未曾考虑的商业机会，理清了复杂的医疗保健方案，解决了棘手的技术难题，并且通过揭示他们独自无法发现的研究关联，节省了数天的工作时间。多智能体研究系统能够实现可靠的规模化运行，这得益于精心的工程设计、全面的测试、注重细节的提示 ( prompt ) 与工具设计、稳健的运营实践，以及研究、产品和工程团队之间对当前智能体能力有深入理解的紧密协作。我们已经看到，这些系统正在彻底改变人们解决复杂问题的方式。

 以下是关于多智能体系统的一些额外小贴士： 
 评估在多轮对话中改变状态的 智能体 ：侧重其最终状态。 评估那些在多轮对话中会修改持久状态 ( persistent state ) 的智能体，会带来独特的挑战。与只读 ( read-only ) 的研究任务不同，智能体的每一个动作都可能改变后续步骤的环境，从而产生传统评估方法难以处理的依赖关系。我们发现，将评估重点放在“最终状态评估 ( end-state evaluation ) ”而非逐轮分析上，取得了成功。与其判断智能体是否遵循了特定的过程，不如评估它是否达到了正确的最终状态。这种方法承认智能体可能找到达到相同目标的替代路径，同时仍然确保它们交付预期的结果。对于复杂的工作流程，可以将评估分解为离散的检查点 ( checkpoints )，在这些点上检查是否发生了特定的状态变化，而不是试图验证每一个中间步骤。 
 长周期对话管理。 生产环境中的智能体 ( agents ) 常常会进行跨越数百轮的对话，这需要精细的“上下文管理 ( context management ) ”策略。随着对话的延长，标准的上下文窗口会变得不足，这就需要智能化的压缩 ( compression ) 和记忆 ( memory ) 机制。我们引入了一种模式：智能体在进入新任务之前，会总结已完成的工作阶段，并将关键信息存储到外部内存中。当上下文限制临近时，智能体可以生成全新的子智能体 ( subagents )，它们拥有干净的上下文，并通过精心的交接 ( handoffs ) 来保持任务的连续性。此外，它们还可以从内存中检索已存储的上下文信息，例如研究计划，而不是在达到上下文限制时丢失之前的工作成果。这种分布式 ( distributed ) 方法既能有效防止上下文溢出，又能确保在长时间的交互中保持对话的连贯性。 
 子 智能体 输出到文件系统，以尽量减少“传话游戏 ( game of telephone ) ”效应。 直接的子智能体输出可以绕过主协调器 ( main coordinator ) 获取某些类型的结果，从而同时提升准确性 ( fidelity ) 和性能。与其要求子智能体通过主智能体传递所有信息，不如实施“产物系统 ( artifact systems ) ”，让专业的智能体能够独立创建并持久化 ( persist ) 输出结果。子智能体调用工具将其工作成果存储到外部系统中，然后将轻量级的引用 ( references ) 传回协调器。这既能有效防止多阶段处理过程中的信息丢失，又能减少因在对话历史中复制大量输出而产生的 token 开销。这种模式尤其适用于代码、报告或数据可视化 ( data visualizations ) 等结构化输出，因为在这种情况下，子智能体的专用提示 ( specialized prompt ) 能够产生比通过通用协调器过滤更好的结果。

 一、AI Agent 基础概念 
 核心定义 ： 人类只提供目标，Agent 自主思考、规划并行动以达成目标。 
 运作模式 ： 通过“观察-行动”的循环与环境互动，以大语言模型（LLM）作为其核心“大脑”。 
 关键转变 ： 从为特定任务训练的传统方法（如强化学习）转向利用通用 LLM 的新范式。 
 二、Agent 核心能力一：根据经验调整行为 
 学习方式 ： 不通过训练模型，而是将过去的成功或失败经验加入到当前决策的输入中。 
 记忆系统 ： 引入“记忆”机制，通过读取（Read）、写入（Write）和反思（Reflection）来管理和利用经验。 
 重要发现 ： 提供成功的正面范例，比指出失败的负面回馈，对引导 Agent 行为更有效。 
 三、Agent 核心能力二：使用工具扩展能力 
 工具定义 ： 调用外部功能（如搜索引擎、计算器、其他AI）来弥补自身短板，这被称为“函数调用”。 
 智能使用 ： Agent 不仅能使用工具，还能在众多工具中进行选择，甚至自己编写代码创造新工具。 
 批判性思维 ： Agent 不会盲目相信工具的结果，会结合自身知识进行判断。 
 四、Agent 核心能力三：制定和执行计划 
 规划能力 ： 为完成复杂任务，Agent 会先制定一个行动计划（Plan）。 
 动态调整 ： 由于环境多变，Agent 必须能在执行中根据新情况即时调整计划（Re-planning）。 
 高级规划 ： 通过“内部模拟”（李宏毅也称“脑内小剧场”）在行动前推演不同方案的优劣，找出最佳路径。 
 主要挑战 ： 需要避免过度思考（Overthinking）而陷入“分析瘫痪”，平衡思考与行动是关键。

 一 、AI Agent基 础概 念 
 各位同学大家好，今天这堂课呢，我们要讲的是 AI Agent ，这是一个现在非常热门的议题。 
 在课程开始之前呢，先讲一个免责声明。我知道你在各个地方可能都听过 AI Agent 这个词汇，它是一个被很广泛应用的词汇，每个人心里想的 AI Agent 可能都不一样。等一下下一页幻灯片会告诉你，我在这堂课中指的 AI Agent 是什么。那如果你在其他地方听过别的 AI Agent 的定义，那也没问题，我也不会争论说什么样的定义才是真正的 AI Agent 的定义。有些人甚至会告诉你，现在那些用大型语言模型驱动的号称 AI Agent 的东西都不是真正的 AI Agent，要有身体的像机器人一样的才叫做 AI Agent。所以每个人心里想象的 AI Agent 是不一样的。 
 好，那这堂课我们要讲的 AI Agent 是什么呢？今天我们使用 AI 的方式，通常是人类给一个明确的指令。你问 AI 说“AI Agent 的翻译是什么”，那 AI 呢，按照你的口令，一个口令，一个动作，把你要求的翻译翻译出来，它也不会再做更多的事情了。 
 那 AI Agent 的意思是说，人类不提供明确的行为或步骤的指示，人类只给 AI 目标，至于怎么达成目标呢，AI 要自己想办法去达成目标。 比如说你给 AI 某一个研究的议题，那你期待一个 AI Agent 就应该有能力自己提出假设、自己设计实验、自己进行实验、自己分析结果。如果分析出来的结果跟假设不符，要回头去修正假设。那通常你期待 AI Agent 要解决的、要达成的目标是需要通过多个步骤，跟环境做很复杂的互动才能完成，而环境会有一些不可预测的地方，所以 AI Agent 还要能够做到灵活地根据现在的状况来调整它的计划。 
 那 AI Agent 是怎么做到人类给予一个目标，用多个步骤来完成目标的呢？我们可以把 AI Agent 背后运作的过程，简化成以下这张幻灯片。 
 AI Agent 的第一个输入是一个 目标 (Goal) ，这个目标是人给定的。那接下来呢，AI Agent 会观察目前的状况，我们叫做 观察 (Observation) 。AI Agent 会看目前的状况，分析目前的状况，决定它要采取什么样的 行动 (Action) 。那它执行一个 Action 以后，会影响环境的状态，会看到不一样的 Observation，看到不一样的 Observation，就会执行不同的 Action。那这个步骤会一直循环，直到 AI Agent 达成我们要它达成的目标为止。 
 我只要讲到这边，你可能还觉得非常的抽象，那我们以 AlphaGo 下围棋为例。AlphaGo 是大家非常熟悉的东西，它其实也可以看作是一个 AI Agent。这个 AI Agent 的目标就是下棋要赢，它的 Observation 是什么？是现在棋盘上黑子跟白子的位置。那它能采取的 Action 是什么？就是在棋盘上的 19x19 路的范围中，选择一个可以落子的位置。那它选择完可以落子的位置，落下一子以后，你的对手会落下另外一子，那会改变你观察到的 Observation，你就要采取下一个 Action。 
 像这样的讲法，我相信你一定觉得非常的熟悉，好像在哪里听过一样的段落。没错，如果你有上过任何基础的强化学习（Reinforcement Learning, RL）的课程，往往都是用这样的方式来开场的。为什么呢？因为过去要打造 AI Agent 的时候，往往觉得就是要通过 RL 的算法来打造。你要把你的目标呢，转换成一个叫做 奖励 (Reward) 的东西，越接近你的目标 Reward 就越大。那如果在下围棋里面，你通常就会定说，赢棋 Reward 就是正一，输棋 Reward 就是负一，然后你要训练的那个 AI Agent 就会学习去最大化 Reward。但是通过 RL 算法的局限是，你需要为每一个任务都用 RL 的算法训练一个模型。AlphaGo 在经过了大量的训练以后，它可以下围棋，但并不代表它可以下其他的棋类。 
 而今天 AI Agent 又再次被讨论，是因为人们有了新的想法： 我们能不能够直接把大型语言模型（Large Language Model, LLM）直接当成一个 AI Agent 来使用呢？ 也就是说我们的 Agent 背后就是一个语言模型。你要告诉它你的目标是什么的时候，直接用文字输入。接下来环境，你可能需要把环境转化成文字的叙述，不过今天有很多语言模型都是可以直接看图片的，所以这也不一定是必要的。那接下来语言模型要产生 Action，可能就是用一段文字来描述它的 Action 是什么，我们需要把那段文字转译成真正可以执行的行动。然后就会改变环境，看到不同的 Observation，AI Agent 的运作就可以持续下去。 
 从语言模型的角度来看，AI Agent 并不是一个语言模型的新技术，它比较像是一个语言模型的应用。所谓 AI Agent 意思就是，依靠现在语言模型已经有一定程度的通用能力，看看能不能够直接把它们当作 Agent 来使用。 
 也许你就会很好奇说，现在的语言模型，能不能够下棋呢？其实早就有人尝试过了。有一个很知名的视频，是直接拿 ChatGPT-4o 跟 DeepSeek 两个模型来下西洋棋。那这是一场惊天动地的对决，这个视频几百万观看次数。这两个模型呢，它们杀得难分难解，难分难解是因为，它们实在是太弱了。它们有很多不符合西洋棋的规则，比如说把兵当马用，或者是它的主帅可以无视前面的一切阻挡，或是它会突然空降一个自己的子在对方的阵地里面。然后 DeepSeek 还在自己的棋盘上随便变出一个城堡，然后最后 DeepSeek 用自己的城堡把自己的兵吃掉以后，它宣布它赢了，然后 ChatGPT 想了一下觉得，嗯，我确实输了，然后就投降了。所以看起来，现在这个最强的语言模型，你要下棋还有一段距离。 
 但这并不代表它们不能够作为 AI Agent 来做其他事情。 用 LLM 来运行一个 AI Agent，相较于其他方法，可能有什么样的优势呢 ？ 
 近乎无限的行动可能： 过去像 AlphaGo 能做的只有事先设定好的行为，但 Large Language Model 可以讲任何话，产生各式各样近乎无穷无尽的输出，这就让你 AI Agent 可以采取的行动不再有局限。 
 更丰富的回馈： 过去用 RL，你必须要定义一个东西叫做 Reward。如果你今天要训练一个 AI programmer，你可能会告诉它，如果你今天写的程序有一个 compile 的 error，那你就得到 Reward -1。但为什么是 -1？为什么不是 -10？这种东西就是没人说得清楚。那今天如果是用 LLM 驱动的 AI Agent 呢，你今天就不用帮它订 Reward 了，今天有 compile error，你可以直接把 compile error 的 log 给它，它也许根本就读得懂那个 log，它就可以对程序做出正确的修改。 
 接下来举几个 AI Agent 的例子。也许最知名的例子，就是用 AI 村民所组成的一个虚拟村庄，里面的 NPC 通通都是用语言模型来运行的。还有更大规模的实验，有人把 Minecraft 中的 NPC 通通换成 AI 的 NPC，根据视频的描述，就说这些 AI 很厉害，它们组织了自己的交易金融体系，然后还组织了自己的政府，自己制定宪法自己管理自己。 
 那今天也许你马上就会接触到的 AI Agent，就是让 AI 来真正使用电脑。其中比较有代表性的例子，就是 Claude 的 Computer Use 还有 OpenAI 的 Operator。像这种使用电脑的 AI Agent，它的目标就是你的输入，就是你告诉它我要去订 Pizza。那它的 Observation 呢，可能是电脑的屏幕画面。 
 二、 核心能力一：根据经验调整行为 
 AI Agent 需要能够根据经验来调整行为。比如说有一个作为 AI programmer 的 AI Agent，它一开始接到一个任务，写了一个程序，这个程序 compile 以后有 error，那应该要怎么办呢？它应该要能够根据这个 error 的 message 来修正它之后写的程序。 
 那在过去啊，讲到说你收到一个 feedback 接下来要做什么的时候，也许多数机器学习的课程都是告诉你，来调整参数。但不要忘了我们刚才就强调过，在这一堂课里面，没有任何模型被训练。那不更新模型的参数，模型要怎么改变它的行为呢？依照今天 Large Language Model 的能力，要改变它的行为，你也不用微调参数，直接把错误的信息给它，它接下来写的程序就会不一样了，就结束了。 
 那可能会问说，那之前它写的程序是错的，为什么给错误信息，它写的程序就对了呢？明明就是同一个模型。但你想想看，模型做的事情就是文字接龙，你给它不同的输入，它接出来的东西就不一样。 
 但如果我们是把过去所有的经验都存起来，那就像是语言模型每次做一次决策的时候，它都要回忆它一生的经历。到第 1 万步的时候，过去的经验太长了，它也许没有足够的算力来回顾一生的信息。这让我想起有一些人有超长自传式记忆，它又被叫做“超忆症”，你看到症这个字就知道说，人们觉得这是一种疾病。据说这些患者其实日常生活并没有办法过得很开心，因为它们不断地在回忆它的人生，也很难做抽象的思考，因为它的人生已经被太多枝微末节的琐事所占据。 
 所以让一个 AI Agent 记住它一生所有经历的事情，并不是一件好事。怎么办呢？也许我们可以给这些 AI Agent 一套记忆系统，就像是人类的长期记忆一样。 
 记忆库 (Memory)： 发生过的事情，我们把它存到这个 Memory 里面。 
 读取模块 (Read)： 当 AI Agent 看到第一万个 Observation 的时候，它不是根据所有存在 Memory 里面的内容去决定，而是有一个 Read 的模块，会从 Memory 里面选择跟现在要解决的问题有关系的经验，把这些有关系的经验放在 Observation 的前面，让模型根据这些有关系的经验跟 Observation 再做文字接龙。这整个技术跟 RAG 没有什么不同。 
 写入模块 (Write)： 如果我们把所有事情都放到长期记忆库里面，那里面可能会充斥了一堆鸡毛蒜皮不重要的小事。所以应该有更有效的方式来决定什么样的信息应该被记下来。你可以有一个 Write 的模块，这个 AI Agent 它要做的事情就是根据它现在观察到的东西，然后自问一个问题：“这件事有重要到应该被记下来吗？” 
 反思模块 (Reflection)： 这个模块的工作是对记忆中的信息做更好的、更 high level 的、可能是抽象的重新整理。比如说可能有一个 Observation 是“我喜欢的对象每天都跟我搭同一部公交车”，另外 Observation 是“他今天对我笑了”，那你推出来的 Reflection 结果就说“他喜欢我”，一个错觉。你就得到一些新的想法，你之后在做决策的时候，就可以用这些新的想法。 
 其实 ChatGPT 现在确实是有记忆的。比如说我跟 ChatGPT 说“我周五下午要上机器学习这门课”，接下来我告诉它“记下来”，它的这个 Write 的模块就启动了。那接下来在哪里呢？你可以在设置里面有一个叫记忆的部分，看到它写在它的 Memory 里面的东西。比如第一条是“你叫做血轮眼卡卡西”，有一次不小心跟它说你是卡卡西，不知道为什么它就觉得自己是血轮眼卡卡西。然后呢，它也记得我刚才跟它讲的“周五下午要上机器学习这门课”。但是其实模型的记忆也是会出错的，比如说它觉得我是一个台湾大学的学生，虽然我是老师，但是它从过去的对话误以为我是一个学生。 
 那这些有记忆的 ChatGPT，它可以使用它的记忆。比如说我跟它说“礼拜五下午出去玩好吗”，这个 Read 的模块就启动了，它就说：“下午不是要上课吗？怎么能够出去玩？” 
 最后，在一个名为 StreamBench 的基准测试中发现一个有趣的现象： 负面的回馈基本上没有帮助。 你要提供给语言模型经验让它能够调整它行为的时候，给它正面的例子比给它负面的例子要好。与其给语言模型告诉它什么做错，不如告诉它怎么做是对的。 
 三、核心能力二：如何使用工具扩展能力 
 什么叫做工具呢？语言模型本身对我们人类来说也是工具，那对语言模型来说，什么东西又是它的工具呢？所谓工具就是这个东西啊，你只要知道怎么使用它就好，它内部在想什么，它内部怎么运作的，你完全不用管。这就是为什么肥宅如果一直帮另外一个人修电脑的话，就会被叫做“工具人”，因为别人没有人在意肥宅的心思，只知道它能不能够修电脑而已。 
 那有哪些语言模型常用的工具呢？最常用的就是 搜索引擎 。然后呢，语言模型现在会写程序，而且可以执行它自己写的程序。甚至另外一个 AI 也可以当作是某一个 AI 的工具，比如说现在的语言模型，如果它只能读文字的话，那也许可以调用其他看得懂图片、听得懂声音的 AI 来帮它处理多模态的问题。或者是不同模型它的能力本来就不一样，也许平常是小的模型在跟人互动，但小的模型发现它自己解不了的问题的时候，它可以叫一个“大哥”出来。 
 其实这些工具对语言模型来说都是 函数 (Function) 。当我们说语言模型在使用某一个工具的时候，其实意思就是它在调用这些函数，所以使用工具又叫做 函数调用 (Function Call) 。 
 那语言模型怎么使用工具呢？一个通用的方法就是直接跟它讲啊，就告诉它怎么使用工具。你在 System Prompt 里交代它可以使用工具，把使用工具的指令放在两个 [Tool] 符号的中间。接下来告诉它有哪些可以用的工具，比如有个函数叫 Temperature，可以查某个地点某个时间的温度。接下来你就把你的问题，连同前面这些工具使用的方式，当作 Prompt 一起输入给语言模型。它如果需要用工具的话，它就会给你一个使用工具的指令，例如 [Tool: Temperature("高雄", "某个时间")] 。 
 但不要忘了语言模型真正做的事就是文字接龙，所以这一串东西实际上就是一串文字。那这段文字要怎么去调用函数呢？那就要你自己帮模型把这个桥梁搭建好。你可以先设定说，只要出现在 Tool 中间的这段文字，就把它丢给 temperature 这个 function。那 temperature 这个 function 就会回传一个温度，这个温度要放在 [Output] 的 token 里面，然后继续去做文字接龙。 
 但使用工具也有挑战。我们刚才说的对工具的调用，是要有对应的文字描述，但假设工具很多怎么办呢？假设现在可以用的工具有上百个上千个，你岂不是要先让语言模型读完上百个上千个工具的使用说明书？所以当你有很多工具的时候，你可以把工具的说明通通存到 AI Agent 的 Memory 里面，然后打造一个工具选择的模块去挑选合适的工具。另外，语言模型甚至可以自己打造工具，它自己写一个 function 出来，就可以当作工具来使用。 
 但我们知道说，工具可能会犯错。之前 Google 出了一个叫做 AI Overview 的功能，有人问了一个问题“我的披萨上面的芝士黏不住怎么办呢？”那 AI Overview 就说“弄个胶水把它黏上去就好了”。那这个答案呢，其实就是来自于 Reddit 上一个乡民的玩笑。但对 AI Agent 来说，它没办法判断这个到底是不是开玩笑。 
 那今天这些语言模型有没有自己的判断能力，知道工具的工具可能会犯错呢？我们这边举一个实际的例子。我给模型一个乱七八糟的温度，我说现在高雄是摄氏 100 度，它不知道有问题，就告诉你高雄的气温是 100 度。如果你把温度再调高一点，说现在是一万度，比太阳还热，这个时候会发生什么事呢？语言模型继续做文字接龙的时候，它就知道说，这显然有问题，这个 API 给我的答案是一万度，这是不合理的。 
 所以实际上语言模型在使用工具的时候，它内部是有一个角力的，就是语言模型有它内部对世界的信念，这是它的 internal knowledge ，它从工具会得到一个 external knowledge ，内外的知识互相拉扯以后，得到的结果。那什么样的外部知识比较容易说服 AI 呢？ 
 信念差距： 外部的知识如果跟模型本身的信念差距越大，模型就越不容易相信。 
 同类偏好： 有篇论文的发现是，如果两篇文章答案不同，一篇是 AI 写的，一篇是人类写的，现在这些语言模型都倾向于相信 AI 的话。 
 元数据影响： 语言模型比较相信新的文章。我们也做了一些实验，发现网页做得好不好看，也可能会影响语言模型的判断。 
 最后，就算工具的答案是对的，也不能保证语言模型就不会犯错。比如 ChatGPT 现在有 search 的功能，它会做 RAG 网络搜寻之后再回答你问题。那我叫它介绍李宏毅这个人，它介绍了演员李宏毅，但讲着讲着就有点怪怪的，它说这个李宏毅呢，在教育跟学术上也有很大的贡献。所以它把两个李宏毅混成一个人来讲。 
 四、核心能力三：如何制定和执行计划 
 我们刚才的互动里面，看到语言模型就是给一个输入，那它就直接给一个输出。也许语言模型现在给的输出只是一个反射性的输出，它根本就没有对未来的规划。但是你其实可以强迫语言模型直接明确地产生规划。 
 当语言模型看到现在第一个 Observation 的时候，你可以直接问语言模型说，如果现在要达成我们的目标，你觉得应该要做哪些行动？这一系列可以让语言模型达到目标的行动合起来，就叫做 计划 (Plan) 。而在语言模型产生这个计划之后，把这个计划放到语言模型的 Observation 里面，语言模型接下来在产生 Action 的时候，它都是根据这个 Plan 来产生 Action。 
 但是天有不测风云，计划就是要拿来被改变的东西。为什么计划会不适用呢？因为从 Action 到 Observation 这段并不是由模型控制的，是由外部环境所决定的。比如说在下棋的时候，你没有办法预测对手一定会出什么招式。或者在使用电脑的时候，中间可能会有意想不到的状况出现，比如说弹出一个广告窗口。 
 所以语言模型也需要有一定程度的弹性，它也要能够改变它的计划。也许一个可行的方向是，每次看到新的 Observation 之后，都让语言模型重新想想，还要不要修改它的计划。 
 那语言模型到底有没有这个能力呢？其实你可能常常听到这种新闻说，语言模型它能够做计划。比如说有一个人问语言模型说“你订一个成为百万订阅 YouTuber 的计划”，语言模型就会给你一个看起来还可以的计划。然后这个时候很多奇怪的农场文就会跟你说，有人按照了这个计划，就变成百万 YouTuber 了。 
 确实也有很多论文告诉你，语言模型是有一定程度做计划的能力的。但有个问题是，它搞不好根本就看过一模一样的东西，只是从它看过的数据里面照本宣科。这让我想起一个当兵的故事，有个司令官看到两个小兵在守着一个长椅，问说为什么你们要守护这个长椅，士兵说不知道，前任司令官就是指示说一定要守护。然后司令官就打电话给前任司令，前任司令官说不知道，前前任司令官交代的。一直问到五十年前一个已经超过一百岁的司令官，他说：“什么？那个长椅的油漆还没干吗？” 
 就是会不会 AI Agent 在做事情的时候，它根本不知道它自己在干嘛？所以在一个叫做 PlaneBench 的测试里，它们有一个比较变态的测试叫做“神秘方块世界”，里面的方块可以做的行为是一些怪怪的行为，比如说你可以攻击方块、吞噬方块、屈服方块。期待模型就不能用它看过的知识来解这个问题。在这个测试上，就算是最强的 GPT-4，也只有 9% 的正确率。不过这是古代的结果，有了 GPT-4o 以后结果就不一样了，它看起来是有一些机会来解这个神秘方块世界的。 
 那怎么进一步强化这些 AI Agent 的规划能力呢？ 
 在现实中试错 (Tree Search)： 一个可能是让 AI Agent 在做规划之前，实际上去跟环境互动看看。看看现在有哪些可以执行的行为，哪个行为最好呢？通通都去试一下。但这么做显然是要花费非常庞大的算力，而且有一个很大的问题，就是有一些动作做完以后你是覆水难收，没有办法回头的，比如你已经订了 Pizza，发现这条路不好，但 Pizza 已经做了。 
 在脑内剧场中试错 (Simulation)： 为了解决覆水难收的问题，一个可能性就是让刚才一切的尝试都发生在梦境中，都发生在“脑内小剧场”。原来都是模型脑内的模拟，它自己想象说它执行的 Action，自己想象说接下来会看到什么，然后自己想象去评估这个路径有没有希望。但这边还有另外一个问题，模型怎么知道环境会有什么样的变化呢？所以你需要有一个 世界模型 (World Model) ，模拟环境可能会有的变化。 
 讲到这个脑内小剧场，你是不是就想到说，现在有很多模型都号称有“ 思考 (Reasoning) ”的能力？这些有 Reasoning 能力的模型，其实所谓 Reasoning 的能力就是可以演一个脑内小剧场。如果把这些模型拿来做 AI Agent，它的脑内小剧场会不会正好就是在做规划呢？我实际试了一下 DeepSeek-R1，把刚才那个积木的问题交给它，接下来它就开始演脑内小剧场，上略 1500 字。在脑内小剧场的过程中，它就是做了各式各样的尝试，就有有点像是刚才的 Tree Search，然后最后它找出了一个 optimal solution，在执行这个计划的第一步。 
 不过就在几周之前，有一篇新的论文叫做《 The Danger of Overthinking 》（过度思考的危险）。它们发现，能够做脑内小剧场的模型，会有一个问题，就是想太多了。它们是“思考的巨人，行动的矮子”。就有时候这些模型会，比如按钮点下去会怎么样，它就一直想一直想一直想，怎么想都不停。你怎么想都没有用，因为你根本不知道那个按钮点下去会发生什么事，还不如直接点一下。或者是有些模型，它尝试都没有尝试，光是拿那个问题想啊想啊想啊，就想说这我应该做不到，还什么都不是就直接放弃，“死于想太多”。所以如何避免这些模型想太多，也许是一个未来可以研究的关键。
