```
 【早说】为智能体而设计：当软件的"用户"不再是人 
 问题不在模型本身，而在模型周围的一切。 
 LangChain 已经证明了这一点 —— 他们仅仅更换了包裹大语言模型的基础设施（同一个模型，同样的权重），就在 TerminalBench 2.0 上从三十名开外一跃升至第五名。另一个独立研究项目让大语言模型自行优化基础设施，达到了 76.4% 的通过率，超越了人工设计的系统。 
 这套基础设施如今有了正式名称：智能体框架（Agent Harness）。 
 何为智能体框架？ 
 这一术语于 2026 年初被正式确立，但概念早已有之。框架是包裹大语言模型的完整软件基础设施：编排循环、工具集成、记忆系统、上下文管理、状态持久化、错误处理和安全护栏。Anthropic 的 Claude Code 文档开门见山地指出：SDK 就是 "驱动 Claude Code 的智能体框架"。OpenAI 的 Codex 团队采用了同样的表述，明确将 "智能体" 和 "框架" 等同起来，以指代使大语言模型真正可用的非模型基础设施。 
 我很喜欢 LangChain 的 Vivek Trivedy 给出的经典公式："凡不属于模型的部分，皆为框架。"

 这里有一个容易混淆的关键区分。所谓的 "智能体" 是一种涌现行为：用户所交互的那个目标导向、能使用工具、能自我纠错的实体。而框架则是产生这种行为的机械装置。当有人说 "我构建了一个智能体"，实际上是说他构建了一套框架，并将其对准了一个模型。 
 Beren Millidge 在 2023 年的文章《作为自然语言计算机的脚手架式大语言模型》中给出了一个精妙的类比。裸露的大语言模型好比一颗没有内存、没有硬盘、没有输入输出的 CPU。上下文窗口充当内存（快速但有限），外部数据库充当磁盘存储（容量大但慢），工具集成相当于设备驱动程序，而框架就是操作系统。正如 Millidge 所写："我们重新发明了冯・诺依曼架构"—— 因为它是任何计算系统的天然抽象。

 工程的三个层次 
 围绕模型存在三个同心的工程层次： 
 提示工程 —— 精心设计模型接收的指令。 
 上下文工程 —— 管理模型看到什么、何时看到。 
 框架工程 —— 涵盖前两者，外加整个应用基础设施：工具编排、状态持久化、错误恢复、验证循环、安全执行和生命周期管理。 
 框架不是提示词的外壳，而是使自主智能体行为成为可能的完整系统。 
 生产级框架的十二大组件 
 综合 Anthropic、OpenAI、LangChain 以及更广泛的从业者社区的实践经验，一套生产级智能体框架包含十二个独立组件。让我们逐一拆解。 
 1. 编排循环 
 这是整个系统的心跳机制。它实现了 "思考 - 行动 - 观察"（TAO）循环，也称为 ReAct 循环。其运行逻辑为：组装提示词、调用大语言模型、解析输出、执行工具调用、将结果回传，如此反复直至任务完成。 
 从机械层面看，它往往不过是一个 while 循环。真正的复杂性在于循环所管理的一切，而非循环本身。Anthropic 将其运行时描述为一个 "笨循环"—— 所有智能都驻留在模型中，框架只负责管理对话轮次。

 2. 工具 
 工具是智能体的双手。它们以模式定义（名称、描述、参数类型）的形式注入大语言模型的上下文，让模型知晓有哪些能力可用。工具层负责注册、模式验证、参数提取、沙盒化执行、结果捕获，以及将结果格式化为模型可读的观察信息。 
 Claude Code 提供六大类工具：文件操作、搜索、执行、网络访问、代码智能和子智能体派生。OpenAI 的 Agents SDK 支持函数工具（通过 @function_tool 装饰器）、托管工具（WebSearch、CodeInterpreter、FileSearch）以及 MCP 服务器工具。 
 3. 记忆 
 记忆在多个时间尺度上运作。短期记忆是单次会话内的对话历史。长期记忆则跨会话持久存在：Anthropic 使用 CLAUDE.md 项目文件和自动生成的 MEMORY.md 文件；LangGraph 使用按命名空间组织的 JSON 存储；OpenAI 支持基于 SQLite 或 Redis 的会话持久化。 
 Claude Code 实现了三层层级结构：轻量级索引（每条约 150 字符，始终加载）、按需拉取的详细主题文件，以及仅通过搜索访问的原始对话记录。一个关键设计原则是：智能体将自身记忆视为 "提示线索"，在行动前务必与实际状态进行核验。

 4. 上下文管理 
 这是许多智能体悄然失效的环节。核心问题在于上下文腐化：当关键内容落在窗口中段位置时，模型性能下降超过 30%（Chroma 研究成果，与斯坦福 "迷失在中间" 的发现相互印证）。即便是百万 token 级别的窗口，随着上下文增长，指令遵循能力也会退化。 
 生产环境的应对策略包括： 
 压缩 ：在接近上下文上限时对对话历史进行摘要（Claude Code 保留架构决策和未解决的缺陷，同时丢弃冗余的工具输出） 
 观察遮蔽 ：JetBrains 的 Junie 隐藏旧的工具输出，但保留工具调用本身可见 
 即时检索 ：维护轻量级标识符，动态按需加载数据（Claude Code 使用 grep、glob、head、tail 而非加载完整文件） 
 子智能体委派 ：每个子智能体进行广泛探索，但仅返回一千到两千 token 的精炼摘要 
 Anthropic 的上下文工程指南一语道破目标：找到最小的高信号 token 集合，使期望结果的可能性最大化。 
 5. 提示词构建 
 这汇集了模型在每一步实际看到的内容。其结构是层级化的：系统提示词、工具定义、记忆文件、对话历史，以及当前用户消息。 
 OpenAI 的 Codex 采用严格的优先级栈：服务器控制的系统消息（最高优先级）、工具定义、开发者指令、用户指令（级联的 AGENTS.md 文件，32 KiB 上限），最后是对话历史。 
 6. 输出解析 
 现代框架依赖原生工具调用机制 —— 模型返回结构化的 tool_calls 对象，而非需要额外解析的自由文本。框架的检查逻辑很简单：有工具调用？执行后继续循环。如果没有工具调用？那就是最终答案。 
 对于结构化输出，OpenAI 和 LangChain 都支持通过 Pydantic 模型实现模式约束的响应。传统方法如 RetryWithErrorOutputParser（将原始提示词、失败的补全结果和解析错误一并反馈给模型）仍可用于处理边缘情况。 
 7. 状态管理 
 LangGraph 将状态建模为流经图节点的类型化字典，通过归约器合并更新。检查点在超步边界处创建，支持中断后恢复和时间旅行式调试。OpenAI 提供四种互斥策略：应用记忆、SDK 会话、服务端 Conversations API，或轻量级的 previous_response_id 链式引用。Claude Code 则另辟蹊径：以 git 提交作为检查点，以进度文件作为结构化草稿板。

 8. 错误处理 
 为什么这一点至关重要？一个十步流程即使每步成功率高达 99%，端到端成功率也仅有约 90.4%。错误的累积效应极为迅猛。 
 LangGraph 区分四类错误：瞬态错误（指数退避重试）、大语言模型可恢复错误（将错误作为 ToolMessage 返回，让模型自行调整）、用户可修复错误（中断并请求人工输入）、以及意外错误（上抛以便调试）。Anthropic 在工具处理器内部捕获失败，将其作为错误结果返回以维持循环运转。Stripe 的生产级框架将重试次数上限设为两次。 
 9. 护栏与安全 
 OpenAI 的 SDK 实现了三级防护：输入护栏（在首个智能体上运行）、输出护栏（在最终输出上运行）、工具护栏（在每次工具调用时运行）。"绊线" 机制一旦触发，立即终止智能体运行。 
 Anthropic 在架构层面将权限执行与模型推理相分离。模型决定尝试什么；工具系统决定什么被允许。Claude Code 对约 40 项独立的工具能力分别设门，分三个阶段执行：项目加载时建立信任、每次工具调用前进行权限检查、高风险操作需用户显式确认。 
 10. 验证循环 
 这是区分玩具演示与生产级智能体的分水岭。Anthropic 推荐三种方式：基于规则的反馈（测试、代码检查器、类型检查器）、视觉反馈（通过 Playwright 截屏用于界面任务）、以及大语言模型充当评判者（由独立的子智能体评估输出）。 
 Claude Code 的创造者 Boris Cherny 指出，赋予模型一种验证自身工作的途径，可将质量提升两到三倍。 
 11. 子智能体编排 
 Claude Code 支持三种执行模式：Fork（父上下文的字节级完整副本）、Teammate（独立终端面板，通过基于文件的信箱通信）、以及 Worktree（拥有独立的 git 工作树，每个智能体一个隔离分支）。OpenAI 的 SDK 支持 "智能体即工具"（专家处理有界子任务）和 "交接"（专家接管全部控制权）。LangGraph 则将子智能体实现为嵌套的状态图。

 循环运转：分步详解 
 了解了各组件之后，让我们追踪它们如何在单次循环中协同运作。 
 第一步（提示词组装） ：框架构建完整输入 —— 系统提示词 + 工具模式 + 记忆文件 + 对话历史 + 当前用户消息。重要上下文被放置在提示词的首尾两端（基于 "迷失在中间" 的研究发现）。 
 第二步（大语言模型推理） ：组装好的提示词发送至模型 API。模型生成输出 token：文本、工具调用请求，或两者兼有。 
 第三步（输出分类） ：若模型生成的是不含工具调用的文本，循环结束。若请求了工具调用，进入执行阶段。若请求了交接，更新当前智能体并重启。 
 第四步（工具执行） ：对于每个工具调用，框架验证参数、检查权限、在沙盒环境中执行，并捕获结果。只读操作可并发执行；有副作用的操作串行执行。 
 第五步（结果封装） ：工具结果被格式化为大语言模型可读的消息。错误被捕获并作为错误结果返回，以便模型自我纠正。 
 第六步（上下文更新） ：结果追加至对话历史。若接近上下文窗口上限，框架触发压缩机制。 
 第七步（循环） ：返回第一步，重复直至终止。 
 终止条件是多层叠加的：模型生成了不含工具调用的响应、超出最大轮次限制、token 预算耗尽、护栏绊线触发、用户中断、或返回安全拒绝。一个简单问题可能只需一到两轮，而一项复杂的重构任务可能跨越多轮串联数十次工具调用。 
 对于跨越多个上下文窗口的长时间运行任务，Anthropic 开发了一种两阶段的 "Ralph 循环" 模式：初始化智能体搭建环境（初始化脚本、进度文件、功能列表、初始 git 提交），随后编码智能体在每个后续会话中读取 git 日志和进度文件以定位自身，选取优先级最高的未完成功能，进行开发，提交代码，并写入摘要。文件系统提供了跨上下文窗口的连续性。

 各大框架如何实现这一模式 
 Anthropic 的 Claude Agent SDK 通过单一的 query() 函数暴露框架接口，该函数创建智能体循环并返回一个异步迭代器，以流式方式输出消息。运行时就是一个 "笨循环"，所有智能都驻留在模型中。Claude Code 采用 "采集 - 行动 - 验证" 循环：采集上下文（搜索文件、阅读代码）、执行行动（编辑文件、运行命令）、验证结果（运行测试、检查输出），如此往复。 
 OpenAI 的 Agents SDK 通过 Runner 类实现框架，提供三种模式：异步、同步和流式。该 SDK 奉行 "代码优先" 理念：工作流逻辑以原生 Python 表达，而非图领域特定语言。Codex 框架在此基础上扩展为三层架构：Codex Core（智能体代码 + 运行时）、App Server（双向 JSON-RPC API）、以及客户端界面（CLI、VS Code、Web 应用）。所有界面共享同一套框架，这就是为什么 "Codex 模型在 Codex 界面上的表现优于通用聊天窗口"。 
 LangGraph 将框架建模为显式的状态图。两个节点（llm_call 和 tool_node）通过条件边相连：若存在工具调用，路由至 tool_node；若不存在，路由至 END。LangGraph 脱胎于 LangChain 的 AgentExecutor—— 后者在 v0.2 中被弃用，原因是难以扩展且缺乏多智能体支持。LangChain 的 Deep Agents 明确使用了 "智能体框架" 一词：内置工具、规划能力（write_todos 工具）、用于上下文管理的文件系统、子智能体派生，以及持久化记忆。 
 CrewAI 实现了基于角色的多智能体架构：Agent（围绕大语言模型的框架，由角色、目标、背景故事和工具定义）、Task（工作单元）、Crew（智能体集合）。CrewAI 的 Flows 层增添了 "在关键处注入智能的确定性骨架"，管理路由和验证，而 Crew 则负责自主协作。 
 AutoGen（正在演进为微软智能体框架）开创了对话驱动的编排模式。其三层架构（Core、AgentChat、Extensions）支持五种编排模式：顺序执行、并发执行（扇出 / 扇入）、群聊、交接，以及 Magentic 模式（管理者智能体维护动态任务台账以协调各专家）。 
 脚手架隐喻 
 脚手架这个隐喻并非修辞装饰，而是精确类比。建筑脚手架是一种临时基础设施，使工人能够建造他们本无法触及的结构。它本身不进行施工，但没有它，工人就无法到达高层。 
 关键洞察在于：建筑竣工后，脚手架就会拆除。 随着模型能力提升，框架的复杂度理应递减。Manus 在六个月内重建了五次，每次重写都在削减复杂度 —— 复杂的工具定义被替换为通用的 shell 执行，"管理智能体" 被简化为朴素的结构化交接。

 这指向了一个协同演化原则：如今模型的后训练已将特定框架纳入循环。Claude Code 的模型学会了使用与之共同训练的那套特定框架。正因这种紧密耦合，改变工具实现可能导致性能下降。 
 框架设计的 "未来验证测试" 是：如果在不增加框架复杂度的前提下，性能能随更强大的模型同步提升，那么这个设计就是合理的。 
 定义每套框架的七个关键抉择 
 每位框架架构师都面临七个选择： 
 1、单智能体还是多智能体。 
 Anthropic 和 OpenAI 都明确表示：先将单智能体的能力发挥到极致。多智能体系统带来额外开销（路由需要更多 LLM 调用，交接过程中上下文会丢失）。只有当工具过载超过约十个重叠工具，或存在明确分离的任务领域时，才进行拆分。 
 2、ReAct 还是先规划后执行。 
 ReAct 在每一步都交织推理与行动（灵活但单步成本更高）。先规划后执行则将规划与执行分离。LLMCompiler 报告称相比顺序执行的 ReAct，速度提升了 3.6 倍。 
 3、上下文窗口管理策略。 
 五种生产级方案：基于时间的清除、对话摘要、观察遮蔽、结构化笔记，以及子智能体委派。ACON 研究表明，通过优先保留推理轨迹而非原始工具输出，可在保持 95% 以上准确率的同时减少 26% 至 54% 的 token 消耗。 
 4、验证循环设计。 
 计算性验证（测试、代码检查器）提供确定性的真值基准。推理性验证（大语言模型充当评判者）能捕获语义层面的问题，但增加延迟。Martin Fowler 的 Thoughtworks 团队将此框定为 "引导"（前馈，行动前纠偏）与 "传感器"（反馈，行动后观测）的对立。 
 5、权限与安全架构。 
 宽松模式（快速但有风险，自动批准大多数操作）与严格模式（安全但缓慢，每个操作都需审批）。选择取决于部署场景。 
 6、工具范围策略。 
 工具越多，性能往往越差。Vercel 从 v0 中移除了 80% 的工具，反而获得了更好的效果。Claude Code 通过懒加载实现了 95% 的上下文缩减。原则是：仅暴露当前步骤所需的最小工具集。 
 7、框架厚度。 
 多少逻辑放在框架中，多少留给模型。Anthropic 押注于薄框架和模型进步。基于图的框架押注于显式控制。Anthropic 会定期从 Claude Code 的框架中删除规划步骤，因为新版模型已将这些能力内化。 
 框架即产品 
 两个使用完全相同模型的产品，仅凭框架设计的差异就能产生天壤之别的性能表现。TerminalBench 的证据不言自明：仅仅更换框架就让智能体的排名跃升了二十位以上。 
 框架既不是一个已经解决的问题，也不是一个可以随意替换的通用层。真正艰深的工程就在这里：将上下文作为稀缺资源加以管理，设计能在错误累积之前将其截获的验证循环，构建提供连续性而不产生幻觉的记忆系统，以及做出架构上的取舍 —— 多少脚手架该自己搭建，多少该留给模型去承担。 
 这个领域正朝着更薄的框架方向演进，因为模型在不断进步。但框架本身不会消亡。即便是最强大的模型，也需要有什么东西来管理它的上下文窗口、执行它的工具调用、持久化它的状态、验证它的工作成果。 
 下次你的智能体出了问题，别怪模型。去看看框架。
```

```
 我想聊聊一件正在悄然改变那些认真对待 AI 工具的开发者们的构建方式的事情。 
 这并非一夜之间发生的。没有哪篇博客文章或哪场技术演讲一声令下就引爆了它。它是渐进式发生的，就像软件领域里大多数好想法的诞生过程一样。开发者们一次又一次撞上同样的问题，摸索出更好的解决办法，而这些更好的办法最终有了自己的名字。 
 这些名字是：提示工程、上下文工程和框架工程。 
 如果你在过去几年里用 AI 构建过任何东西，你很可能三者都做过，只是自己没意识到而已。这篇文章要做的，就是让这一演进过程变得清晰可见、有章可循，让你能更从容地决定把精力花在哪里。 
 "框架工程" 这个名字，在 2026 年初短短六周之内，从一个小众实践者的习惯，变成了软件开发圈里最热门的概念。Mitchell Hashimoto 在 2026 年 2 月 5 日写了相关文章，OpenAI 紧随其后于 2 月 11 日跟进，Martin Fowler 不久后也发表了一篇分析。这个术语就此流行起来。 
 但要理解框架工程为何重要，你得先看看我们是怎么一路走到这里的。故事要从两年前说起 —— 从提示词开始。 
 【早阅】用 Claude 编程：提示词入门 
 提示工程 
 它是什么 
 当 AI 语言模型变得触手可及时，开发者们最先注意到的一件事是：你提问的方式，对答案的质量有着巨大的影响。 
 如果你问 "修复这段代码"，你会得到一种答案。 
 询问 "修复这段代码，解释哪里出了问题，并确保修复方案兼容 Python 3.10"，你会得到一个有用得多的答案。 
 这个观察结果在成千上万的开发者中被重复了成千上万次，最终凝结成了一门学科。人们开始称之为 "提示工程"，并分享各自的心得。 
 措辞为何如此重要 
 语言模型读取你的提示词，和人类阅读一句话的方式完全不同。它做的是 —— 基于训练中学到的一切，预测你这段文字最有用的后续内容是什么。当你加入更多细节、更多约束、更清晰的目标时，你实际上是在收窄它可能给出的答案空间。这通常意味着，你会得到更接近你真正想要的东西。 
 用一种简单的方式来理解： 
 提示工程 
 写一条提示词、阅读回答、调整提示词 —— 这个循环基本就是提示工程的全部工作。概念简单，但知道如何调整，里面大有学问。 
 【第3572期】JSON 提示词：打造完美 AI 输出的终极指南 
 主要技巧 
 零样本提示（Zero-shot） 
 意味着你直接提问，不给任何示例。"把这句话翻译成法语。" 干净直接。对于模型在训练中大量见过的任务，效果很好。 
 少样本提示（Few-shot） 
 意味着在正式提问之前，你先给模型两三个示例。如果你希望模型以特定格式输出结果，直接展示输出应该长什么样，远比用文字描述可靠得多。就像带新同事时，直接给他看 "做完是什么样"，比写两页规格说明书管用得多。 
 少样本提示结构 
 思维链提示（Chain-of-thought） 
 意味着你让模型逐步推理问题，而不是直奔答案。只需加上一句 "在给出最终答案前，请一步步想清楚"，就能显著提升模型在数学、逻辑和推理任务上的准确率。当模型展示自己的推理过程时，它更容易发现自己的错误。 
 思维树提示（Tree-of-thought） 则更进一步。它不是沿着一条推理链走到底，而是让模型探索多种可能的路径，逐一评估，然后选出最优解。速度更慢，消耗的 token 更多，但对于真正棘手的问题，往往能给出更好的结果。

 提示工程力所不及之处 
 提示工程在处理一次性任务时表现出色。写一条好提示词，得到一个好答案，然后继续做别的事。 
 问题出在于，当你试图构建可复用的东西，或者把 AI 工具嵌入日常工作流的时候。每次新对话都是从零开始的。模型不记得你上次说了什么。如果有些规则需要模型遵守，你每次都得重新交代。如果你的问题依赖于变化中的信息 —— 比如代码库的当前状态或最新的日志输出 —— 你得自己手动粘贴进去。 
 提示工程也解决不了团队一致性的问题。如果六个开发者在使用同一个 AI 工具，他们很可能写着六个不同版本的提示词，得到六种参差不齐的输出质量。 
 这些问题，正是上下文工程应运而生要解决的。 
 【早阅】复杂代码库中的AI提效秘籍：如何通过上下文工程避开大模型的“愚蠢区” 
 上下文工程 
 它是什么 
 上下文工程关注的是模型能获得哪些信息，而不仅仅是你如何措辞提问。 
 模型只能基于你提供的内容来工作。它不知道你私有代码库里有什么，不知道你的团队上周会议上做了什么决定，也不知道今天早上你的日志里出现了什么错误。如果这些信息中有任何一条与当前任务相关，你就得自己把它放进对话里。 
 上下文工程，就是把这件事做得深思熟虑、有条有理的实践。 
 上下文如何构建 
 给模型提供所需上下文，有几种方式。 
 系统提示词 是位于对话之上的指令，在每一条消息中始终生效。你可以在这里放置诸如 "你是一位专注于 Python 的代码审查员"、"始终用通俗语言回答，绝不使用技术术语" 或 "在未征得确认之前，绝不建议删除数据" 之类的规则。把它们想象成常备命令。 
 对话历史 是当前会话中所有已说内容的记录。当你在新请求中包含过往消息时，模型就能回溯之前的决定，保持前后一致。没有这些，模型对之前讨论过什么一无所知。 
 检索文档 是一种技术 —— 你从知识库中搜索与当前问题相关的信息，并将其粘贴到提示词中。这通常被称为 RAG，即检索增强生成（Retrieval Augmented Generation）。其核心思想是：与其让模型凭借通用训练知识去猜测，不如让它阅读你从特定来源拉取的真实、最新的信息。

 工具调用 是让模型在对话过程中主动获取信息或执行操作的一种方式。你不必事先准备好所有上下文，模型可以自己索取它需要的东西。它可能调用一个天气 API、查询数据库或读取文件。这种方式更加动态，对于你无法预知确切需要哪些信息的任务尤为适用。

 模型上下文协议 
 MCP 正是在这里登场的。Anthropic 将 MCP 作为一个开放标准发布，用于连接 AI 工具与外部数据源及服务。在 MCP 出现之前，每个 AI 工具都必须为每个数据源单独构建定制连接器。有了 MCP 之后，你只需构建一次连接器，任何支持 MCP 的工具都能直接使用。 
 这就像 USB 改变了硬件世界一样。USB 之前，每种设备都有自己的接口。USB 之后，一个标准接口几乎通吃一切。MCP 试图为 AI 的数据连接做同样的事。

 上下文工程力所不及之处 
 上下文工程是一大进步。拥有良好上下文的模型比仅靠精心措辞提示词的模型更稳定、更准确、更有用。 
 但一个拥有良好上下文的模型，终究只是一个模型。它无法判断自己即将做的事情是否有风险。它在即将搞坏什么东西时不会犹豫。它不知道什么时候该停下来。正如 Louis Bouchard 在 2026 年 3 月关于框架工程的文章中一针见血地指出的那样："更大的上下文窗口并不能神奇地把一个不靠谱的智能体变成一个可靠的系统。" 
 如果你只是用模型完成一次性任务，上下文工程就够了。但如果模型在执行多步骤的任务 —— 编写并运行代码、调用工具、在长时间会话中运行并需要断点续接 —— 仅靠上下文工程，是挡不住事情走向失控的。 
 这时候，你需要的是一个框架。 
 【早阅】Dropbox 工程师职业发展框架：什么是影响力？ 
 框架工程 
 它是什么 
 在现实世界中，harness（安全带 / 安全装置）是一组绑带和连接件，让人或物体在危险环境中工作时保持安全。攀岩者穿戴安全带，高架脚手架上的建筑工人穿戴安全带。安全带不做具体的活儿，人来做活儿。安全带确保的是 —— 万一出了差错，后果被控制在有限范围内。 
 框架工程将同样的理念应用于 AI 工具。 
 你围绕 AI 模型构建一套规则、检查机制和安全保障。模型仍然负责干活。框架确保的是：如果模型做错了什么，问题能被及早捕获，不会蔓延扩散。 
 Mitchell Hashimoto 对此有一个我觉得特别清晰的表述：当 AI 犯了一个错误时，你的任务是构建某种机制，确保这个特定的错误永远不会再次发生。不是仅仅修正这次的输出，而是改变整个系统，让模型下次不可能再犯同样的错。 
 这是一种完全不同于 "反复调整提示词直到得到好答案" 的思维方式。 
 框架的五个层次 
 一个精心构建的框架有五个层次。每个层次提供不同类型的安全保障。你不需要从第一天就把五层全部建好，但理解它们有助于你判断，对于你正在构建的东西而言，哪些层次最为关键。 
 框架工程的五个层次 
 第一层：边界限制 
 你首先要定义的是：AI 工具被允许做什么，以及它绝对不允许做什么。 
 这里说的不是指令或指南，而是硬性的结构限制。你不是在请求模型小心行事，你是在搭建环境，使得某些事情根本不可能发生。 
 一些常见的例子： 
 沙箱环境。 你在容器或虚拟机中运行 AI 工具，它只能看到特定文件，无法触及工作区之外的任何东西。即使模型生成了试图删除范围之外文件的代码，环境也不会让它得逞。 
 只读访问。 对于系统中某些敏感部分，你只授予 AI 工具读取权限，不给写入权限。它能看，但不能动。 
 网络规则。 你可以限制工具能够调用的外部地址。这能防止 AI 智能体向它不应该通信的地方发起意料之外的 API 调用。 
 限定文件访问范围。 你告诉工具它被允许在哪些目录中工作。如果你要求它修复支付服务的一个 bug，即使认证服务在同一个代码仓库里，它也不应该去碰。 
 这个层次之所以排在第一位，是因为它不依赖于模型做出正确的行为。它不依赖好的提示词或精心的指令。这些限制由环境本身来强制执行。这使得它们具有一种指令所不具备的可靠性。 
 框架工程 —— 边界限制 
 第二层：指令 
 第二层是你告诉工具应该做什么、遵循什么约定、避免哪些错误的地方。 
 这不同于一次性的提示词。这些是适用于工具处理的每一项任务的常备指令。它们存放在一个随项目一起的文件中，通常命名为 AGENTS.md 或 CLAUDE.md 之类，取决于你使用的工具。 
 【第2415期】Vue Directive指令的详细使用原理 
 这个文件可能包含： 
 你的团队使用的编码风格 
 哪些库是经过审批的，哪些不是 
 如何处理错误（记录日志，不要默默吞掉） 
 任务不明确时该怎么做（询问而非猜测） 
 过去曾导致问题的模式 
 文件、函数和变量的命名约定 
 Hashimoto 的核心洞见是：每次 AI 犯错时，你都应该更新这个文件。 不只是修正输出，而是更新文件。这样，教训就被记录下来了，工具的下一次运行从一开始就内置了这条教训。 
 随着时间推移，这个文件会变成你的团队在特定项目中与 AI 协作所积累的一切经验的日志。它是机构知识 —— 以模型能够真正读懂的形式记录下来。 
 指令 
 当我遇到 Secrets Manager 的 DNS 解析问题时，修复的关键不仅仅是改代码，而是理解为什么旧模式是错的，并把这种理解写下来，让错误不再重演。你的 AGENTS.md 就是那份总结，只不过是写给你的 AI 工具看的。 
 第三层：检查验证 
 第三层是你验证 AI 产出的东西是否真正能用、是否符合你的标准的地方。 
 对大多数开发者来说，这是最熟悉的一层，因为 CI/CD 流水线一直在做的就是这件事。区别在于，现在你需要把这些检查变成 AI 工具无法跳过的硬性关卡。 
 你需要关注的检查包括： 
 代码规范检查（Linting）。 代码是否遵循了格式和风格规则？Linter 能自动捕获这些问题，而且每次给出的结果完全一致。你不需要人工去逐行审视缩进。 
 类型检查。 如果你使用的是强类型语言，代码是否通过了类型检查器？这能在代码运行之前就拦截整整一类 bug。 
 单元测试。 现有的测试还能通过吗？AI 有没有破坏之前正常运行的东西？这些必须每次都跑。 
 集成测试。 被修改的代码与系统的其余部分能否正确协作？这更难自动化，但能捕获单元测试遗漏的问题。 
 安全扫描。 AI 是否引入了明显的安全隐患？有自动化工具可以做这件事。它们并不完美，但能捕获常见错误。 
 这一切中最关键的词是 "关卡"。AI 工具在这些检查通过之前不能继续推进。这不是建议，而是硬性要求。 
 检查验证 
 一个能够自我监视并对问题做出反应的系统，远比一个依赖人类去发现异常的系统可靠得多。验证层对于 AI 生成的输出而言，正是如此。 
 有一点值得注意：这些检查是确定性的。同样的代码输入，每次都会得到同样的结果。无论你用的哪个模型、提示词怎么写、今天星期几，结果都不会变。正是这种一致性，赋予了这一层如此重要的价值。 
 第四层：恢复 
 第四层回答的是一个听起来显而易见、却常常被忽视的问题：检查失败了怎么办？ 
 如果没有恢复层，你只剩下两个糟糕的选项：AI 工具陷入死循环，不断试图修复一个它解决不了的问题；或者每次出了问题都需要人工介入。当你试图构建可靠的系统时，这两种情况都不可接受。 
 恢复层为你提供结构化的失败处理方式。 
 带次数限制的重试。 如果检查失败，把失败信息反馈给 AI，让它再试一次。但要设上限。如果连续失败三次，就停下来升级处理。一个不断重试又不断失败的 AI 既浪费时间，还可能把事情越搞越糟。 
 回滚。 如果 AI 做了一系列更改，系统状态反而比之前更差了，你需要一种方式把一切撤回到上一个已知的正常状态。如果你事先设置得当，版本控制就能实现这一点。 
 缩小范围。 有时候任务太大，AI 做到一半就迷失了方向。把任务拆成更小的片段，逐个单独重试，往往比重试整个任务效果更好。 
 升级给人类。 有些失败超出了自动化系统的处理能力。你需要一条清晰的路径，让这些情况摆到能够决定下一步怎么做的人面前。 
 框架工程 —— 恢复 
 恢复层是区分演示品和生产系统的分水岭。演示品可以承受有人盯着每次运行的代价。生产系统则需要在常规情况下自行处理故障。 
 第五层：审查 
 第五层是审查步骤，由另一个独立的 AI 工具或人类来完成。

 关键词是分离。完成工作的 AI 不应该是审查工作的那一个。有充分记录表明，模型倾向于认可自己的输出，即使它能看到其中的问题。它们对自己过于乐观。 
 一个独立的审查者，被赋予 "你唯一的任务就是找出这个输出中的问题" 这样的提示词，能够捕获原始工具遗漏的东西。它对已做出的决定没有任何执念，是用全新的眼光在审视。 
 有些团队设置了三步流程：一个工具规划方案，另一个工具执行工作，第三个工具审查成果。每个工具扮演不同角色，接收不同的指令集。

 框架工程中的审查流程 
 对于涉及安全的系统，最后的人工审查步骤不是可选项，而是任何东西上线前的最终关卡。前面的 AI 层所做的是缩小需要人类关注的范围，但最终拍板的仍然是人。 
 在 SSL 中，信任不是一个简单的 "是或否" 的决定，而是一条链 —— 每个环节都必须经过验证，你才能信任下一个。框架的工作方式如出一辙。每一层验证某个具体的方面。最终输出之所以值得信赖，是因为链条中的每一环都经受住了考验。 
 三种方法如何协同 
 提示工程、上下文工程和框架工程不是相互竞争的理念，而是同一事物的三个层面。如果你想构建可靠运行的 AI 工具，三者缺一不可。

 如何起步而不过度工程化 
 我是一个务实的开发者。除非复杂度能解决真实问题，否则我不愿意增加它。 
 所以，以下是我会根据风险大小来构建框架的方式： 
 如果你在探索或构建个人项目： 只从第二层开始。为你的项目写一份好的 AGENTS.md ，AI 每次犯错时就更新它。仅此一项就能大幅提升一致性。个人项目不需要沙箱或审查流水线。 
 如果你要在团队工作流中引入 AI： 加上第三层。把现有的代码规范检查和测试接入，作为 AI 无法跳过的关卡。这能给你信心 ——AI 不会在暗处悄悄搞坏东西。确保对人类编写的代码运行的检查，同样也对 AI 编写的代码运行。 
 如果 AI 会执行影响数据或服务的操作： 加上第一层和第四层。锁定工具能触及的范围。设置好回滚机制。在问题发生之前就定义好出了问题该怎么办。 
 如果 AI 在处理涉及安全或面向客户的系统： 加上第五层。内置审查步骤。至少，任何进入生产环境的东西都要经过人工审查。其他层级负责缩小需要人工关注的范围，人工审查负责做最终决定。 
 关于信任的一点思考 
 在同时与传统软件系统和 AI 工具打交道的过程中，有一件事始终萦绕在我心头：信任是你逐步建立的，一层一层来。 
 当你部署一个新的微服务时，你不会从第一天就完全信任它。你监控它，加上告警，观察它在真实负载下的表现。随着时间推移，当它证明了自己，你才更多地依赖它。 
 AI 工具也是一样。框架就是你系统性地建立信任的方式，而不是心存侥幸地等着最好的结果。 
 框架的每一层都是一个检查点。你每增加一个检查点，就多了一个理由可以说："我信任这个输出，因为我知道它通过了这些特定的测试，经过了第二个工具的审查，并且它能尝试做的事情本身就被严格限定了。" 
 这比 "我看了一下输出，看起来还行" 要坚实得多。 
 结语 
 从提示词到上下文再到框架的演进，是任何新技术从新奇走向生产应用的自然路径。 
 早期，兴奋点在于这项技术到底能做什么。你发现精心编写的提示词能得到比随意提问好得多的答案。太棒了。你分享有效的做法。 
 然后你开始用它做正经工作，你意识到，对一个问题得到一个好答案，和构建一个可靠的系统，是两回事。你开始思考上下文、一致性，思考模型需要知道什么才能做好它的工作。 
 然后你把它放进生产环境，你意识到可靠性不仅仅取决于好的输入。它取决于出了问题时会发生什么。它取决于确保错误被捕获、故障被处理、输出可以被信赖。 
 这就是框架工程。它不光鲜亮丽，不是出现在演示中的那部分。但它是区分 "有时能用的工具" 和 "你真正能依赖的系统" 的关键所在。
```

```
 【第3665期】WebMCP：让每个网站都能与智能代理互动的新标准 
 将编排层的职责交由 Claude 团队负责后，你只需专注于定义代理的行为 —— 撰写提示词、提供工具、设定约束条件。至于执行会话、运行环境以及编排过程中的诸多细节，Anthropic 的托管云运行时会一一代劳。 
 我们将以一个 GitHub 仓库审查代理为例来展示这一过程。它能够检出指定仓库、分析源代码与文档、发现潜在问题，并据此生成一份实用的 Markdown 报告。 
 这里的关键在于：这种方式让你的精力聚焦于提示词和行为设计，而非基础设施。 
 为什么 Claude 托管代理给人感觉与众不同 
 在传统的代理架构中，你总是需要亲自实现或适配编排机制、集成各类工具调用、处理运行时基础设施、决定代理的托管方式、应对并发执行、考虑文件处理，还要建立结果流式传输的机制。 
 【第3690期】驾驭 Opus 4.7：Claude Code 最佳实践全攻略 
 Claude Managed Agents 在其托管环境中替你打理了这些繁杂事务。你不必再把大量时间耗费在搭建脚手架上，而是可以将更多精力投入到定义： 
 代理应该做什么 
 它可以使用哪些工具 
 它应运行在怎样的环境中 
 你期望它产出什么样的结果 
 这才是真正的范式转变。它依然是工程实践，但更侧重于提示词优先、行为优先的开发模式。 
 我们要构建什么 
 我们将创建一个工作流，它能够完成以下任务： 
 将 GitHub 仓库挂载到一个托管会话中 
 检查仓库结构、文档和代码布局 
 识别架构异味与风险区域 
 生成一份 Markdown 报告，包含按优先级排列的 30 天行动计划

 为什么这很重要 
 使用 Claude 托管代理时，你依然需要精心设计代理 —— 提示词、工具、能力边界、输出格式，这些考量一个都不能少。但在运行时层面，你需要操心的事情已经大大减少了。 
 【第3638期】从 DevTools 到智能性能代理：用 AI 重塑前端性能分析的新方式 
 焦点的转变体现在： 
 花在搭建运行时环境上的时间更少了 
 对基础设施的顾虑更少了 
 为代理循环的可扩展性头疼的时候更少了 
 投入到打磨代理行为本身的精力更多了 
 生产环境注意事项 
 在生产系统中，我不会在每次请求时都重新初始化环境和代理。这些应当只创建一次，随后复用同一对象，并且对提示词和网络访问权限进行严格的版本管理与最小化控制。 
 我还会在工作流前端加设一个安全的应用层，而非直接暴露接口。 
 "托管" 并不意味着这里不存在架构设计，只是意味着架构被提升到了一个新的层次。 
 结语 
 Claude 托管代理之所以引人注目，是因为它将开发者的关注点从实现细节转向了行为与结果。 
 诚然，你可以花大量时间为自己的需求构建运行时，但托管代理提供了一条捷径 —— 让你得以专注于真正需要建模的部分：模型、提示词、工具和环境。其余的一切，都由托管系统在幕后为你打理。
```

## 5. 【早说】为智能体而设计：当软件的"用户"不再是人

- 日期: 2026-05-06 09:02
- 链接: https://mp.weixin.qq.com/s?__biz=MjM5MTA1MjAxMQ==&mid=2651279008&idx=1&sn=f35082d75dec9a87a3059edc3237f419

```
Teddy Riker 2026-05-06 09:02 福建 
 产品团队的使命也随之改写——你不再只为那个想点选、想核验的人而设计，而要为他身后那位直觉、上下文与局限皆迥异的"中间人"而设计。 
 前言 
 当 Salesforce 宣告 "Headless 化"、Ramp 的 MCP 周活三月暴涨十倍，一场静默的范式更迭已然展开：未来八成的软件交互，将由智能体代劳。产品团队的使命也随之改写 —— 你不再只为那个想点选、想核验的人而设计，而要为他身后那位直觉、上下文与局限皆迥异的 "中间人" 而设计。教会智能体如何成事、搭建反馈闭环、弥合上下文落差，方能在这场变局中立于不败之地。 
 本文来自 5 分钟新知精选。今日前端早读课文章由 @Teddy Riker 分享，@飘飘编译。 
 译文从这开始～～ 
 如果你和我一样，常常流连于 X（推特）的某个角落，刷过那些 "我如何用 Obsidian 打造第二大脑"、"Anthropic 刚刚彻底终结了某某行业" 之类的帖子，那你大概也见过这样一种论调：UI 已死。任何产品，若不能通过 MCP、API、CLI 或其他类似方式供智能体调用，便难逃被淘汰的命运。 
 【第3681期】拆解编程智能体：让大模型真正会写代码的六大核心组件 
 这股趋势在 Ramp 已然显现。过去三个月间，我们 MCP 上的周活跃用户增长了十倍 —— 越来越多的客户正通过 Claude、ChatGPT 等各类智能体与我们的产品打交道。 
 就在上周，Salesforce 成为首批拥抱这一理念的行业巨头之一。 
 VentureBeat 报道称： 
 周三，Salesforce 发布了其 27 年历史上最具野心的架构变革 ——"Headless 360"。这一宏大计划将平台的每一项能力都以 API、MCP 工具或 CLI 命令的形式开放出来，让 AI 智能体无需打开浏览器，便能驾驭整个系统。 
 此番宣告于旧金山召开的年度 TDX 开发者大会上揭晓，一举向开发者推出了逾百款全新工具与技能。它正面回应了悬于企业级软件头顶的那个终极之问：在一个 AI 智能体能够推理、规划并执行的世界里，企业还需要一个带图形界面的 CRM 吗？ 
 Salesforce 的回答是：不需要 —— 而这恰恰是关键所在。 
 这是 Salesforce 的一招妙棋，且我想绝非易于决断之举。随便问问销售人员，他们多半会向你倾诉自己有多么不待见 Salesforce。但这款产品之所以无处不在，正是仰仗其用户体验所积淀的熟悉感。销售主管们无意让团队从头适应新技术 —— 一致性，往往比功能本身更受推崇。 
 贝尼奥夫和他的团队坦然承认：这道护城河正在崩塌。他们顺势而为，接受了一个事实 —— 未来绝大多数的产品使用，将由 Claude、ChatGPT 以及其他用户从未谋面的后台程序所驱动。 
 我并不认为 UI 行将就木。人们依然想要点选操作，想要亲眼看见自己的配置，想要核验工作是否完成。但二八格局已然倒转：未来与软件交互的那 "八成"，将经由智能体来完成。这不仅改变了我们要构建什么，更重塑了我们该如何去构建。 
 【早说】智能体工程的八重境界 
 全新的交互范式 
 过去二十年间，人们与软件交互的主要路径一直是： 
 用户 → 界面 → 数据库 
 打开产品，点点选选，把事办成。界面，就是你感知软件的方式。对大多数人而言，界面即产品。 
 而随着智能体承担起越来越多的工作，一个新的层级悄然浮现： 
 用户 → 用户的智能体（如 Claude）→ 数据库 
 智能体代用户行事。它替你读取、写入、操控产品，省去你亲力亲为之劳。倏忽之间，界面消失了，智能体径直与底层系统对话。 
 但即便如此，格局仍在飞速演变。软件公司正在（也理应）设计属于自己的智能体与能力。于是，新的范式更近乎这样： 
 用户 → 用户的智能体 → 软件的智能体 → 数据库 
 在这一模式下，软件方的智能体替用户方的智能体处理复杂事务：套用业务逻辑、执行规则约束、补足对方所欠缺的上下文。两个大语言模型协同作战，共同推动结果落地。 
 【早说】编程智能体如何重塑工程、产品与设计 
 教会智能体如何把事做成 
 我大部分的头脑风暴、写作与构思都是与大语言模型一同完成的。一份草稿写就待发时，我便通过 Notion 的 MCP 服务器将其推送过去。多年来我一直是 Google Docs 的死忠，是 Notion 的 MCP 让我倒戈相向。 
 作为 Notion MCP 的用户，有一点令我深为称道：每次我让智能体撰写内容，它总能呈现得恰到好处。表格、要点、斜体、列表，应有尽有 —— 智能体从不掉链子。 
 这并非偶然，而是匠心使然。 
 notion-create-pages 这个工具的说明开篇便是：" 如需获取完整的 Markdown 规范，请始终首先调取 MCP 资源 notion://docs/enhanced-markdown-spec 。切勿臆测或杜撰 Markdown 语法。" 于是当我让智能体往某个页面写内容时，它做的第一件事便是抓取规范，再行落笔。每一项 Notion 特有的设定，都被明确标注出来，与通用模型的默认行为划清界限。 
 倘在旧时光景，这份规范会安静地躺在 API 文档里，对接 Notion 的开发者得自行研读、消化吸收，再写一层转换逻辑。如今，Notion 把规范直接、即时地交到智能体手上 —— 正在它需要之时。 
 倘若你用过 Slack 的 MCP，大概体会过截然相反的窘境。你的智能体想当然地按标准 Markdown 行事，全然不顾 Slack 那套独有的格式规则。结果你花在调整格式上的工夫，比自己动手写这条消息还要多： 
 诚然，格式指引就摆在网上，你也可以将其存档，再教你的智能体如何取用。可这未免太麻烦了，本不该如此。 
 想一想，调用你智能体的那一方需要知道些什么才能把事办成，然后主动奉上。别让他们自己去摸索。 
 【第3589期】写好工具，智能体才更聪明：Claude 的自我优化实践 
 搭建反馈闭环 
 Ramp 的 MCP 初次上线时，可观测性是我们最棘手的难题。我们能看到工具调用的总量，却看不见催生这些调用的对话上下文。光有数字，无从知晓什么奏效了、什么出岔子了，乃至用户究竟想做成什么。 
 我们从几个方向破局： 
 1、强制每次调用附带 "调用缘由"。 
 每一次 MCP 或 CLI 工具调用，都要求智能体附上一个 rationale 参数，说明为何发起此次请求。我们看不到对话本身，但凭借这些缘由便能还原其意图。归纳缘由中的规律，便能洞悉用户真正想要做的事。 
 2、一个反馈工具。 
 我们专门发布了一款独立工具，供智能体在受阻或遭遇行不通的模式时调用。智能体会上报：它原本想做什么、试过哪些办法、又卡在了何处。 
 3、因工具而异的 "种子参数"。 
 我们为各个工具量身添加专门的参数，捕捉日后可能用得上的上下文 —— 那些智能体本就掌握、否则我们只能靠揣测的信息。 
 设想你正在打造一款客服平台，向客户提供拉取工单的工具。日子一久，你在缘由日志里反复瞥见同一类措辞："撰写故障报告"、"起草事故纪要"、"为故障复盘汇总工单"。 
 这不就是一个全新的产品功能吗！一款 "故障报告生成" 工具大可应运而生：识别相关工单、评估严重程度、定位受影响的客户群，再以一种立场鲜明的格式起草摘要。 
 待此功能上线，你或许会陆续收到针对它的反馈："报告把三天前的工单也一并拉了进来，那些跟本次故障无关"，或是 "它老把免费用户的工单算进来，可这些不该出现在复盘里"。一转眼，你竟有智能体在向你的智能体精准地指点开发方向。智能体诚然会出现幻觉，可在反馈这件事上，它们往往比绝大多数你能交付服务的人类用户还要具体、还要前后如一。 
 报告若拉来无关工单，便添个日期范围参数；若不该收录免费用户，便加个客群筛选项。每一道反馈闭环，都化作产品进化的一个新阶梯。 
 留意上下文落差 
 在任何智能体交互中，你的系统握有调用方智能体所不知的上下文，而调用方智能体也握有你的系统所不知的上下文。设计这类交互时，你应当对双方各自的优势心中有数。 
 假设 Diego 出了一趟差。Diego 的 AI 幕僚长在 Slack 上接到了费用管理系统智能体的提醒：他这趟差旅有几笔开销尚未报销。此刻，两个智能体的目标同时锁定同一件事：把这些开销妥帖地报上去。 
 它们各自携来的上下文截然不同。 
 Diego 的幕僚长所掌握的： 
 Diego 的日历：清楚哪些会议何时、与谁召开 
 Diego 的邮箱：酒店与航班的确认邮件就附在其中 
 Diego 的 Slack：能将那顿 Kokkari 的晚餐与他邀请 Acme 团队的某条对话串联起来 
 Diego 的票据（从邮件附件与相册中提取） 
 费用管理系统所掌握的： 
 原始交易数据（如商家、交易时间） 
 公司的报销政策 
 公司的总账科目 
 公司过往的归类规律 
 传统 API 的做法，是把皮球一脚踢回给用户："这里有一笔交易需要总账编码。请调用此接口获取 150 个总账科目选项，然后挑一个。" 
 而精心设计的智能体交互恰恰反其道而行之 —— 它不索要总账编码，它索要的是上下文：这是客户餐叙、团队聚餐，还是私人行程？幕僚长智能体便能从某条日历记录或某段 Slack 对话中觅得答案。费用管理系统再据此填上正确的编码 —— 它原先所欠缺的那块上下文。 
 Diego 和他的智能体始终无需操心总账编码为何物，而财务团队则得到了准确的归类。双方各贡献其所知，共同交付一个对 Diego—— 以及他的会计 —— 皆有裨益的结果。 
 设计这种智能体之间的交互时，请务必留意上下文落差。坦承你的智能体力有不逮之处，并无不妥 —— 你们本就在服务同一位用户。 
 界面，曾经横亘在 Diego 与他的费用系统之间；如今，它横亘在他的智能体与你的智能体之间。 
 这一转变，重新定义了产品团队的使命。你过去为之设计的是一个想要事半功倍、规避差错、亲眼见证成果的人；如今你仍在为同一个人设计，却隔着一位中间人 —— 其直觉、上下文与局限，都与那个人迥异。教会智能体如何把事做成、搭建反馈闭环、留意上下文落差 —— 这三件事追问的其实是同一个根本命题：调用你智能体的那一方，需要什么才能把活干漂亮？而这些，你都奉上了吗？ 
 大多数公司会推出一个 MCP、打个勾、然后翻篇了事。他们的使用量会增长几个季度，随后陷入停滞。日子一久，客户会涌向那些在细节上下足了功夫的产品，绕开那些没有的。 
 请以你昔日伺候人类用户的那份用心，去伺候智能体。不知不觉间，签字付钱的那一位，将会是它。
```

## 6. 【早说】依赖冷却期，是在让你搭别人的便车

- 日期: 2026-04-30 09:03
- 链接: https://mp.weixin.qq.com/s?__biz=MjM5MTA1MjAxMQ==&mid=2651279004&idx=1&sn=4f08579ed837965b843b1a013e64901f

```
Cal Paterson 2026-04-30 09:03 福建 
 依赖冷却期一夜走红，眼看就要被奉为行业最佳实践。 
 前言 
 依赖冷却期一夜走红，眼看就要被奉为行业最佳实践。可这套 "等几天再升级" 的法子，本质上是在搭便车 —— 把没设防的人当作免费小白鼠。本文主张以 "上传队列" 取而代之：在中央索引层面把发布与分发解耦，让安全防线一次筑牢，而非让每个项目反复劳神。 
 本文由 5 分钟新知精选。今日前端早读课文章由 @Cal Paterson 分享，@飘飘编译。 
 译文从这开始～～ 
 与其人人设防，不如源头把关：上传队列才是供应链安全的正解 
 依赖冷却期（dependency cooldowns）忽然成了香饽饽。它在很短时间内便广受推崇，眼看就要被一路绿灯送进 "行业最佳实践" 的篮子里。 
 【第3633期】管理依赖，其实是管理工程思维 
 其立意是：在新版本发布后先按兵不动 N 天再升级，而非贸然跟进，这样一来，任何被人暗中植入的恶意代码大概率已被旁人察觉，问题版本也已被 "撤回"。乍看之下，这思路相当奏效 —— 毕竟绝大多数供应链攻击确实会在几天之内浮出水面。 
 然而，依赖冷却期对身体力行者的好处不过尔尔，却把可观的代价转嫁给了所有其他人。而且，它压根没触及问题的症结：发布与分发本是两码事，没人说得清为何非得把二者捆在一起。 
 依赖冷却期 —— 单兵作战之困 
 说白了，依赖冷却期的运作机制，无非是搭别人受难的便车。整套方案的根基，便是寄望于那些没机灵到去设个冷却期的人，充当新版本的免费 "小白鼠"。一旦出了纰漏，倒霉的是这些可怜虫 —— 他们被黑了，事情曝光了，问题版本在冷却期满之前就被撤了下来。 
 哪怕这套办法对个人确实管用，要把它当作整个生态系统都该恪守的合理或合乎道义的制度，我看也是难以为继的。 
 另一个症结在于，依赖冷却期需要五花八门的人都跟着忙活。Python 如今已经有了多少个包管理器了？八个？每一个都得实现冷却机制；而每一个项目都得动手配置冷却期 —— 可不同的包管理器往往各有一套配置法门，一点也不直观，更谈不上轻松。 
 实际上，连冷却期的拥趸自身也难逃池鱼之殃。你在项目配置里精心设下的冷却期，太容易在不经意间被绕开。就拿 Python 来说，前些日子若你在项目配置之外随手敲一句 pip install litellm ，立马就着了道。所以这套冷却之法既不周密，也算不得多保险。 
 【早阅】让依赖更轻，让构建更快：tinyglobby 的演进之路 
 到了某一天，多半是借由大语言模型的辅助，或是老掉牙的复制粘贴 —— 眼下大多数项目配置正是这般炮制出来的 —— 一份 "尽责的" 冷却期配置便摇身一变成了事实上的默认范式。套用 Greenspun 的名言略加改写：任何足够普及的依赖冷却期，最终都会演化成一个临时拼凑、规范含混、漏洞百出、运转迟缓的上传队列。 
 上传队列 —— 集中行动的诸多妙处 
 显而易见的另一条路是：与其让每一个人在每一个包管理器、每一个项目里反反复复地配置冷却期，倒不如在中央依赖服务器上一劳永逸地办妥此事。这便是 "上传队列"—— 让新发布的包静候一段时日，而后再行分发。 
 这里所说的 "发布"，指的是把版本（tarball、whl、gem 之类）推送到中央索引（npm、pypi、rubygems）。而 "分发"，则是中央索引开始把该版本提供给公众下载。 
 在发布之后、分发之前的这段窗口期里，可以做的事情着实不少：跑一遍内部 lint 工具；把包交给外部的自动化安全扫描程序检阅（若能查出端倪，其品牌大可被显赫地挂上首页）；公开展示构建产物的 diff；甚至允许那些自愿请缨的 "志愿" 测试者先行尝鲜。 
 上传队列并非空中楼阁 ——Debian 项目早已身体力行。包被上传到仓库后，至少要等上 2 至 10 天才能进入 "testing" 发行版。上传队列把包的发布和包的分发拆作了两件事。 
 发布，是把包做好、推送到仓库；分发，则是把包公之于众。这两件事并无非得同时进行的道理，无非是各语言生态各自独立的包索引在发展过程中遗留下来的一桩历史巧合罢了。 
 上传队列与依赖冷却期所图者同，却无前者那一身毛病。它解决了搭便车的难题：那些不擅配置的人，再也不会被冷却派当作免费的小白鼠。包管理器无需为此开发新功能；项目无需平添一项配置；安全检查工具也无需对此挑刺。即便你在某台开发机上随手装了个未经配置的小包，也照样安然无恙。 
 【早阅】如何有效清除NPM和NPX缓存？ 
 让 "突袭" 无处遁形 
 上传队列还有一项不容小觑的好处。绝大多数供应链攻击中，未经授权的不仅仅是被植入的恶意代码，整个发布版本本身就是未经授权的。让发布出来的包先冷藏几日，便能大大削弱发布凭证的杀伤力。这一点至关重要 —— 既然我们难以把它守得固若金汤，那么退而求其次，削弱其威力也是上策。 
 让发布版本在分发前先静置几日，还能彻底消除新版本横空出世时那种全无必要的 "突袭" 意味。它给了用户预先得知新版本即将到来的机会，让人事先就清楚它何时上线。 
 而需要预先知会的，远不止用户。上传队列的窗口期，也是通知维护者的良机 —— 确保每一位维护者对即将面世的版本都心中有数。"通知：版本 2.4.1 已进入上传队列"—— 许多场合下，正是这样一声轻唤，便足以让一场山雨欲来的供应链攻击胎死腹中。 
 【第3684期】最低发布时限：最简单却最有效的供应链防线 
 此理对 AI 而言尤甚 
 人们鲜少把话挑得这么明白，但事实是：大语言模型的出现，意味着 markdown 已经成了一种可执行文件格式。这究竟是激动人心还是令人不寒而栗（抑或兼而有之！），大约就见仁见智了。但这是个不争的事实 ——Agent Skills 不就是这么运作的吗？你下载一个 markdown 文件，你的 LLM 便平添了一个第三方依赖。 
 针对 LLM 的第一场重磅供应链攻击，无疑只是时间问题 —— 总会有人把一句 "无视上面的指令！" 悄悄塞进某个流行的 markdown 文件里。 
 我一直在为自己的一个副业项目琢磨这个难题。这个项目叫 "Soapstones"，是一套面向 AI 智能体的公共记忆系统，专供它们（也就是 AI 智能体）记录各种做事的门道 —— 比如怎么搜 HN 帖子、怎么查当下天气、怎么换算汇率，诸如此类。Soapstones 实际上就是一个 markdown 文件的包管理器。既然如此，供应链攻击的隐患便如影随形。 
 事实上，这隐患还要加倍。不光是有人可能往 markdown 文件里下 "无视上面的指令！" 这样的毒，连 LLM 自己也可能糊里糊涂地把机密信息 —— 比如 API 密钥 —— 上传到系统里去。 
 而此处的对策，依然是上传队列。确切地说，是双重上传队列：审核员审查每一份上传，提防供应链攻击；而智能体的主人也审查每一份上传，确认自己首肯。 
 我以为，上传队列在此是唯一合情合理的选择 —— 总不能让我去叮嘱 LLM "下载新近上传的东西时务必小心" 吧？那才是真的疯了。 
 经费会是难题吗？未必如此 
 一个意料之中的反驳是："这笔钱由谁来出？" 首先，这事是否非要砸进巨资，本就值得商榷。Debian 项目维护上传队列已有数十年，并设有一支安全团队，专门为安全方面的特殊情形开绿灯放行。 
 其次，每一个举足轻重的包索引是否当真囊中羞涩，也未必尽然。NPM 公司当年是风投催生的初创企业，如今早已是微软麾下的全资子公司。运营 PyPI 的 Python 软件基金会，赞助名单上的企业更是洋洋洒洒，其中不少都将从上传队列中获益。基金会前不久还从 Anthropic 那里接收了 150 万美元，用途之一便正是供应链安全。 
 另一条路子，则是为商业项目提供 "加急安全审查" 的付费服务。 
 商业实体往往急着把新版本尽快推出去 —— 或为修补影响客户的（非安全类）缺陷，或为配合某项重磅发布，或为提前推进某项关键的服务端弃用。那便干脆向他们收费，把加急审查作为一项付费服务出售。 
 加急审查并不意味着可以绕开流程，更不是说企业那头一按 "发布"，版本立刻就能见诸天下 —— 所有自动化检测仍须跑完。它的意思无非是：通过人工把关，让某个特定版本的实际等待时间大幅压缩。 
 包索引本就需要安全响应团队：撤回问题版本、维持禁运期、应对域名抢注、协调 0day 漏洞披露等等。向商业项目收取加急审查费，是为这套体系交叉补贴的一条妙径 —— 让企业的燃眉之急，反过来供养起服务于整个生态的安全机器。 
 个人理性，集体疯魔 
 依赖冷却期这种事，自己用着，效果尚可。我们多多少少都有几分这样的做派。某些东西我自己也不愿当头一个尝鲜的 —— 家里那台电视机顶盒，就是我们家不容有失的关键基础设施。 
 然而，把每一次升级时因人而异、因事而异的主观应对，硬生生固化为整个社区的最佳实践，性质便完全不同了。我可不愿自己的安全，要建立在别人先被黑一回的基础之上。
```

```
 ease-out ：元素进场或退场时使用。开头快、结尾慢，符合人对 "抵达" 和 "离开" 的直觉感受，也是拿不准时最安全的默认选项。 
 ease-in-out ：元素在屏幕上移动或变形时使用。两头缓、中间快，运动感最自然，像现实中物体的加速与减速。 
 ease ：悬停等轻量交互使用。节奏柔和、不抢注意力，恰好匹配悬停这种 "轻触即走" 的交互分量。 
 linear ：匀速运动，适合加载旋转、进度条等需要持续、均匀推进的场景。 
 有了这张流程图，智能体便不再需要凭空猜测，只需顺着判断条件走一遍，就能选出符合你审美哲学的缓动曲线。 
 照这个思路，我们可以把动画的其余要素逐一覆盖，让智能体在每种情形下都心中有数、信手拈来。时长便是另一块拼图： 
 这是一份 UI 动画时长指南，帮你为不同类型的元素设定恰当的动画节奏： 
 动画时长参考表 
 元素类型 
 建议时长 
 微交互（按钮反馈、开关切换等） 
 100–150ms 
 常规 UI（提示气泡、下拉菜单等）

 核心规则： 
 UI 动画时长不应超过 300ms—— 再长就会让人觉得 "卡"，用户等不了 
 元素越大，动画越慢；元素越小，动画越快 —— 大面积的变化需要更多时间来消化 
 退场动画可以比入场快约 20%—— 用户关闭某物时，期望它迅速消失，不必恋恋不舍 
 时长与距离匹配 —— 移动路径越长，动画时长应越长，反之亦然 
 这套规则的底层逻辑仍然是对真实世界的模拟：大物体惯性大、启停慢，小物体轻盈灵活；东西走得远自然需要更多时间。把这些物理直觉量化成具体的毫秒数，智能体在写动画时就有了清晰的刻度尺，而不是随手写一个 500ms 让整个界面慢吞吞。 
 任何关于 "审美" 的抉择，都可以这样封装成一份技能文件 —— 无论是版式布局、图标设计，还是色彩理论。以下便是我某份技能文件中关于字体排印的几条规矩： 
 这是一份字体排印规则清单，确保界面文字既好看又好读： 
 【第3619期】让字距随字体自适应变化的 CSS 技巧 
 字体排印七条准则 
 1、正文行宽限制在 65 字符左右 ，不要让文字铺满整个容器。行太长，眼睛从行尾跳回行首时容易跟丢，阅读节奏也会被打散。 
 2、价格列使用 tabular-nums ，让数字等宽对齐。否则不同数字宽度不一，列看起来参差不齐，尤其在表格中非常刺眼。 
 3、截断文本用 … 字符而非三个点 ... ，这样省略号会跟随容器宽度自然截断，而不是在固定字符数处生硬断开。 
 4、大写标签适当加大字间距 。大写字母本身就显得紧凑，不加间距会让人觉得挤在一起喘不过气。 
 5、声明一组备用字体，其 x 高度和字重要与主字体匹配 ，这样字体加载过程中不会产生布局跳动，页面始终稳定。 
 6、下划线只留给链接 。需要强调非链接文字时，用字重或颜色来处理。下划线一旦滥用，用户就无法一眼分辨哪些文字可以点击，哪些不可以。 
 7、界面中的强调用粗体，斜体留给引用和语气重音 。斜体在排版传统中属于编辑语境，放到 UI 里容易显得格格不入，粗体才是界面层级最自然的表达方式。 
 这七条规则的共同特征是：它们都不是装饰性的讲究，而是直接影响可读性、可用性和视觉稳定性的实际决策。写进技能文件后，智能体在生成任何涉及文字的界面时，都不会再犯这些常见的排版失误。
```

```
 症结何在？ 
 1. 它是在故纸堆里练出来的 
 它缺乏新鲜的训练素材。 
 它对标准模板的依赖近乎病态 —— 没办法，整个互联网充斥的就是这些玩意儿。现代 CSS？它几乎一无所知。 
 2. 它真的看不见 
 它是个语言模型，不是渲染引擎！ 
 它在数学上的拉胯是出了名的，你就算把截图甩到它脸上，也几乎无济于事。它不过是在黑暗里瞎捅刀子。

 AI ："大功告成！为您呈上这份精雕细琢的 UI。"

 它参不透我们架构决策背后的那个 "为什么"。 
 SDD（规范驱动开发）、BDD（行为驱动开发）或状态机或许能给它指点迷津，可惜这些方法论与之配套的优质范例，压根就没怎么进过模型的训练集。 
 我们无非是在要求一个庞大的文本预测器当场搭建新的逻辑桥梁。我们能把它推到那一步，但要考虑的东西实在太多，非得一条一条掰开揉碎了讲明白，它才能开始搭起我们想要的那些连接。 
 【早说】命令行才是AI工具的未来 
 4. 对运行环境毫无掌控 
 它管不着自己写的代码住在哪儿。 
 它能写出一手让人又爱又恨的 Rust、TypeScript 或 Python—— 但这些语言有个得天独厚的优势：它们的运行环境是可预测的，甚至可以锁版本（比如钉死在 v14.4！）。 
 HTML 和 CSS 可没这待遇。你锁不住浏览器类型、锁不住窗口尺寸、锁不住浏览器版本、锁不住用户的输入方式（键盘、鼠标、触屏、语音）、更锁不住用户的个性化偏好…… 这是一锅彻头彻尾的环境乱炖。 
 类似的清单还长着呢 —— 渲染引擎在敲定最终输出前，要同时杂耍般处理的各种场景、上下文和变量，一个比一个多。LLM 对这些毫无掌控，于是干脆视而不见，除非你把它们摆到它眼前。

 我们不是一个静止的靶子。我们不是一种可以被学会的模式。

 缺乏意图与动机：fry69 认为，模型虽然掌握了机械的逻辑，但缺乏 “生命过程”—— 即那些由无数次失败积累而来的设计直觉、动机和经验。

 2. 实用派：把它当成 “脚手架” 而非 “代笔” 
 这部分开发者认为 AI 的价值取决于你怎么用： 
 脚手架 vs 定制：Katrin K. 将 AI 比作 Bootstrap。如果你只是需要一个快速原型或不介意平庸的设计，AI 非常棒；但一旦涉及定制化，AI 就会变成阻碍。 
 “Vibe Coding”（氛围编程）：Andreas Møller 认为，如果你想让 AI 直接搞定整个前端，你会打架打得很辛苦，但如果你只是让它搭建一个基础框架然后自己微调，它非常高效。 
 3. 反击派：AI 已经在进步，甚至能教 “老鸟”

 审美的丧失：Anthony Whitford 提出了一个终极难题：“你该如何向 AI 定义并沟通什么是‘品味（Taste）’？” 
 讨论总结： 
 评论区的核心矛盾点在于：你是把 AI 看作一个试图替代你的 “初级开发者”，还是一个受你调教的 “高级工具”？ 大多数开发者承认 AI 在处理 CSS 复杂矩阵、响应式环境和人类多变性方面确实表现拙劣，但也有人通过深度定制化的 Prompt 和工作流，开始在前端开发中挖掘出 AI 的巨大潜力。
```

```
 Opus 4.7 的 token 用量和行为表现，会因使用场景的不同而有所差异 —— 是部署单轮用户输入的自主异步编程智能体，还是多轮用户交互的同步协作编程智能体。在交互式场景中，模型会在每次用户输入后进行更多推理：这有助于提升长会话中的连贯性、指令遵循能力和代码质量，但相应地也会消耗更多 token。 
 为了在 Claude Code 中充分发挥 Opus 4.7 的潜力，我们发现一个很有效的方法是：把 Claude 当作一位能力出众、值得委以重任的工程师，而非需要你逐行指导的结对编程搭档。具体建议如下： 
 在第一轮对话中就把任务说清楚。 
 结构完善的任务描述 —— 包含意图、约束条件、验收标准以及相关文件路径 —— 能为 Opus 4.7 提供充足的上下文，从而输出更高质量的结果。若将模糊的需求分散在多轮对话中逐步传达，往往会降低 token 使用效率，有时还会影响整体质量。 
 减少不必要的用户交互次数。 
 每一轮用户输入都会带来额外的推理开销。尽量将问题集中在一起，一次性提供模型所需的全部上下文，让它持续推进。 
 在合适的场景下使用自动模式。 
 对于你信任模型可以安全执行、无需频繁确认的任务，自动模式能显著缩短周转时间。它尤其适合那些已在前期提供了完整上下文的长时间运行任务。自动模式目前已面向 Claude Code Max 用户开放研究预览版 —— 你可以通过 Shift+Tab 来切换开启。 
 设置任务完成通知。 
 你可以要求 Claude 在完成任务时播放提示音，它能自行创建基于钩子的通知机制。 
 【早说】不用IDE，只用计划文件和语音：所知道的Claude Code全部技巧 
 Opus 4.7 推荐的投入层级设置 
 Opus 4.7 在 Claude Code 中的默认投入层级现已调整为 xhigh。这是介于 high 和 max 之间新增的层级，让用户在处理困难问题时，能更精细地把控推理深度与响应速度之间的平衡。我们建议大多数智能体编程工作采用 xhigh，尤其适用于对智能水平要求较高的任务，如 API 与数据结构设计、遗留代码迁移，以及大型代码库审查。 
 以下是各投入层级的详细使用建议： 
 medium 和 low： 
 适用于对成本敏感、对延迟敏感或范围明确的简单任务。在较难的任务上，模型的表现不及更高层级，但仍然优于同等层级下的 Opus 4.6—— 有时甚至消耗更少的 token。 
 high： 
 在智能水平与成本之间取得平衡。如果你需要同时运行多个会话，或者希望在不大幅牺牲质量的前提下控制开支，可以选择 high。 
 xhigh（默认，推荐）： 
 适用于大多数编程和智能体场景的最佳设置。具备出色的自主性和智能水平，同时避免了 max 层级在长时间智能体运行中可能出现的 token 用量失控问题。 
 max： 
 能在真正困难的问题上榨取额外的性能提升，但收益递减，且更容易出现过度思考。建议仅在特定场景下刻意使用，比如在评测中测试模型的能力上限，或者对智能水平要求极高且不计成本的任务。 
 如果你正在升级到新模型，我们建议你针对投入层级进行实验调优，而不是简单沿用旧的设置。你可以在同一任务过程中灵活切换不同的投入层级，从而更高效地管理 token 用量和推理深度。 
 【早说】用Karpathy的自动研究法，让Claude技能全自动进化 
 我们将 Opus 4.7 的默认投入层级设为 xhigh，是因为我们认为这是最适合大多数编程任务的配置。如果你是现有的 Claude Code 用户，且此前未手动设置过投入层级，系统会自动为你升级至 xhigh。当然，你仍可随时手动调整。 
 善用自适应思维 
 Opus 4.7 不再支持固定思维预算的扩展思考模式，取而代之的是自适应思维。该机制让思考在每一步都变为可选，由模型根据上下文自行判断何时需要进行更深入的思考。面对简单查询，它能迅速作答；当某个步骤无需深入思考时，它会果断跳过；而在思考 token 最能发挥价值的地方，它则会充分投入。在整个智能体运行过程中，这种机制的累积效果是响应更快、体验更佳。 
 自适应思维在本次发布中有了显著改进 —— 尤其是 Opus 4.7 不再像以前那样容易陷入过度思考。 
 如果你希望对思考频率有更多掌控，可以在提示词中直接说明： 
 想要更多思考时，可以这样写："请仔细逐步思考后再作答；这个问题比表面看起来更复杂。" 
 想要减少思考时，可以这样写："优先快速回复，无需深入思考。拿不准时，直接给出答案。" 这样可以节省 token，但在较难的步骤上可能会损失一些准确性。 
 值得了解的行为变化 
 从 Opus 4.6 到 4.7，有几项默认行为发生了变化。如果你此前针对旧模型精心调校过提示词或调用框架，以下内容值得关注： 
 回复长度会根据任务复杂度自动调整。 
 Opus 4.7 不再像 Opus 4.6 那样默认输出冗长内容。简单查询会得到简短回答，开放性分析则会得到详尽阐述。如果你的使用场景对回复的长度或风格有特定要求，请在提示词中明确说明。我们发现，提供你期望的表达风格的正面示例，效果优于 "不要这样做" 之类的否定式指令。 
 模型调用工具的频率降低，转而更多依赖推理。 
 在很多场景下这会带来更好的效果。如果你希望模型更频繁地使用工具（例如在智能体工作中更积极地搜索或读取文件），请在提示词中明确描述何时以及为何应当使用工具。 
 【第3681期】拆解编程智能体：让大模型真正会写代码的六大核心组件 
 默认生成子智能体的频率有所降低。 
 Opus 4.7 在决定是否将工作委派给子智能体时更为审慎。如果你的使用场景需要并行子智能体（例如跨文件扇出处理或处理多个独立条目），建议在提示词中明确指出。例如： 
 如果某项工作可以在单次回复中直接完成（比如重构一个你已经看到的函数），请不要派生子智能体。当需要跨多个条目扇出处理或同时读取多个文件时，请在同一轮中派生多个子智能体。 
 下一步可以尝试什么 
 Opus 4.7 在长时间运行的任务上表现优于此前的所有模型。这使它非常适合那些过去受限于人工监督瓶颈的场景，例如复杂的多文件变更、指向不明的疑难调试、跨服务的代码审查，以及多步骤的智能体工作。 
 我们建议将投入层级保持在 xhigh，然后看看仅凭第一轮对话能带你走多远。
```

```
 前言 
 AI 让代码生成变得廉价，却让理解力成为稀缺品。代码库看似整洁、测试全绿，人类对系统的真正理解却在悄然流失。这就是理解力负债 —— 它不像技术债务那样昭然若揭，而是无声积累，直到最糟糕的时刻集中爆发。 
 本文来自 5 分钟新知精选，今日前端早读课文章由 @addyosmani 分享，@飘飘编译。最近并行开发 vibe coding，有一种侥幸心理功能代码能跑就好，等有问题或有时间再去细看。 
 译文从这开始～～ 
 理解力负债，是人类过度依赖 AI 与自动化所付出的隐性代价，它悄然侵蚀着我们的智识与记忆。对工程师而言，这一问题在智能体驱动的工程实践中尤为突出。 
 【第3656期】深入理解 React 的 useEffectEvent：彻底告别闭包陷阱 
 当团队大量使用 AI 编程工具时，有一种成本不会体现在你的效率指标上 —— 尤其是当审查 AI 生成的海量代码变得令人不胜其烦之时。这种成本日积月累，终有一天必须偿还，而且还要连本带利。这就是所谓的理解力负债，也可以称之为认知负债。 
 理解力负债，是系统中实际存在的代码量与人类真正理解的代码量之间不断扩大的鸿沟。 
 与技术债不同，技术债务会通过不断增长的阻力昭示自己的存在 —— 构建越来越慢、依赖关系盘根错节、每次碰到某个模块都心生畏惧。理解力负债则不然，它滋生的是虚假的安全感。代码库看起来整洁干净，测试全线飘绿。而清算的时刻总是悄无声息地降临，往往就在最糟糕的节骨眼上。 
 【早阅】技术债治理手册：五步作战计划 
 玛格丽特 - 安妮・斯托里曾描述过一个学生团队的案例：到了第七周，他们突然发现自己连最简单的改动都无法完成，一碰就出问题。根源并不在于代码写得有多乱，而在于团队中没有一个人能说清当初为什么做出那些设计决策，也没人能解释系统各部分之间究竟是如何协同运作的。关于系统的整体认知，已经烟消云散了。 
 这就是理解力负债在实时地利滚利。 
 我在 Hacker News 的讨论帖中读到过工程师们真切地思索这个结构性问题的声音 —— 不是那种老生常谈的乐观派与怀疑派的对立，而是一个行业在真正试图厘清：当瓶颈已经转移，严谨究竟应该是什么模样。 
 展示了 AI 辅助对编程理解力的影响：使用 AI 进行代码生成的受试者，其理解力得分相较于使用 AI 进行概念探索的受试者有显著下降 
 Anthropic 近期发表的一项题为《AI 如何影响技能形成》的研究，揭示了过度依赖 AI 编程助手的潜在弊端。在一项针对 52 名软件工程师学习新代码库的随机对照试验中，使用 AI 辅助的参与者完成任务所花的时间与对照组大致相当，但在后续的理解力测试中得分低了 17%（50% 对 67%）。其中调试能力的下降幅度最大，概念理解和代码阅读能力的降幅虽小一些，但同样具有统计学意义。研究者特别强调，被动式的委托 ——"别管过程，能跑就行"—— 对技能发展的损害远甚于主动的、以提问为导向的 AI 使用方式。 
 【早阅】不要让技术债破坏你的产品路线图 
 存在速度的不对称性问题 
 AI 生成代码的速度远远超过人类评估代码的速度。这话听起来不言自明，但其深层含义很容易被低估。 
 当团队中的开发者亲手写代码时，人工审查流程虽然一直是瓶颈，但它是一个富有成效且具有教育意义的瓶颈。阅读同事的 Pull Request 本身就是一种强制性的理解过程。它能暴露隐含的假设，捕捉到与半年前系统架构设计相悖的决策，并将 "代码库到底在做什么" 这一关键知识，分散到每一个负责维护它的人手中。 
 AI 生成的代码打破了这个反馈闭环。产出的量太大了。而且输出在语法上干净利落，格式往往无可挑剔，表面上看完全正确 —— 这恰恰是过去让人放心点击合并的那些信号。然而，表面的正确并不等于系统层面的正确。代码库看上去健康无恙，理解力却在深处被悄然掏空。 
 我读到一位工程师说过：瓶颈从来都是一个有能力的开发者对项目的理解。AI 并没有消除这个约束，它只是制造了一种 "你已经挣脱了它" 的幻觉。 
 而且这种倒置比表面看来更加尖锐。在代码生产成本高昂的年代，资深工程师审查代码的速度快于初级工程师编写代码的速度。AI 颠覆了这一格局：如今一个初级工程师生成代码的速度，可以轻松超过一个资深工程师审慎审查代码的速度。那个曾经让代码审查得以发挥实质作用的制约因素，已经被移除了。曾经的质量关卡，如今沦为了一个吞吐量问题。 
 我热爱测试，但测试并非万全之策 
 面对审查瓶颈，本能地加大确定性验证的力度 —— 单元测试、集成测试、静态分析、代码检查工具、格式化工具 —— 这完全可以理解。在大量依赖 AI 编程智能体的项目中，我自己也经常这样做。用自动化来突破审查瓶颈，让机器去检查机器。 
 这确实有用，但天花板很低。 
 一套能覆盖所有可观测行为的测试套件，在很多情况下会比它所验证的代码本身还要复杂。而你无法推理的复杂性，并不能带来安全感。更深层的问题在于：你无法为自己压根没想到要定义的行为编写测试。 
 没人会写一条测试断言 "被拖拽的元素不应变为完全透明"。当然不会有人写。这种可能性从未出现在任何人脑海中。这恰恰是那类会悄然溜过的缺陷 —— 不是因为测试套件写得差，而是因为根本没人想到要往那里看。 
 【第3660期】别再为覆盖率而写测试：真正可靠的单元测试应该这样做 
 还有一种值得专门指出的特定失败模式。当 AI 改变了实现行为，并随之更新了数百个测试用例来匹配新的行为时，问题就从 "这段代码是否正确？" 变成了 "这些测试变更是否都有必要？我的覆盖率是否足以捕获那些我没有想到的问题？" 测试无法回答这个问题。唯有理解力才能。 
 数据也开始印证这一点。研究表明，将 AI 用于代码生成委托的开发者，在理解力测试中的得分低于 40%；而将 AI 用于概念探索 —— 提出问题、权衡利弊 —— 的开发者，得分则超过 65%。工具本身并不会摧毁理解力，摧毁理解力的是你使用工具的方式。 
 测试是必要的，但远远不够。 
 依赖规格说明书是好事，但也不是全部答案 
 一种常见的解决方案建议是：先写一份详尽的自然语言规格说明书，将其附在 PR 中。审查说明书而非代码，相信 AI 忠实地将意图转化成了实现。 
 这种想法的诱惑力，恰如瀑布式开发方法论当年的诱惑力：先严格定义问题，再去执行。关注点清晰分离，干净利落。 
 问题在于，将规格说明书转化为可运行的代码，涉及大量隐含决策 —— 边界情况、数据结构、错误处理、性能取舍、交互模式 —— 这些是任何说明书都无法穷尽的。两个工程师基于同一份说明书实现出来的系统，在可观测行为上会存在诸多差异。两种实现都没有错，只是不同而已。而其中许多差异，终将以无人预料的方式影响到用户。 
 关于详尽规格说明书，还有一个值得点明的问题：一份详尽到能完整描述程序的说明书，本质上就是程序本身 —— 只不过用的是一种不可执行的语言。编写一份足以替代代码审查的详尽说明书，其组织成本很可能会抵消甚至超过使用 AI 来执行它所带来的生产力收益。况且，你依然没有审查实际生成的东西。 
 更深层的问题是：很多时候根本不存在所谓 "正确的" 规格说明书。需求是在构建过程中浮现的，边界情况是在使用中暴露的。"在动手之前就能完整定义一个非平凡系统" 这一假设，已经被反复检验，屡屡碰壁。AI 并没有改变这一点，它只是在人类未经深思熟虑的情况下，多加了一层隐含决策。 
 向历史学习 
 数十年来，在分布式团队中管理软件质量 —— 面对参差不齐的上下文背景和沟通带宽 —— 已经沉淀出了真正经过实战检验的方法论。这些经验不会因为团队成员变成了一个模型就烟消云散。 
 【第3287期】Canyon: 提升JavaScript代码质量的全面覆盖率分析工具 
 AI 改变的是：成本（大幅降低）、速度（大幅提升）、人际管理开销（基本归零）。AI 没有改变的是：需要有人具备深厚的系统上下文，维持对代码库实际行为及其背后缘由的连贯理解。 
 这就是理解力负债所迫使我们面对的、令人不太舒服的价值重新分配。 
 随着 AI 产出量的攀升，那个真正理解系统的工程师会变得更有价值，而非更没有价值。能够看一眼代码差异就立刻判断哪些行为是承重墙；能够记住八个月前在压力之下为何做出那个架构决策；能够分辨一次重构究竟是安全的，还是在悄然改变用户所依赖的某些东西 —— 这种能力，将成为整个系统赖以运转的稀缺资源。 
 度量体系也存在盲区 
 理解力负债之所以如此危险，是因为你现有的度量体系中没有任何指标能够捕捉到它。 
 效率指标看起来完美无瑕。DORA 指标稳如磐石。PR 数量节节攀升。代码覆盖率一片绿灯。 
 绩效评审委员会看到的是效率的提升。他们看不到理解力的缺失，因为组织衡量产出的所有工具，没有一个能捕捉到这个维度。激励机制在它所衡量的范围内运转得无可挑剔，只是它所衡量的，已不再涵盖真正重要的东西。 
 这就是理解力负债比技术债务更加隐蔽的原因。技术债务通常是一种有意识的权衡 —— 你选择了捷径，你大致知道它藏在哪里，你可以安排时间偿还。理解力负债则是无声无息地积累，往往没有任何人刻意做出放任它的决定。它是数百次代码审查的总和 —— 每次代码看起来都没问题，测试全部通过，队列里还有下一个 PR 等着处理。 
 组织中那条 "审查过的代码就是被理解了的代码" 的默认假设，已经不再成立。工程师们批准了自己并未完全理解的代码，而这些代码如今带着隐性的背书在系统中运行。责任已经被不知不觉地分摊了出去。 
 监管的脚步比你以为的更近 
 每个发展过快的行业，最终都会引来监管。科技行业一直异乎寻常地免于这种命运，一方面因为软件故障通常是可恢复的，另一方面因为行业发展的速度远超监管者跟进的步伐。 
 但这扇窗口正在关闭。当 AI 生成的代码运行在医疗系统、金融基础设施和政府服务中时，一旦事故涉及生命安全或重大资产损失，"这是 AI 写的，我们没有完整审查过" 这句话，在事后调查报告中是站不住脚的。 
 那些现在就开始培养理解力纪律的团队 —— 将真正的理解而非仅仅通过测试视为不可妥协的底线 —— 当清算时刻到来时，将比那些一味追求合并速度的团队处于远为有利的位置。 
 理解力负债真正要求我们做什么 
 当下正确的问题不是 "如何生成更多代码？" 而是 "如何真正理解我们交付的每一行代码？"—— 唯有如此，我们才能确保用户始终获得高质量的体验。 
 这一视角的转换带来切实的后果。它意味着，在代码动笔之前，就要毫不含糊地明确一项变更究竟要做什么。它意味着，将验证视为结构性的约束条件，而非事后的补救措施。它意味着，维护系统层面的心智模型，使你能在架构尺度上捕捉 AI 的失误，而非逐行排查。它还意味着，对自己坦诚面对 "测试通过了" 与 "我理解这段代码做了什么以及为什么这样做" 之间的本质区别。 
 代码的生成成本变低了，并不意味着理解力就可以廉价地省略。理解的工作，才是工作本身。 
 AI 负责翻译，但总得有人去理解它产出了什么、为什么以这种方式产出、那些隐含的决策是否正确 —— 否则，你不过是在推迟一张终将全额到期的账单。 
 理解力的账，早晚都要还。而这笔债的利息，增长得比你想象的快得多。
```
